# Vertical Slice Ticket

## 목표

서비스 전체 UI 기준(T-0128)에 경기 리스트 계열 화면의 **테이블형 표시 규칙**을 추가하여,
경기목록 / 스케줄  / 결과입력 화면의 정보 구조를 통일한다.

이번 슬라이스 완료 시 다음이 가능해야 한다.

- 경기 리스트 계열 화면이 보드형 카드 나열이 아니라 **코트 카드 > division 카드 > 테이블** 구조를 따름
- 리그 / 토너먼트 / 휴식시간 표시 규칙이 통일됨
- 경기목록 / 스케줄  / 결과입력 화면이 같은 정보 구조를 공유함
- 이후 schedule / result 관련 UI 티켓이 이 규칙을 기준으로 구현 가능

## UI

경기 리스트 계열 화면의 공통 표시 규칙을 아래로 통일한다.

### 공통 상위 구조

```text
코트(카드)
  division(카드)
    리그
      [테이블]

    토너먼트
      [테이블]
```

즉

- 코트 = 카드
- division = 카드
- 실제 반복 데이터 = 테이블

카드 중첩은 2단까지만 사용하고, 경기/슬롯/결과 행은 카드가 아니라 테이블 행으로 표시한다.

---

### 1. 경기목록 표시 규칙

구조

```text
코트(카드)
  division(카드)
    리그
      [테이블]
      시간 / [A조] 팀명1 VS 팀명2 / 상태 / 점수

    토너먼트
      [테이블]
      시간 / [4강] 1위팀 VS 4위팀 / 상태 / 점수
```

#### 리그 테이블 컬럼

- 시간
- 경기
- 상태
- 점수

#### 토너먼트 테이블 컬럼

- 시간
- 경기
- 상태
- 점수

---

### 2. 스케줄  표시 규칙

구조

```text
코트(카드)
  division(카드)
    리그
      [테이블]
      시간 / [A조] 팀명1 VS 팀명2(드래그앤드롭) / 코트(드롭다운)

    토너먼트
      [테이블]
      시간 / [4강] 1위팀 VS 4위팀(드래그앤드롭) / 코트(드롭다운)
```

#### 리그 테이블 컬럼

- 시간
- 경기
- 코트

#### 토너먼트 테이블 컬럼

- 시간
- 경기
- 코트

#### 편집 규칙

- DnD는 행 단위 경기 교체/정렬 기준으로 해석
- 시간은 슬롯 기준으로 고정
- 코트는 드롭다운으로 변경
- 휴식시간도 일반 행으로 표시

휴식시간 예시

- 시간: 10:00 - 10:10
- 경기: [휴식시간]
- 코트: 코트1

---

### 3. 결과입력 표시 규칙

구조

```text
코트(카드)
  division(카드)
    리그
      [테이블]
      시간 / [A조] 팀명1 VS 팀명2 / 0:0(입력) / 상태(배지) / 저장(버튼)

    토너먼트
      [테이블]
      시간 / [4강] 1위팀 VS 4위팀 / 코트 / 0:0(입력) / 상태(배지) / 저장(버튼)
```

#### 리그 테이블 컬럼

- 시간
- 경기
- 점수 입력
- 상태
- 저장

#### 토너먼트 테이블 컬럼

- 시간
- 경기
- 코트
- 점수 입력
- 상태
- 저장

### 상태 표시 규칙

공통 배지 예시

- 진행중
- 완료
- 미배정
- 주의 필요

상태 배지는 경기목록 / 스케줄 / 결과입력 화면에서 동일한 스타일과 이름을 사용한다.

---

### 적용 대상 페이지

이 규칙은 이후 최소 아래 페이지들에 적용할 수 있어야 한다.

- /admin/tournaments/[id]/matches
- /admin/tournaments/[id]/schedule
- /admin/tournaments/[id]/result

필요 시 팀/선수 리스트와는 다른 “경기 리스트 계열 전용 표시 규칙”으로 유지한다.

## 권한

이번 슬라이스는 권한 로직 변경이 목적이 아니다.

기존 권한 정책을 유지하되, UI 표시 구조만 통일한다.

- organizer: 기존 편집 UI 유지
- team_manager / player / viewer: 기존 조회 정책 유지

- 이후 schedule / result / matches 화면이 같은 테이블형 기준으로 구현 가능해짐
- 기존 기능 동작에는 영향 없음
