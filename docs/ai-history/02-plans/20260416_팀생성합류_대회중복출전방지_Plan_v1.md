#### 1-1. `supabase/migrations/0246_rosters.sql`

**생성 테이블**: `rosters`, `roster_members`

**`rosters` 테이블**:
```
컬럼:
  id                uuid PK default gen_random_uuid()
  application_id    uuid NOT NULL REFERENCES tournament_team_applications(id) ON DELETE CASCADE
  team_id           uuid NOT NULL REFERENCES teams(id) ON DELETE RESTRICT
  tournament_id     uuid NOT NULL REFERENCES tournaments(id) ON DELETE RESTRICT
  status            text NOT NULL DEFAULT 'draft'
                    CHECK IN ('draft', 'submitted', 'confirmed', 'rejected')
  submitted_at      timestamptz NULL
  confirmed_at      timestamptz NULL
  confirmed_by      uuid NULL REFERENCES auth.users(id) ON DELETE SET NULL
  rejection_reason  text NULL
  created_at        timestamptz NOT NULL DEFAULT now()
  updated_at        timestamptz NOT NULL DEFAULT now()

제약:
  UNIQUE(application_id)  -- 신청 1건당 로스터 1개
```

**설계 결정 이유**:
- `application_id` UNIQUE: 대회 신청 1건 = 로스터 1개 (1:1 관계)
- `team_id`, `tournament_id`는 application에서 denormalize: 로스터 조회 시 JOIN 없이 tournament 기준 중복 검사 가능
- `confirmed_by`: organizer가 확정한 경우 기록 (현재 MVP에서는 자동 확정도 고려)

**`roster_members` 테이블**:
```
컬럼:
  id          uuid PK default gen_random_uuid()
  roster_id   uuid NOT NULL REFERENCES rosters(id) ON DELETE CASCADE
  user_id     uuid NOT NULL REFERENCES auth.users(id) ON DELETE RESTRICT
  created_at  timestamptz NOT NULL DEFAULT now()

제약:
  UNIQUE(roster_id, user_id)   -- 같은 로스터 내 중복 선수 방지
  INDEX(user_id)               -- 중복 출전 검사 성능
```

**인덱스**:
```sql
-- 중복 출전 검사용 복합 인덱스
CREATE INDEX idx_roster_members_tournament_check
  ON roster_members(user_id)
  INCLUDE (roster_id);

-- 로스터 상태별 조회용
CREATE INDEX idx_rosters_tournament_status
  ON rosters(tournament_id, status);
```

**롤백**:
```sql
DROP TABLE IF EXISTS roster_members;
DROP TABLE IF EXISTS rosters;
```

#### 1-2. `supabase/migrations/0247_rosters_rls.sql`

**RLS 활성화 + 정책 정의**

**`rosters` 테이블 정책**:
```
ENABLE ROW LEVEL SECURITY;

SELECT:
  - is_team_member_for_team(team_id)  -- 해당 팀 멤버면 읽기 가능
  - OR is_organizer()

INSERT:
  - is_team_manager_for_team(team_id)  -- captain만 생성
  - AND status = 'draft'               -- 최초 상태는 draft만

UPDATE:
  - is_team_manager_for_team(team_id) AND status IN ('draft', 'submitted')
  - OR is_organizer()

DELETE:
  - is_team_manager_for_team(team_id) AND status = 'draft'
```

**`roster_members` 테이블 정책**:
```
ENABLE ROW LEVEL SECURITY;

SELECT:
  - 해당 roster의 team에 is_team_member_for_team(team_id)
  - OR is_organizer()

INSERT:
  - 해당 roster의 team에 is_team_manager_for_team(team_id)
  - AND 해당 roster.status IN ('draft', 'submitted')

DELETE:
  - 해당 roster의 team에 is_team_manager_for_team(team_id)
  - AND 해당 roster.status IN ('draft', 'submitted')
```

**재사용**: `public.is_team_member_for_team()`, `public.is_team_manager_for_team()`, `public.is_organizer()` — 기존 0088에 정의된 헬퍼 함수 그대로 사용.

**롤백**:
```sql
DROP POLICY IF EXISTS ... ON rosters;
DROP POLICY IF EXISTS ... ON roster_members;
ALTER TABLE rosters DISABLE ROW LEVEL SECURITY;
ALTER TABLE roster_members DISABLE ROW LEVEL SECURITY;
```

#### 1-3. `supabase/migrations/0248_roster_rpcs.sql`

**SECURITY DEFINER RPC 5개 정의**

**1) `upsert_roster(p_application_id uuid) RETURNS jsonb`**

```
목적: application_id에 대한 로스터가 없으면 생성, 있으면 기존 반환
호출자: team captain

로직:
  1. application의 team captain 확인 (team_members.role_in_team = 'captain')
  2. SELECT * FROM rosters WHERE application_id = p_application_id
  3. 있으면 RETURN {ok: true, roster_id: existing.id, created: false}
  4. 없으면:
     - application에서 team_id, tournament_id 조회
     - INSERT INTO rosters (application_id, team_id, tournament_id, status='draft')
     - RETURN {ok: true, roster_id: new.id, created: true}
```

**2) `add_roster_member(p_roster_id uuid, p_user_id uuid) RETURNS jsonb`**

```
목적: 로스터에 선수 추가
호출자: team captain

로직:
  1. roster 조회 + team의 captain 확인
  2. roster.status IN ('draft', 'submitted') 확인 (confirmed/rejected면 에러)
  3. p_user_id가 해당 팀의 team_members에 있는지 확인
     → 팀원이 아니면 에러: "팀 소속 선수만 로스터에 추가할 수 있습니다"
  4. UNIQUE 제약에 의해 중복 추가 자동 방지
  5. INSERT INTO roster_members (roster_id, user_id)
  6. RETURN {ok: true}
```

**3) `remove_roster_member(p_roster_id uuid, p_user_id uuid) RETURNS jsonb`**

```
목적: 로스터에서 선수 제거
호출자: team captain

로직:
  1. roster 조회 + captain 확인
  2. roster.status IN ('draft', 'submitted') 확인
  3. DELETE FROM roster_members WHERE roster_id = p_roster_id AND user_id = p_user_id
  4. RETURN {ok: true}
```

**4) `submit_roster(p_roster_id uuid) RETURNS jsonb`**

```
목적: 로스터 제출 (draft → submitted)
호출자: team captain

로직:
  1. roster 조회 + captain 확인
  2. roster.status = 'draft' 확인
  3. SELECT COUNT(*) FROM roster_members WHERE roster_id = p_roster_id
     → 0명이면 에러: "최소 1명 이상의 선수가 필요합니다"
  4. UPDATE rosters SET status = 'submitted', submitted_at = now() WHERE id = p_roster_id
  5. RETURN {ok: true}

(MVP에서는 submitted 후 자동 confirmed 처리 여부 결정 필요 → 우선 submitted 상태 유지)
```

**5) `confirm_roster(p_roster_id uuid) RETURNS jsonb`**

```
목적: 로스터 확정 + 동일 대회 중복 출전 방지 검사
호출자: organizer (또는 MVP에서는 captain이 submit 시 자동 호출 방식 고려)

로직:
  1. roster 조회 (status = 'submitted' 확인)
  2. is_organizer() 확인
  3. ===== 핵심: 중복 출전 방지 검사 =====
     v_tournament_id = roster.tournament_id
     v_team_id = roster.team_id

     SELECT rm.user_id
     FROM roster_members rm
     JOIN rosters r ON r.id = rm.roster_id
     WHERE r.tournament_id = v_tournament_id
       AND r.status = 'confirmed'
       AND r.team_id != v_team_id
       AND rm.user_id IN (
         SELECT user_id FROM roster_members WHERE roster_id = p_roster_id
       )
     → 결과가 1명이라도 있으면:
       RETURN {ok: false, error: "이미 같은 대회 다른 팀 로스터에 포함된 선수가 있습니다", 
               conflicting_user_ids: [...]}

  4. UPDATE rosters SET status = 'confirmed', confirmed_at = now(), confirmed_by = auth.uid()
  5. RETURN {ok: true}
```

### Phase 5. 권한 정책 (RLS)

Phase 1-2 (`0247_rosters_rls.sql`)에서 처리. 별도 파일 없음.

**정책 요약**:

| 테이블 | SELECT | INSERT | UPDATE | DELETE |
|--------|--------|--------|--------|--------|
| `rosters` | 팀 멤버 OR organizer | 팀 captain (status='draft'만) | 팀 captain (draft/submitted) OR organizer | 팀 captain (draft만) |
| `roster_members` | 팀 멤버 OR organizer | 팀 captain (roster가 draft/submitted일 때) | 불가 | 팀 captain (roster가 draft/submitted일 때) |

**기존 헬퍼 함수 재사용**:
- `public.is_organizer()` — 0002에 정의
- `public.is_team_member_for_team(uuid)` — 0088에 정의
- `public.is_team_manager_for_team(uuid)` — 0088+0207에 정의 (내부는 captain 체크)

**RLS 한계 및 대응**:
- `roster_members`의 INSERT/DELETE 정책에서 roster의 team_id 확인이 필요 → subquery 또는 JOIN 사용
  - 예: `EXISTS(SELECT 1 FROM rosters r WHERE r.id = roster_id AND is_team_manager_for_team(r.team_id) AND r.status IN ('draft', 'submitted'))`
- RPC(`add_roster_member`, `remove_roster_member`)는 SECURITY DEFINER이므로 RLS 우회 → RPC 내부에서 직접 권한 검증

## 6. 리스크 및 주의사항

### 6-1. `getUserTeamStatus` 다중 팀 호환성

**문제**: 현재 `getUserTeamStatus`는 `.limit(1)`로 첫 번째 팀만 확인. 여러 팀 captain이면 항상 "captain" 반환 (문제 없음). 여러 팀 중 하나가 player인 경우도 "team_member" 반환 후 중단 → 다른 팀에서 captain인 경우 누락.

**현재 사용처**:
- `app/(app)/teams/new/page.tsx`: Phase 0에서 로직 제거
- 다른 사용처 있을 시 검토 필요

**대응**: Phase 0에서 captain 체크 제거 후, `getUserTeamStatus`의 다른 사용처를 검색해서 영향 확인. 사용처가 더 있다면 함수 로직도 수정.

### 6-2. RLS의 roster_members 정책 복잡도

**문제**: `roster_members`에서 INSERT/DELETE 권한 확인 시 roster의 team_id와 status를 서브쿼리로 확인해야 함. RLS 정책이 복잡해지면 성능 이슈 발생 가능.

**대응**: RPC(`add_roster_member`, `remove_roster_member`)를 SECURITY DEFINER로 만들어 RLS 우회 + RPC 내부에서 명시적 권한 검증. RLS는 SELECT 정책만 엄격하게 적용, INSERT/DELETE는 RPC 경유를 강제.

### 6-3. `confirm_roster` 중복 검사 성능

**문제**: `confirm_roster` RPC의 중복 출전 검사 쿼리가 roster_members 전체를 스캔할 수 있음.

```sql
-- 검사 대상 쿼리
SELECT rm.user_id FROM roster_members rm
JOIN rosters r ON r.id = rm.roster_id
WHERE r.tournament_id = v_tournament_id
  AND r.status = 'confirmed'
  AND r.team_id != v_team_id
  AND rm.user_id IN (SELECT user_id FROM roster_members WHERE roster_id = p_roster_id)
```

**대응**: Phase 1에서 `idx_rosters_tournament_status (tournament_id, status)` + `idx_roster_members_user_id (user_id)` 인덱스 생성. 대규모에서도 Index Scan 가능.

### 6-4. 신청 상태와 로스터 상태의 독립성

**문제**: `tournament_team_applications.status = 'confirmed'` (대회 참가 확정)와 `rosters.status = 'confirmed'` (로스터 확정)는 별개. 대회 참가 확정된 팀만 로스터를 제출/확정할 수 있어야 하나, 현재 `submit_roster` RPC 설계에서 이 체크가 명시되어 있지 않음.

**대응**: `submit_roster` RPC 내부에서 `application.status = 'confirmed'` 체크 추가. 대회 참가가 확정되지 않은 팀은 로스터 제출 불가.

### 6-5. 팀 멤버 변경과 로스터 정합성

**문제**: 로스터를 submitted/confirmed 후 팀 멤버가 팀을 탈퇴하면 roster_members에 남아 있는 상태 발생.

**대응 (MVP)**: `team_members.user_id` FK는 ON DELETE RESTRICT이 아닌 설계. 탈퇴 기능이 아직 미구현이므로 현재 시점에서는 고려하지 않음. 추후 팀 탈퇴 기능 구현 시 roster_members 연계 처리 필요.

### 6-6. 로스터 대상 application 범위

**문제**: 현재 `tournament_team_applications`에는 payment_pending, paid_pending_approval, confirmed 등 다양한 상태가 있음. 로스터 관리 페이지에서 어떤 상태의 신청에 대해 로스터를 생성/편집할 수 있게 할지 명확히 해야 함.

**결정**: `confirmed` 상태의 신청에 대해서만 로스터 생성/편집/제출 가능. `submit_roster` RPC에서 application.status = 'confirmed' 체크.

### 6-7. `app/(app)/teams/new/page.tsx` import 정리

Phase 0에서 `getUserTeamStatus` 관련 코드 제거 시 import 라인도 함께 제거해야 함:
- `import { getUserTeamStatus } from "@/lib/api/team-applications"` 제거 (해당 파일에서만 사용 시)
- TypeScript 컴파일 에러 방지
