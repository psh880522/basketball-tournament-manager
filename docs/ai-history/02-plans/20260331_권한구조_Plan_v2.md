## 1. 구현 목표 요약

현재 `organizer / team_manager / player / spectator` 4종 역할을 **`organizer / manager / player`** 3종으로 재편한다.

| 변경 항목 | 현재 | 목표 |
|---|---|---|
| 전역 역할 종류 | organizer, team_manager, player, spectator | organizer, manager, player |
| 신규 가입 기본 역할 | spectator | player |
| 팀 생성/관리 담당 | team_manager | player |
| 운영 스태프 역할 | 없음 (organizer가 전담) | manager (경기 결과·상태 관리) |
| 코트 관리 | organizer 전용 | organizer 전용 유지 (manager 불가) |
| 역할 변경 UI | 없음 | organizer 전용 사용자 관리 페이지 |
| 사이드 네비게이션 | 헤더 메뉴 | 좌측 사이드바 (역할별 메뉴 구성) |
| team_members.role_in_team 값 | 'manager' / 'player' | 'captain' / 'player' |
| profiles 직접 role 수정 | 가능 (보안 취약점) | RPC를 통해서만 변경 가능 |

### Phase 1 — DB 마이그레이션 [완료]

#### Step 1-1: enum 전환 + 기본값 변경

**신규 파일:** `supabase/migrations/0205_role_system_v2.sql`

```sql
-- 1) 새 enum 생성
CREATE TYPE public.app_role_v2 AS ENUM ('organizer', 'manager', 'player');

-- 2) profiles 컬럼 타입 교체
--    기존 값 매핑: spectator → player, team_manager → player
ALTER TABLE public.profiles
  ALTER COLUMN role DROP DEFAULT;

ALTER TABLE public.profiles
  ALTER COLUMN role TYPE public.app_role_v2
  USING (
    CASE role::text
      WHEN 'organizer'    THEN 'organizer'::public.app_role_v2
      WHEN 'manager'      THEN 'manager'::public.app_role_v2
      WHEN 'team_manager' THEN 'player'::public.app_role_v2
      WHEN 'spectator'    THEN 'player'::public.app_role_v2
      ELSE                     'player'::public.app_role_v2
    END
  );

ALTER TABLE public.profiles
  ALTER COLUMN role SET DEFAULT 'player';

-- 3) 구 enum 제거 (컬럼 교체 후에만 가능)
DROP TYPE public.app_role;

-- 4) 새 enum 이름 변경
ALTER TYPE public.app_role_v2 RENAME TO app_role;
```

**트레이드오프:**
- PostgreSQL enum 직접 수정(`ALTER TYPE ... ADD VALUE`)은 트랜잭션 안에서 새 타입 즉시 사용 불가한 경우가 있어, 새 타입 생성 + 컬럼 교체 + 구 타입 삭제 방식을 사용한다.
- 기존 `team_manager`/`spectator` 사용자가 모두 `player`로 전환된다. 운영 스태프가 있다면 마이그레이션 후 사용자 관리 UI에서 `manager`로 승격 필요.

#### Step 1-2: is_manager() RLS 함수 추가 + 불필요 함수 정리

**신규 파일:** `supabase/migrations/0206_rls_manager_role.sql`

```sql
-- 1) manager 체크 함수 신규 추가
--    is_manager()는 organizer + manager 포함
--    (organizer는 manager의 모든 권한을 가지므로)
CREATE OR REPLACE FUNCTION public.is_manager()
RETURNS boolean
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
SET row_security = off
AS $$
BEGIN
  RETURN EXISTS (
    SELECT 1 FROM public.profiles p
    WHERE p.id = auth.uid()
      AND p.role IN ('organizer', 'manager')
  );
END;
$$;

-- organizer 전용 체크는 기존 is_organizer()를 그대로 사용.

-- 2) 구 is_team_manager() 함수 제거 (role='team_manager' 기반 → 불필요)
DROP FUNCTION IF EXISTS public.is_team_manager();
-- 대체: is_team_manager_for_team(team_uuid) 함수가 이미 0088에 존재.
--       단, 해당 함수 내부의 role_in_team = 'manager' 비교는
--       Step 1-4에서 'captain'으로 변경.
```

#### Step 1-3: team_members.role_in_team 값 'manager' → 'captain' 변경

**신규 파일:** `supabase/migrations/0207_team_members_captain.sql`

```sql
-- 1) CHECK 제약 업데이트
ALTER TABLE public.team_members
  DROP CONSTRAINT IF EXISTS team_members_role_in_team_check;

ALTER TABLE public.team_members
  ADD CONSTRAINT team_members_role_in_team_check
  CHECK (role_in_team IN ('captain', 'player'));

-- 2) 기존 데이터 마이그레이션
UPDATE public.team_members
  SET role_in_team = 'captain'
  WHERE role_in_team = 'manager';

-- 3) is_team_manager_for_team 함수 내 비교값 업데이트
CREATE OR REPLACE FUNCTION public.is_team_manager_for_team(team_uuid uuid)
RETURNS boolean
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
SET row_security = off
AS $$
BEGIN
  RETURN EXISTS (
    SELECT 1 FROM public.team_members tm
    WHERE tm.team_id = team_uuid
      AND tm.user_id = auth.uid()
      AND tm.role_in_team = 'captain'   -- 'manager' → 'captain'
  );
END;
$$;
```

**변경 이유:** 전역 역할의 `manager`(운영 스태프)와 팀 내 역할의 `manager`(팀 대표)가 동명이어서 혼동 발생 가능. `captain`으로 명칭을 분리한다.

#### Step 1-5: profiles 역할 변경 보안 수정 + RPC 추가

**신규 파일:** `supabase/migrations/0209_profiles_role_rpc.sql`

```sql
-- 1) organizer 전용 역할 변경 RPC (SECURITY DEFINER)
--    이 함수를 통해서만 role 변경 가능.
--    profiles_update_own 정책은 그대로 유지하되,
--    프론트엔드에서 role 변경 UI는 이 RPC를 통해서만 노출.
CREATE OR REPLACE FUNCTION public.update_user_role(
  target_user_id uuid,
  new_role text
)
RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
BEGIN
  -- 호출자가 organizer인지 확인
  IF NOT public.is_organizer() THEN
    RAISE EXCEPTION 'Forbidden: only organizer can change roles';
  END IF;

  -- 허용 값: player, manager만 변경 가능 (organizer 승격은 DB 직접 처리)
  IF new_role NOT IN ('player', 'manager') THEN
    RAISE EXCEPTION 'Invalid role: only player or manager allowed';
  END IF;

  -- organizer 본인 역할은 변경 불가
  IF EXISTS (
    SELECT 1 FROM public.profiles
    WHERE id = target_user_id AND role = 'organizer'
  ) THEN
    RAISE EXCEPTION 'Cannot change organizer role';
  END IF;

  UPDATE public.profiles
  SET role = new_role::public.app_role
  WHERE id = target_user_id;
END;
$$;
```

**트레이드오프:**
- PostgreSQL RLS `WITH CHECK`는 `OLD.*` 참조 불가. SECURITY DEFINER RPC + 프론트엔드 이중 방어로 보완.
- organizer 본인 역할 변경 방지 로직을 RPC 내부에 추가.

#### Step 1-6: 운영 기능 RLS — manager 허용 범위 (코트 제외)

**신규 파일:** `supabase/migrations/0210_manager_operation_rls.sql`

```sql
-- matches: manager도 UPDATE 가능 (경기 결과 입력/상태 변경)
-- (현재 matches RLS 정책 이름 확인 후 교체)
DROP POLICY IF EXISTS "matches_update_organizer" ON public.matches;
CREATE POLICY "matches_update_organizer_manager"
  ON public.matches FOR UPDATE
  USING (public.is_manager())
  WITH CHECK (public.is_manager());

-- standings: manager도 업데이트 가능 (순위 재계산)
DROP POLICY IF EXISTS "standings_upsert_organizer" ON public.standings;
CREATE POLICY "standings_upsert_manager"
  ON public.standings FOR ALL
  USING (public.is_manager())
  WITH CHECK (public.is_manager());

-- courts: organizer 전용 유지 (manager 불가)
-- → 변경 없음. 기존 courts RLS 정책 현행 유지.
```

**코트 관리 organizer 전용 유지 이유:** 코트는 대회 운영 인프라 설정에 해당하며 경기 진행 중 임의 변경은 리스크가 크다. manager는 코트 배정(assign)은 수행 가능하지만, 코트 생성/삭제는 organizer만 허용.

#### Step 1-7: 사용자 이메일 조회용 DB View 생성

**신규 파일:** `supabase/migrations/0211_user_profiles_view.sql`

```sql
-- auth.users와 profiles를 JOIN한 View
-- 이메일은 auth.users가 유일한 원본 소스이므로 비정규화 없이 View로 조회
CREATE OR REPLACE VIEW public.user_profiles AS
  SELECT
    p.id,
    p.role,
    p.created_at,
    u.email
  FROM public.profiles p
  JOIN auth.users u ON u.id = p.id;

-- View에 RLS 적용 (Supabase는 기본적으로 View에 별도 RLS 미적용)
-- 대신 security_invoker=true 옵션으로 View 호출자의 권한으로 실행
ALTER VIEW public.user_profiles SET (security_invoker = true);
-- 이렇게 하면 profiles 테이블의 RLS(profiles_select_own_or_organizer)가 View 접근에도 적용됨.
-- organizer만 전체 사용자 조회 가능, 일반 사용자는 본인 행만 조회.
```

**이메일 처리 방식 결정 (v1 profiles email 컬럼 방식 폐기):**

| 방법 | 장점 | 단점 | 채택 |
|---|---|---|---|
| profiles.email 컬럼 추가 | 단순, 빠름 | 비정규화, 이메일 변경 시 sync 필요, 데이터 불일치 위험 | ❌ 폐기 |
| Supabase Admin API | 항상 최신 email | service_role key 노출 위험, Server Action 환경 복잡 | ❌ 보류 |
| DB View (security_invoker) | 정규화 유지, 속도·안정성 우수, RLS 자연 적용 | auth.users 직접 참조 (Supabase 내부 스키마 의존) | ✅ 채택 |

View + `security_invoker=true`는 Supabase에서 공식 지원하는 패턴이며, 데이터 동기화 문제 없이 항상 최신 이메일을 안정적으로 제공한다.

// user_profiles View를 통해 id + role + email 조회
// RLS(profiles_select_own_or_organizer)가 View에 security_invoker로 적용
export async function listUsersWithRole(): Promise<{
  data: { id: string; email: string; role: Role }[] | null;
  error: string | null;
}> {
  const supabase = await createSupabaseServerClient();
  const { data, error } = await supabase
    .from("user_profiles")   // View 이름
    .select("id, role, email")
    .order("created_at", { ascending: false });

  if (error) return { data: null, error: error.message };
  return { data, error: null };
}

#### Step 6-2: 역할 변경 서버 액션

**신규 파일:** `app/admin/users/actions.ts`

```typescript
"use server";

import { getUserWithRole } from "@/src/lib/auth/roles";
import { createSupabaseServerClient } from "@/src/lib/supabase/server";
import { revalidatePath } from "next/cache";

type UpdateRoleInput = {
  targetUserId: string;
  newRole: "player" | "manager";
};

export async function updateUserRoleAction(input: UpdateRoleInput): Promise<ActionResult> {
  const caller = await getUserWithRole();

  if (caller.status === "unauthenticated") return { ok: false, error: "로그인 필요." };
  if (caller.role !== "organizer") return { ok: false, error: "권한 없음." };
  if (!input.targetUserId) return { ok: false, error: "대상 사용자 ID 필요." };

  const supabase = await createSupabaseServerClient();

  // SECURITY DEFINER RPC 호출 (DB 레벨에서 organizer 재검증 + organizer 보호)
  const { error } = await supabase.rpc("update_user_role", {
    target_user_id: input.targetUserId,
    new_role: input.newRole,
  });

  if (error) return { ok: false, error: error.message };

  revalidatePath("/admin/users");
  return { ok: true };
}
```

## 6. 트레이드오프 및 고려 사항

### 6-1. 기존 team_manager 사용자 일괄 player 전환

enum 전환 시 기존 `team_manager` 사용자는 자동으로 `player`로 변환된다. `team_members.role_in_team = 'captain'` 기반 팀 관리 권한은 그대로 유지되므로 기능적 차이는 없다.

### 6-2. captain vs manager 네이밍 명확화

`team_members.role_in_team`:
- `'captain'` — 팀 대표, 팀 내 관리 권한
- `'player'` — 일반 팀원

전역 `profiles.role`:
- `'manager'` — 운영 스태프 (경기 진행 관리)
- `'player'` — 일반 사용자 (팀 참가, 대회 조회)

두 개념이 명확히 분리되어 혼동 방지.

### 6-3. is_manager() 범위 (organizer 포함)

`is_manager()`는 `organizer + manager`를 포함한다. RLS 정책에서 organizer가 manager 기능을 수행하기 위해 별도 조건 없이 자연스럽게 처리된다.

### 6-4. 코트 관리 organizer 전용 유지

코트는 대회 인프라 설정이므로 manager의 접근을 허용하지 않는다. 단, 코트 배정(`ASSIGN_COURT` stepKey)은 경기 배정 맥락이므로 별도 판단 필요. v2에서는 `ASSIGN_COURT`도 organizer 전용으로 유지.

### 6-5. 사이드바 레이아웃 전환 방식

Route Group 분리(`(app)/layout.tsx`, `(public)/layout.tsx`) vs 단일 `app/layout.tsx`에서 조건 분기 중 선택 필요. Route Group은 구조가 깔끔하지만 기존 `app/` 하위 모든 페이지 경로 이동 필요. 단일 layout 조건 분기는 기존 라우트 유지 가능.

### 6-6. user_profiles View + security_invoker

`security_invoker=true`는 PostgreSQL 15+ 에서 지원. Supabase는 PostgreSQL 15를 기본 지원하므로 문제 없음. View를 통한 `auth.users` 조인이 공식 Supabase 패턴이므로 안정적.
