v2 문서에 추가된 사용자 메모를 반영한 변경사항:

| 메모 내용 | 반영 결과 |

### 신규 파일: `src/lib/auth/guards.ts`

```typescript
// src/lib/auth/guards.ts
import { getUserWithRole, isOperationRole, type Role } from "./roles"

type GuardSuccess = { ok: true; userId: string; role: Role }
type GuardFailure = { ok: false; error: string }
export type GuardResult = GuardSuccess | GuardFailure

const AUTH_ERRORS = {
  unauthenticated: "로그인이 필요합니다.",
  error: "인증 오류가 발생했습니다.",
  empty: "프로필이 없습니다.",
} as const

/** organizer 전용 작업에서 사용 */
export async function requireOrganizer(): Promise<GuardResult> {
  const result = await getUserWithRole()
  if (result.status !== "ready") {
    return { ok: false, error: AUTH_ERRORS[result.status] ?? "인증 오류가 발생했습니다." }
  }
  if (result.role !== "organizer") {
    return { ok: false, error: "권한이 없습니다." }
  }
  return { ok: true, userId: result.user!.id, role: result.role }
}

/** organizer + manager 허용 작업에서 사용 (결과 입력, 순위 계산 등) */
export async function requireOperationRole(): Promise<GuardResult> {
  const result = await getUserWithRole()
  if (result.status !== "ready") {
    return { ok: false, error: AUTH_ERRORS[result.status] ?? "인증 오류가 발생했습니다." }
  }
  if (!isOperationRole(result.role)) {
    return { ok: false, error: "권한이 없습니다." }
  }
  return { ok: true, userId: result.user!.id, role: result.role! }
}
```

## 4. Phase 4: `tournamentProgress.ts` 쿼리 병렬화

**파일:** `lib/api/tournamentProgress.ts`

### 현재 구조 (15회 순차 await)

```
1. tournaments 조회 → 필수 선행 (tournament 존재 여부, status 확인)
2. divisions 조회  → 필수 선행 (divisionIds 추출 후 groups 쿼리에 필요)

// 2단계 (필수 선행)

  groupMatches, completedGroupMatches,
  standings, totalMatches, completedMatches,
  tournamentMatches, finalCompleted, finalExists,
  courtsCount, scheduledMatches
] = await Promise.all([ /* 13개 쿼리 */ ])
```

**예상 효과:** 이론상 ~80% 응답 시간 단축 (순차 15회 → 3단계 병렬)

**주의 사항:**
- `divisionIds`가 빈 배열일 때 `.in("division_id", [])` 단락 처리 필수 (기존 `countGroupsByDivisionIds` 헬퍼의 동일한 처리 로직 유지)
- `Promise.all` 내 개별 에러는 `{ error }` 형태로 반환되므로 각각 에러 체크 유지

```typescript
// lib/api/schedule-slots.ts 끝부분에 추가

/** matches 페이지에서 슬롯 목록을 단순 flat list로 조회할 때 사용 */
export type ScheduleSlotRow = {
  id: string
  tournament_id: string
  division_id: string | null
  court_id: string | null
  slot_type: "break" | "maintenance" | "buffer" | "match" | "tournament_placeholder"
  start_at: string
  end_at: string
  label: string | null
  match_id: string | null
  sort_order: number
}

export async function listScheduleSlots(
  tournamentId: string
): Promise<{ data: ScheduleSlotRow[] | null; error: string | null }> {
  // scheduleSlots.ts의 listScheduleSlots()와 동일한 로직 그대로 이식
  const supabase = await createSupabaseServerClient()
  const { data, error } = await supabase
    .from("schedule_slots")
    .select("id,tournament_id,division_id,court_id,slot_type,start_at,end_at,label,match_id,sort_order")
    .eq("tournament_id", tournamentId)
    .order("sort_order", { ascending: true })
    .order("start_at", { ascending: true })
  return { data: data as ScheduleSlotRow[] | null, error: error ? error.message : null }
}
```

*Note: `getScheduleSlots()`는 복잡한 그룹핑 구조를 반환하는 별도 함수이므로 `listScheduleSlots()`와 다르다. 두 함수 모두 필요하다.*

## 8. 트레이드오프 및 고려 사항

### Phase 2: 공통 인증 가드

**`ensureOrganizer()` vs `requireOrganizer()`** 통합 시 에러 메시지 정책:
- 기존 `bracket.ts`, `schedule-slots.ts`: `status !== "ready"` → 단순히 "권한이 없습니다."
- 기존 `tournaments.ts`: status별 세분화 ("로그인 필요", "프로필 없음" 등)
- **v3 방향**: 세분화된 에러 메시지로 통일 (guards.ts의 `AUTH_ERRORS` 매핑 사용)

### Phase 3: 공통 타입

**`schedule/actions.ts`도 Phase 3 대상에 포함됨:**
현재 로컬 `type ActionResult`를 선언하고 있고, Phase 6에서 이 파일을 수정하므로 Phase 3에서 함께 처리한다.

### Phase 4: 성능 개선

**Vercel 환경 고려사항:**
- 13개 병렬 Supabase 연결이 free tier에서 문제없는지 확인 필요
- Supabase SDK는 내부적으로 connection pool 사용하므로 일반적으로 문제없음
