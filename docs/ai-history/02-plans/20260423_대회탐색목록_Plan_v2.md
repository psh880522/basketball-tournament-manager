## 1. 구현 범위 요약

### 1-1. 이번에 구현할 것

| 항목 | 설명 |
|---|---|
| 탭 기반 대회 목록 | 전체/모집중/진행중/종료/내 신청 탭 (URL searchParams) |
| 키워드 검색 | 대회명·장소 ilike 검색, debounce 300ms |
| 날짜 범위 필터 | start_date >= dateFrom, start_date <= dateTo |
| 확장 대회 카드 | 상태뱃지, 부문 칩, 참가비 범위, 정원(capacity), CTA 분기 |
| 사용자 신청 상태 표시 | `listAllMyTeamApplications()` 결과를 카드별 매핑 |
| 모바일 바텀시트 필터 | 공용 `BottomSheet` UI 컴포넌트 기반, 날짜 필터만 포함 |
| 데스크톱 인라인 필터 | date input, md 이상에서만 노출 |
| 활성 필터 칩 표시 | 적용된 날짜 필터를 칩으로 표시, 개별/전체 초기화 |
| 로딩 스켈레톤 | Suspense fallback으로 목록 로딩 중 표시 |

### 1-2. 제외할 것

| 항목 | 이유 |
|---|---|
| `confirmed_count` 표시 (신청 수) | N+1 쿼리 비용 회피, MVP에서는 capacity만 표시 |
| 신청 마감 임박 뱃지 | `application_open_at/close_at` 컬럼 drop됨 (0240 마이그레이션) |
| **지역 필터** | **대회 생성 시 지역 데이터(location)를 입력하지 않아 필터 효용 없음** |
| 카드형/리스트형 전환 | MVP 범위 외 |
| 개인화 추천 (인기/마감 임박) | 사용자 region 데이터 없음, 집계 비용 과다 |
| 페이지네이션 / 무한 스크롤 | 현재 대회 수가 적어 전체 로드로 충분 |
| `draft` 상태 대회 노출 | RLS에서 player에게 비공개, 구현 불필요 |

### Phase 3. 서버 액션 / API

**변경 파일: `lib/api/tournaments.ts`**

#### 3-1. `getTournamentList()` 함수 신규 추가

```
// lib/api/tournaments.ts에 추가
export async function getTournamentList(
  params: TournamentListParams
): Promise<ApiResult<TournamentListItem[]>>
```

**의사코드:**

```
1. createSupabaseServerClient() 로 supabase 인스턴스 생성

2. 기본 select 구성:
   supabase
     .from('tournaments')
     .select(`
       id, name, location, start_date, end_date, status, poster_url,
       divisions(id, name, entry_fee, capacity, sort_order)
     `)
     .is('deleted_at', null)

3. status 필터 적용:
   - params.ids 있으면: .in('id', params.ids)
   - params.status 없으면: .in('status', ['open', 'closed', 'finished'])
   - params.status 단일 값: .eq('status', params.status)
   - params.status 배열: .in('status', params.status)

4. keyword 필터 적용 (params.keyword 있을 때):
   .or(`name.ilike.%${keyword}%,location.ilike.%${keyword}%`)

5. dateFrom 필터 적용 (params.dateFrom 있을 때):
   .gte('start_date', dateFrom)

6. dateTo 필터 적용 (params.dateTo 있을 때):
   .lte('start_date', dateTo)

7. 정렬:
   .order('status', custom order: open > closed > finished > draft)
   .order('start_date', ascending: true)
   → Supabase에서 복합 order를 지원하지 않으므로
     status 우선 정렬은 클라이언트 측 sort로 처리:
     const STATUS_ORDER = { open: 0, closed: 1, finished: 2, draft: 3 }
     data.sort((a, b) => STATUS_ORDER[a.status] - STATUS_ORDER[b.status] || ...)

8. data 정규화:
   각 row의 divisions를 sort_order ASC로 재정렬
   DivisionSummary 타입으로 매핑 (id, name, entry_fee, capacity만 추출)

9. ApiResult<TournamentListItem[]> 반환
```

> **v1 대비 변경**: 기존 step 5 "region 필터 적용" 제거. 이후 step 번호 재정렬. 주의사항의 "keyword와 region 중복 적용 방지" 항목 제거.

function getCtaConfig(tournament, myApp, role):
  // 비로그인 또는 player가 아님
  if role !== 'player':
    return { secondary: { label: '상세보기', href: `/tournament/${id}` } }

  // open이 아닌 대회
  if tournament.status !== 'open':
    return { secondary: { label: '상세보기', href: `/tournament/${id}` } }

  // open + 신청 없음
  if !myApp:
    return {
      secondary: { label: '상세보기', href: `/tournament/${id}` },
      primary: { label: '참가 신청', href: `/tournament/${id}/apply` }
    }

  // open + 활성 신청 있음 (상태별)
  switch myApp.status:
    case 'payment_pending':
      return {
        secondary: { label: '상세보기', href: `/tournament/${id}` },
        ghost: { label: '입금 안내', href: `/my-applications/${myApp.id}` }
      }
    case 'paid_pending_approval':
    case 'waitlisted':
      return {
        secondary: { label: '신청 현황', href: `/my-applications/${myApp.id}` }
      }
    case 'confirmed':
      return {
        secondary: { label: '상세보기', href: `/tournament/${id}` }
      }
    case 'cancelled':
    case 'expired':
      return {
        secondary: { label: '상세보기', href: `/tournament/${id}` },
        primary: { label: '다시 신청', href: `/tournament/${id}/apply` }
      }
```

#### 4-4. `SearchBar.tsx` 신규

파일: `app/(app)/tournaments/_components/SearchBar.tsx`

- `"use client"` 클라이언트 컴포넌트
- props: `{ defaultValue?: string }`
- 내부 상태: `query` (string)
- `useEffect`로 debounce 300ms 구현 (inline setTimeout/clearTimeout 패턴)
- debounce 후 `router.replace('/tournaments?q=...&tab=...&dateFrom=...&dateTo=...')` — 기존 파라미터 유지
- 빈 값이면 `q` 파라미터 삭제

```
// debounce 패턴
useEffect(() => {
  const timer = setTimeout(() => {
    const params = new URLSearchParams(searchParams.toString())
    if (query.trim()) {
      params.set('q', query.trim())
    } else {
      params.delete('q')
    }
    router.replace(`/tournaments?${params.toString()}`)
  }, 300)
  return () => clearTimeout(timer)
}, [query])
```

    params.delete('dateTo')

      <div className="px-4 pb-8 pt-2">

        </div>

        </Button>
      </div>
    </BottomSheet>
  )
}

async function TournamentListSection({ tab, q, dateFrom, dateTo, role }) {
  // 1. 신청 목록 조회 (인증 사용자만)
  const appsResult = role ? await listAllMyTeamApplications() : null
  const allApps = appsResult?.data ?? []

  // 2. myApplicationMap 구성 (STATUS_PRIORITY 기반)
  const myApplicationMap = buildApplicationMap(allApps)

  // 3. 대회 목록 조회
  let tournaments: TournamentListItem[] = []
  let error: string | null = null

  if (tab === 'mine') {
    // 활성 신청의 tournament_id 추출
    const myTournamentIds = getActiveApplicationTournamentIds(allApps)
    if (myTournamentIds.length > 0) {
      const result = await getTournamentList({ ids: myTournamentIds })
      tournaments = result.data ?? []
      error = result.error
    }
  } else {
    const statusFilter = tabToStatus(tab)  // 'all' → undefined, 'open' → 'open' 등
    const result = await getTournamentList({ status: statusFilter, keyword: q, dateFrom, dateTo })
    tournaments = result.data ?? []
    error = result.error
  }

  // 4. 렌더링
  ...
}
```

**myApplicationMap 구성 주의사항:**

한 사용자가 같은 대회에 여러 팀으로 신청할 수 있다 (`team_members` N:M 구조).  
Map 구성 시 활성 상태(`confirmed` > `waitlisted` > `paid_pending_approval` > `payment_pending`)를 우선으로 단 하나만 선택:

```
const STATUS_PRIORITY: Record<ApplicationStatus, number> = {
  confirmed: 0,
  waitlisted: 1,
  paid_pending_approval: 2,
  payment_pending: 3,
  cancelled: 4,
  expired: 5,
}

for (const app of allApps) {
  const existing = myApplicationMap.get(app.tournament_id)
  if (!existing || STATUS_PRIORITY[app.status] < STATUS_PRIORITY[existing.status]) {
    myApplicationMap.set(app.tournament_id, app)
  }
}
```

## 6. 리스크 및 주의사항

### 6-1. 동일 대회에 여러 팀 신청 시 myApplicationMap 충돌

**문제:** `listAllMyTeamApplications()`는 captain+player 모든 팀의 신청을 반환. 한 사용자가 여러 팀의 captain이거나 팀 A에서 captain, 팀 B에서 player인 경우 같은 `tournament_id`에 대해 여러 row가 반환될 수 있음.

**해결:** Map 구성 시 STATUS_PRIORITY 기준으로 가장 활성 상태 신청 하나만 선택 (Phase 4-8 참고).

### 6-2. Supabase `.or()` + `.ilike()` 쿼리 조합

**문제:** `keyword`로 `name`과 `location`을 동시에 검색할 때 `.or('name.ilike.%keyword%,location.ilike.%keyword%')` 형태로 작성해야 한다. PostgREST에서 `%`가 URL 인코딩 처리되어 이스케이프가 필요할 수 있음.

**해결:** Supabase JS SDK는 `.or()`에 전달하는 문자열을 자동으로 처리함. 단, `%` 포함 값은 Supabase 내부적으로 처리되므로 추가 이스케이프 불필요. 실제 동작을 반드시 테스트.

### 6-3. TournamentListSection async 서버 컴포넌트와 Suspense key

**문제:** `page.tsx`에서 searchParams가 변경될 때 Suspense 경계가 새 fallback을 보여주려면 `key` prop이 변경되어야 한다. `key`가 없으면 기존 결과를 유지하면서 백그라운드 업데이트가 되어 로딩 상태가 표시되지 않을 수 있음.

**해결:** `<Suspense key={searchParamsKey}>` 패턴 적용.  
`searchParamsKey = [tab, q ?? '', dateFrom ?? '', dateTo ?? ''].join('|')`

> **v1 대비 변경**: `region` 제거됨.

### 6-4. `Button` 컴포넌트 size prop 없음

**문제:** `Button` 컴포넌트 (`components/ui/Button.tsx`)에 `size` prop이 없어 카드 내 소형 버튼을 구현할 때 불일치.

**해결:** `Button`에 `size` prop을 추가하는 것은 기존 코드에 영향을 줄 수 있으므로, `className`으로 오버라이드:
```tsx
<Button variant="secondary" className="text-xs px-3 py-1.5">상세보기</Button>
```
`Button`의 `baseClasses`가 `inline-flex items-center justify-center ... px-4 py-2 text-sm`이므로 `className`으로 덮어씀.

### 6-5. Supabase 관계형 select에서 divisions 배열 반환

**문제:** `getTournamentList()`에서 `divisions(id, name, entry_fee, capacity, sort_order)` 관계 select 시 PostgREST가 단일 row의 경우도 배열로 반환하는지 확인 필요. 반환 타입 단언(`as unknown as TournamentListItem[]`) 시 divisions 필드 처리 주의.

**해결:** 기존 `getMyParticipatedTournaments()`의 관계형 select 데이터 처리 패턴 (`as Record<string, unknown>` → 필드별 타입 단언) 참고. 또는 결과의 각 row에서 `row.divisions ?? []`로 안전하게 처리.

### 6-6. searchParams 타입 처리 (Next.js App Router)

**문제:** Next.js 15+에서 `page.tsx`의 searchParams가 `Promise<Record<string, string | string[]>>`로 변경됨. 현재 프로젝트의 다른 page.tsx들이 어떻게 처리하는지 확인 필요.

**해결:** 기존 `app/(app)/admin/page.tsx`의 searchParams 처리 패턴을 그대로 따름. `await params`가 필요한 경우 동일하게 적용.

### 6-7. 모바일 바텀시트 body overflow 제어 (SSR 환경)

**문제:** `document.body.style.overflow`는 클라이언트 전용 API. `"use client"` 없는 상황에서 접근하면 에러.

**해결:** `BottomSheet.tsx`는 반드시 `"use client"`로 선언하고, `useEffect` 내에서만 `document.body` 접근. `FilterBottomSheet`에서는 별도 처리 불필요 (`BottomSheet`가 담당).

### 6-8. 내 신청 탭에서 취소/만료된 신청 처리

**문제:** `mine` 탭에서 `cancelled`/`expired` 신청을 포함할지 여부가 UX에 영향.

**해결:** `mine` 탭은 활성 신청(`payment_pending`, `paid_pending_approval`, `confirmed`, `waitlisted`)이 있는 대회만 표시. 취소/만료만 있는 대회는 제외.
