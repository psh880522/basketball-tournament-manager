#### 마이그레이션 1: `supabase/migrations/0230_divisions_application_config.sql`

**목적**: `divisions` 테이블에 신청 관련 컬럼 추가

**신규 컬럼**:

| 컬럼명 | 타입 | 기본값 | 설명 |
|---|---|---|---|
| `entry_fee` | integer | 0 | 기본 참가비 (원 단위) |
| `capacity` | integer | NULL | 정원 (NULL = 무제한) |
| `application_open_at` | timestamptz | NULL | 신청 시작 시각 (NULL = tournament.status에만 의존) |
| `application_close_at` | timestamptz | NULL | 신청 마감 시각 (NULL = tournament.status에만 의존) |

**`application_open_at` / `application_close_at` 설계 원칙** (**v2 추가**)

현재 신청 가능 여부는 `tournament.status = 'open'`으로만 제어된다. 이 두 필드는 해당 메커니즘을 **대체하지 않고 추가적으로** 동작한다.

- `NULL`(기본값): 신청 기간 제어를 `tournament.status`에만 위임. 기존 동작과 동일.
- 값이 설정된 경우: `tournament.status = 'open'` 조건을 **통과한 이후**, division별로 추가 시간 범위를 제한.

**실제 신청 가능 조건** (RPC 내 검증 순서):
1. `tournament.status = 'open'` ← 기존 1차 게이트, RPC에서도 반드시 확인
2. `division.application_open_at IS NULL OR now() >= application_open_at` ← division 시작 시각
3. `division.application_close_at IS NULL OR now() <= application_close_at` ← division 마감 시각

MVP에서는 두 필드를 운영자가 선택적으로 설정할 수 있도록 UI를 제공하되, NULL로 두면 기존처럼 tournament.status만으로 동작한다.

**제약 추가**:
- `divisions_capacity_check`: `capacity IS NULL OR capacity >= 0`
- `divisions_entry_fee_check`: `entry_fee >= 0`

**롤백**:
```sql
ALTER TABLE divisions DROP COLUMN IF EXISTS entry_fee;
ALTER TABLE divisions DROP COLUMN IF EXISTS capacity;
ALTER TABLE divisions DROP COLUMN IF EXISTS application_open_at;
ALTER TABLE divisions DROP COLUMN IF EXISTS application_close_at;
```

#### 마이그레이션 4: `supabase/migrations/0233_applications_submit_rpc.sql`

**목적**: 신청 생성 RPC — 정원 확인·동시성 처리·이력 기록을 원자적으로 처리

**함수 `submit_tournament_application(p_tournament_id, p_team_id, p_division_id, p_user_id) → jsonb`**:

핵심 의사코드 (**v2 수정: tournament.status 체크 추가, division 날짜 조건 NULL 처리 명시**):
```
BEGIN
  -- [추가] 1차 게이트: tournament.status 확인 (기존 applyToTournament()과 동일한 검증)
  SELECT status INTO v_tournament_status FROM tournaments WHERE id = p_tournament_id
  IF NOT FOUND OR v_tournament_status != 'open' → return error "현재 신청 가능한 대회가 아닙니다"

  -- [추가] team_manager 확인
  IF NOT EXISTS (team_members WHERE team_id = p_team_id AND user_id = p_user_id AND role_in_team = 'manager')
    → return error "이 팀의 매니저만 신청할 수 있습니다"

  SELECT * FROM divisions WHERE id = p_division_id AND tournament_id = p_tournament_id FOR UPDATE  ← 행 잠금
  IF NOT FOUND → return error "유효하지 않은 division"

  -- 2차 게이트: division별 신청 기간 (NULL이면 스킵 — tournament.status로만 제어)
  IF application_open_at IS NOT NULL AND now() < application_open_at
    → return error "이 부문의 신청이 아직 시작되지 않았습니다"
  IF application_close_at IS NOT NULL AND now() > application_close_at
    → return error "이 부문의 신청이 마감되었습니다"

  occupied = COUNT WHERE division_id = ? AND status IN ('payment_pending', 'paid_pending_approval', 'confirmed')

  IF capacity IS NULL OR occupied < capacity
    → status = 'payment_pending', payment_due_at = now() + 24h, waitlist_position = NULL
  ELSE
    → status = 'waitlisted', waitlist_position = MAX(waitlist_position) + 1, payment_due_at = NULL

  INSERT tournament_team_applications ← base_entry_fee = division.entry_fee, final_amount = entry_fee
  INSERT application_status_history (from_status = NULL, to_status = status)
  RETURN {ok: true, status, application_id}
EXCEPTION unique_violation → return error "이미 이 디비전에 신청이 존재합니다"
END
```

**SECURITY DEFINER**, `SET row_security = off` 사용 (기존 `create_team_with_manager()` RPC 패턴 동일)

**롤백**: `DROP FUNCTION IF EXISTS submit_tournament_application;`

#### 마이그레이션 5: `supabase/migrations/0234_applications_transition_rpcs.sql`

**목적**: 상태 전환 RPC 묶음

**함수 1: `promote_next_waitlisted(p_division_id uuid) → void`**

```
의사코드:
  SELECT id, applied_by FROM tournament_team_applications
    WHERE division_id = p_division_id AND status = 'waitlisted'
    ORDER BY waitlist_position ASC LIMIT 1
    FOR UPDATE SKIP LOCKED
  IF NOT FOUND → return (no-op)
  UPDATE status = 'payment_pending', payment_due_at = now() + 12h, waitlist_position = NULL
  INSERT application_status_history (from='waitlisted', to='payment_pending')
```

**함수 2: `mark_payment_done(p_application_id uuid, p_user_id uuid, p_depositor_name text, p_depositor_note text) → jsonb`**

```
의사코드:
  SELECT * FROM tournament_team_applications WHERE id = ? FOR UPDATE
  IF NOT FOUND → error "신청을 찾을 수 없습니다"
  IF status != 'payment_pending' → error "입금 대기 상태가 아닙니다"
  IF applied_by != p_user_id → error "본인 신청만 처리 가능합니다"
  UPDATE status = 'paid_pending_approval', depositor_name, depositor_note, paid_marked_at = now()
  INSERT application_status_history (from='payment_pending', to='paid_pending_approval')
  RETURN {ok: true}
```

**함수 3: `cancel_application(p_application_id uuid, p_user_id uuid) → jsonb`**

```
의사코드:
  SELECT * FROM tournament_team_applications WHERE id = ? FOR UPDATE
  IF NOT FOUND → error
  -- 본인 팀의 매니저인지 확인
  IF NOT EXISTS (team_members WHERE team_id = app.team_id AND user_id = p_user_id AND role_in_team = 'manager')
    → error "취소 권한이 없습니다"
  IF status NOT IN ('payment_pending', 'paid_pending_approval', 'confirmed', 'waitlisted')
    → error "취소할 수 없는 상태입니다"

  prev_status = status
  UPDATE status = 'cancelled', cancelled_at = now()
  INSERT application_status_history (from=prev_status, to='cancelled')

  IF prev_status IN ('payment_pending', 'paid_pending_approval', 'confirmed')
    → CALL promote_next_waitlisted(division_id)

  RETURN {ok: true}
```

**함수 4: `confirm_application(p_application_id uuid, p_admin_user_id uuid, p_memo text) → jsonb`** (운영자 전용)

```
의사코드:
  SELECT * FROM tournament_team_applications WHERE id = ? FOR UPDATE
  IF NOT FOUND → error
  IF status NOT IN ('payment_pending', 'paid_pending_approval') → error "확인 가능한 상태가 아닙니다"
  prev_status = status
  UPDATE status = 'confirmed', confirmed_at = now(), approved_by = p_admin_user_id, approved_at = now(), admin_memo = p_memo
  INSERT application_status_history (from=prev_status, to='confirmed', changed_by=p_admin_user_id)
  RETURN {ok: true}
```

**함수 5: `admin_cancel_application(p_application_id uuid, p_admin_user_id uuid, p_memo text) → jsonb`** (운영자 전용)

```
의사코드:
  SELECT * FROM tournament_team_applications WHERE id = ? FOR UPDATE
  IF NOT FOUND → error
  IF status IN ('expired', 'cancelled') → error "이미 종료된 신청입니다"
  prev_status = status
  UPDATE status = 'cancelled', cancelled_at = now(), admin_memo = p_memo
  INSERT application_status_history (from=prev_status, to='cancelled', changed_by=p_admin_user_id)
  IF prev_status IN ('payment_pending', 'paid_pending_approval', 'confirmed')
    → CALL promote_next_waitlisted(division_id)
  RETURN {ok: true}
```

**함수 6: `expire_overdue_applications() → integer`** (Edge Function에서 호출)

```
의사코드:
  SELECT id, division_id, applied_by FROM tournament_team_applications
    WHERE status = 'payment_pending' AND payment_due_at < now()
    FOR UPDATE SKIP LOCKED
  FOR EACH app:
    UPDATE status = 'expired', expired_at = now()
    INSERT application_status_history (from='payment_pending', to='expired', changed_by=app.applied_by)
    INSERT notifications (user_id=app.applied_by, type='application_expired', ...)
    CALL promote_next_waitlisted(app.division_id)
  RETURN count of expired rows
```

#### `lib/api/applications.ts` — 타입 수정

**`ApplicationStatus`**: 6개로 전면 교체

type ApplicationStatus =

const OCCUPYING_STATUSES = ['payment_pending', 'paid_pending_approval', 'confirmed']

|---|---|---|
| `submit_tournament_application` | authenticated | team_manager 여부 + tournament.status DB 내 확인 |
| `mark_payment_done` | authenticated | applied_by = p_user_id 확인 |
| `cancel_application` | authenticated | team_members 매니저 확인 |
| `confirm_application` | authenticated | is_organizer() 확인 (또는 서버 액션에서 guard) |
| `admin_cancel_application` | authenticated | is_organizer() 확인 |
| `promote_next_waitlisted` | authenticated | 내부 헬퍼, 직접 호출 제한 불필요 |
| `expire_overdue_applications` | service_role | Edge Function에서만 호출 |

## 6. 리스크 및 주의사항

### 리스크 1: 기존 'approved' 상태 데이터 마이그레이션

**위험도**: 높음  
**내용**: 마이그레이션 0231에서 `'approved'` → `'confirmed'`로 상태 변환. 이 시점부터 코드는 `'confirmed'`를 사용하므로, **마이그레이션 적용과 코드 배포를 동시에** 진행해야 함. 마이그레이션 적용 후 구코드가 `'approved'`로 조회하면 데이터 불일치 발생.

**대응**: 배포 시 마이그레이션 적용 직후 코드 교체를 한 번에 수행. 로컬 개발 환경에서 먼저 전체 검증.

---

### 리스크 2: UNIQUE 제약 변경

**위험도**: 높음  
**내용**: `UNIQUE(tournament_id, team_id)` 제거 후 Partial Index `(team_id, division_id) WHERE status IN (...)` 추가. 기존 코드에서 `(tournament_id, team_id)` uniqueness를 가정하는 부분이 있을 수 있음.

**대응**: 
- `getMyApplicationStatus()`에서 `.limit(1)`로 1건만 가져오는 것을 확인하고, divisionId 기준 조회로 변경
- `applyToTournament()`의 `error.code === "23505"` 처리 에러 메시지 확인 (division 단위 중복 메시지로 변경)

---

### 리스크 3: 동시 신청 Race Condition

**위험도**: 높음  
**내용**: `submit_tournament_application()` RPC 외부에서 신청 생성 시 정원 초과 가능.

**대응**:
- 모든 신청 생성 경로를 `submit_tournament_application()` RPC로만 처리
- 기존 `applyToTournament()` 함수 내부를 RPC 호출로 교체
- 더미팀 추가(`createDummyTeamAction`)도 RPC 또는 서버 액션에서 occupied 체크 추가

---

### 리스크 4: promote_next_waitlisted 누락

**위험도**: 중간  
**내용**: 자리를 반환하는 모든 경로(사용자 취소, 운영자 취소, 만료 처리)에서 `promote_next_waitlisted()` 호출이 빠지면 대기자가 영구 대기 상태에 빠짐.

**대응**: `cancel_application`, `admin_cancel_application`, `expire_overdue_applications` RPC 모두에서 `promote_next_waitlisted()` 호출 코드 반드시 포함. 코드 리뷰 체크리스트로 관리.

---

### 리스크 5: Edge Function 실행 실패

**위험도**: 중간  
**내용**: Edge Function이 실패하거나 cron이 멈추면 `payment_pending` 상태가 기한 초과 후에도 영구적으로 유지됨.

**대응**:
- Edge Function 내 에러 핸들링 및 로깅
- Supabase Dashboard에서 Function 실행 로그 모니터링
- 운영자가 수동으로 만료 처리할 수 있는 버튼 추가 (선택적)

---

### 리스크 6: 알림 실패가 메인 플로우를 막음

**위험도**: 낮음  
**내용**: `createNotification()` 내부에서 DB INSERT 실패 시 예외 전파 시 상태 전환 자체가 롤백됨.

**대응**: `lib/api/notifications.ts`의 `createNotification()`에서 에러 발생 시 `throw` 하지 않고 `console.error`로만 기록. 알림은 best-effort.

---

### 리스크 7: 카운트다운 UI와 서버 만료 시각 불일치

**위험도**: 낮음  
**내용**: `StatusCard.tsx`에서 `payment_due_at` 기준 클라이언트 측 카운트다운이 0이 되어도, Edge Function이 아직 실행되지 않아 DB 상태는 `payment_pending`일 수 있음.

**대응**: UI에서 "기한이 지났습니다. 잠시 후 상태가 업데이트됩니다." 안내 문구 표시. 상태 새로고침 버튼 제공. 실제 만료 처리는 서버 기준임을 명확히.

---

### 리스크 8: downstream 코드의 'approved' 잔재

**위험도**: 중간  
**내용**: bracket, matches, standings 등 기존 하위 기능이 `status = 'approved'` 로 팀 목록을 조회할 수 있음. 마이그레이션 후 해당 조회가 0건 반환됨 → 브래킷 생성, 경기 배정 등 기능 파손.

**대응**: 구현 전 `rg "'approved'" --include="*.ts" --include="*.tsx" app/ lib/`로 전수 검색 후 목록 작성. Phase 3-A에서 일괄 변경.

---

### 리스크 9: application_open_at/close_at과 tournament.status 이중 게이트 혼선 (**v2 추가**)

**위험도**: 낮음  
**내용**: `submit_tournament_application` RPC가 division 날짜만 체크하고 `tournament.status`를 체크하지 않으면, tournament가 'closed' 상태일 때도 신청이 가능해질 수 있음.

**대응**: RPC 의사코드에 `tournament.status = 'open'` 체크를 1차 게이트로 명시(v2에서 반영 완료). 구현 시 해당 체크가 RPC 내 **첫 번째**로 실행되도록 순서 주의.

### 구현 난이도 요약

| Phase | 난이도 | 핵심 위험 |
|---|---|---|
| Phase 1 (DB) | 높음 | 기존 데이터 마이그레이션, UNIQUE 제약 변경 |
| Phase 2 (타입) | 낮음 | 없음 |
| Phase 3-A (downstream 수정) | 중간 | 'approved' 잔재 누락 |
| Phase 3-B/C (API/액션) | 중간 | RPC 호출 통일, 알림 best-effort 처리 |
| Phase 3-D (Edge Function) | 중간 | cron 설정, 실패 모니터링 |
| Phase 4 (프론트엔드) | 중간 | 6가지 상태 UI 분기, 카운트다운, division 편집 폼 추가 |
| Phase 5 (RLS) | 낮음 | Phase 1에 포함 |

### 권장 구현 접근법

1. **로컬 Supabase 환경**에서 마이그레이션 0231 먼저 검증 (기존 데이터 변환 확인)
2. **'approved' → 'confirmed' downstream 수정**을 Phase 1 직후 Phase 3-A로 즉시 처리하여 코드-DB 정합성 유지
3. **RPC 단위 테스트**: 각 RPC에 대해 Supabase SQL Editor에서 직접 호출 테스트
4. **E2E 시나리오 테스트**: 신청 → 입금 완료 표시 → 운영자 확인 → 확정 전체 흐름 수동 확인
5. **Edge Function**은 Phase 4 완료 후 별도로 배포 및 검증

### 후속 확장 포인트 (이번 구현에서 확보)

- `final_amount`, `discount_amount` 컬럼 → PG 결제 연동 시 결제 금액으로 활용
- `notifications` 테이블의 `type` + `data` → 채널 확장 시 라우팅 기준으로 활용
- `application_status_history` → 운영자 감사(audit) 로그 및 분쟁 해결 기준
- `payment_due_at` → 향후 정책 테이블로 대체 가능
- `application_open_at/close_at` → 현재 NULL 기본값으로 tournament.status에 의존하지만, 향후 division별 세밀한 모집 기간 운영 시 활용 가능
