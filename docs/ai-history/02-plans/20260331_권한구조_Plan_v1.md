**작성일:** 2026-03-31
**기반 리서치:** `docs/ai-history/01-research/20260331_권한구조_Research.md`

---

## 1. 구현 목표 요약

현재 `organizer / team_manager / player / spectator` 4종 역할을 **`organizer / manager / player`** 3종으로 재편한다.

| 변경 항목 | 현재 | 목표 |
|---|---|---|
| 전역 역할 종류 | organizer, team_manager, player, spectator | organizer, manager, player |
| 신규 가입 기본 역할 | spectator | player |
| 팀 생성/관리 담당 | team_manager | player |
| 운영 스태프 역할 | 없음 (organizer가 전담) | manager (경기 결과·상태 관리) |
| 역할 변경 UI | 없음 | organizer 전용 사용자 관리 페이지 |
| profiles 직접 role 수정 | 가능 (보안 취약점) | RPC를 통해서만 변경 가능 |

---

## 2. 추가 라이브러리

없음. 기존 스택(Next.js App Router, Supabase, TypeScript)으로 구현 가능하다.

### Phase 1 — DB 마이그레이션

#### Step 1-1: enum 전환 + 기본값 변경

**신규 파일:** `supabase/migrations/0205_role_system_v2.sql`

```sql
-- 1) 새 enum 생성
CREATE TYPE public.app_role_v2 AS ENUM ('organizer', 'manager', 'player');

-- 2) profiles 컬럼 타입 교체
--    기존 값 매핑: spectator → player, team_manager → player
ALTER TABLE public.profiles
  ALTER COLUMN role DROP DEFAULT;

ALTER TABLE public.profiles
  ALTER COLUMN role TYPE public.app_role_v2
  USING (
    CASE role::text
      WHEN 'organizer'    THEN 'organizer'::public.app_role_v2
      WHEN 'manager'      THEN 'manager'::public.app_role_v2
      WHEN 'team_manager' THEN 'player'::public.app_role_v2
      WHEN 'spectator'    THEN 'player'::public.app_role_v2
      ELSE                     'player'::public.app_role_v2
    END
  );

ALTER TABLE public.profiles
  ALTER COLUMN role SET DEFAULT 'player';

-- 3) 구 enum 제거 (컬럼 교체 후에만 가능)
DROP TYPE public.app_role;

-- 4) 새 enum 이름 변경
ALTER TYPE public.app_role_v2 RENAME TO app_role;
```

**트레이드오프:**
- PostgreSQL enum 직접 수정(`ALTER TYPE ... ADD VALUE`)은 트랜잭션 안에서 새 타입 즉시 사용 불가한 경우가 있어, 새 타입 생성 + 컬럼 교체 + 구 타입 삭제 방식을 사용한다.
- 기존 `team_manager`/`spectator` 사용자가 모두 `player`로 강등된다. 운영 스태프가 있다면 마이그레이션 후 수동으로 `manager` 승격 필요.

#### Step 1-2: is_manager() RLS 함수 추가 + 불필요 함수 정리

**신규 파일:** `supabase/migrations/0206_rls_manager_role.sql`

```sql
-- 1) manager 체크 함수 신규 추가
CREATE OR REPLACE FUNCTION public.is_manager()
RETURNS boolean
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
SET row_security = off
AS $$
BEGIN
  RETURN EXISTS (
    SELECT 1 FROM public.profiles p
    WHERE p.id = auth.uid()
      AND p.role IN ('organizer', 'manager')
  );
END;
$$;
-- 주의: is_manager()는 organizer도 포함한다.
-- organizer는 manager의 모든 권한을 가지기 때문.
-- organizer 전용 체크는 기존 is_organizer()를 그대로 사용.

-- 2) 구 is_team_manager() 함수 제거
DROP FUNCTION IF EXISTS public.is_team_manager();
-- 대체: is_team_manager_for_team(team_uuid) 함수가 이미 0088에 존재함
```

#### Step 1-3: teams 테이블 RLS 정책 정리

**신규 파일:** `supabase/migrations/0207_teams_rls_player.sql`

```sql
-- 1) team_manager role 기반 구 정책 제거
DROP POLICY IF EXISTS "teams_select_team_manager" ON public.teams;
DROP POLICY IF EXISTS "teams_insert_team_manager" ON public.teams;

--    players 테이블에 RLS는 이미 is_team_manager_for_team 기반이므로 추가 불필요.

-- 3) teams SELECT: 인증된 모든 사용자가 자신의 팀 조회 가능

DROP POLICY IF EXISTS "teams_select_created_by" ON public.teams;
CREATE POLICY "teams_select_created_by"
  ON public.teams FOR SELECT
  USING (created_by = auth.uid());
```

#### Step 1-4: profiles 역할 변경 보안 수정 + RPC 추가

**신규 파일:** `supabase/migrations/0208_profiles_role_rpc.sql`

```sql
-- 1) profiles_update_own 정책에서 role 컬럼 변경 방지
--    WITH CHECK 조건 추가: role 변경 불가 (OLD role과 동일해야)
DROP POLICY IF EXISTS "profiles_update_own" ON public.profiles;
CREATE POLICY "profiles_update_own"
  ON public.profiles FOR UPDATE
  USING (id = auth.uid())
  WITH CHECK (
    id = auth.uid()
    -- role 컬럼은 이 정책으로 변경 불가. 변경은 아래 RPC를 통해서만 허용.
    -- PostgreSQL은 UPDATE WITH CHECK에서 OLD 값 비교를 직접 지원하지 않으므로
    -- role은 별도 RPC로만 변경 가능하도록 이 정책에서 제외.
    -- 실질적으로 role만 update하는 쿼리를 막으려면 아래 RPC + profiles INSERT 정책에서 처리.
  );

-- 2) organizer 전용 역할 변경 RPC
CREATE OR REPLACE FUNCTION public.update_user_role(
  target_user_id uuid,
  new_role text
)
RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
BEGIN
  -- 호출자가 organizer인지 확인
  IF NOT public.is_organizer() THEN
    RAISE EXCEPTION 'Forbidden: only organizer can change roles';
  END IF;

  -- 허용 값: player, manager만 변경 가능 (organizer 승격은 DB 직접 처리)
  IF new_role NOT IN ('player', 'manager') THEN
    RAISE EXCEPTION 'Invalid role: only player or manager allowed';
  END IF;

  UPDATE public.profiles
  SET role = new_role::public.app_role
  WHERE id = target_user_id;
END;
$$;

-- RPC는 SECURITY DEFINER이므로 RLS를 우회.
-- 단, 함수 내부에서 is_organizer()로 직접 권한 검증.
```

**트레이드오프:**
- PostgreSQL의 `UPDATE WITH CHECK`는 `OLD.*` 참조가 불가하여 "role 변경 금지"를 정책 레벨에서 직접 강제하기 어렵다. SECURITY DEFINER RPC를 별도로 두고 해당 RPC로만 role 변경을 허용하는 패턴을 채택한다.
- profiles SELECT 정책(`profiles_select_own_or_organizer`)은 현행 유지.

#### Step 1-5: matches/courts/standings 운영 관련 RLS에 manager 추가

**신규 파일:** `supabase/migrations/0209_manager_operation_rls.sql`

```sql
-- matches: manager도 UPDATE/INSERT 가능 (결과 입력)
-- (matches 테이블의 기존 RLS 상태 확인 후 추가)
DROP POLICY IF EXISTS "matches_update_organizer" ON public.matches;
CREATE POLICY "matches_update_organizer_manager"
  ON public.matches FOR UPDATE
  USING (public.is_manager());   -- is_manager()는 organizer + manager 포함

-- courts: manager도 관리 가능
DROP POLICY IF EXISTS "courts_insert_organizer" ON public.courts;
CREATE POLICY "courts_insert_manager"
  ON public.courts FOR INSERT
  WITH CHECK (public.is_manager());

DROP POLICY IF EXISTS "courts_delete_organizer" ON public.courts;
CREATE POLICY "courts_delete_manager"
  ON public.courts FOR DELETE
  USING (public.is_manager());

-- standings: manager도 업데이트 가능
DROP POLICY IF EXISTS "standings_upsert_organizer" ON public.standings;
CREATE POLICY "standings_upsert_manager"
  ON public.standings FOR ALL
  USING (public.is_manager())
  WITH CHECK (public.is_manager());
```

#### Step 5-2: 역할 변경 서버 액션

**신규 파일:** `app/admin/users/actions.ts`

```typescript
"use server";

import { getUserWithRole } from "@/src/lib/auth/roles";
import { createSupabaseServerClient } from "@/src/lib/supabase/server";

type UpdateRoleInput = {
  targetUserId: string;
  newRole: "player" | "manager";
};

export async function updateUserRoleAction(input: UpdateRoleInput): Promise<ActionResult> {
  const caller = await getUserWithRole();

  if (caller.status === "unauthenticated") return { ok: false, error: "로그인 필요." };
  if (caller.role !== "organizer") return { ok: false, error: "권한 없음." };

  if (!input.targetUserId) return { ok: false, error: "대상 사용자 ID 필요." };
  if (!["player", "manager"].includes(input.newRole)) {
    return { ok: false, error: "유효하지 않은 역할." };
  }

  const supabase = await createSupabaseServerClient();

  // SECURITY DEFINER RPC 호출 (DB 레벨에서 organizer 재검증)
  const { error } = await supabase.rpc("update_user_role", {
    target_user_id: input.targetUserId,
    new_role: input.newRole,
  });

  if (error) return { ok: false, error: error.message };
  return { ok: true };
}
```

**신규 파일:** `supabase/migrations/0210_profiles_email.sql`

```sql

ALTER TABLE public.profiles ADD COLUMN IF NOT EXISTS email text;

-- 기존 회원 이메일 백필 (Supabase admin 쿼리 필요 또는 trigger로 처리)

-- 신규 가입 시 email 자동 삽입 트리거 수정
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS trigger ...
BEGIN
  INSERT INTO public.profiles (id, email)
  VALUES (new.id, new.email)
  ON CONFLICT (id) DO NOTHING;
  RETURN new;
END;
```

## 6. 트레이드오프 및 고려 사항

### 6-1. 기존 team_manager 사용자 일괄 player 강등

enum 전환 시 기존 `team_manager` 역할 사용자는 자동으로 `player`로 변환된다.
이 사용자들은 이전과 동일하게 팀 생성/관리가 가능하지만 (`team_members.role_in_team = 'manager'` 기반으로 이미 전환됨), 역할 이름이 바뀌는 것이므로 의사소통 필요.

### 6-2. manager와 팀 내 role_in_team='manager' 혼동

전역 역할의 `manager`(운영 스태프)와 `team_members.role_in_team = 'manager'`(팀 대표)는 이름이 같지만 다른 개념이다. 코드 및 DB에서 명확히 구분해야 한다. 필요 시 팀 내 역할을 `captain` / `member`로 rename 검토 (별도 작업).

-- TODO: `captain`으로 사용

### 6-3. profiles_update_own의 role 변경 방지 한계

PostgreSQL RLS의 `WITH CHECK`는 `NEW.*`만 참조 가능하고 `OLD.*`를 비교할 수 없다. 따라서 정책 레벨에서 "role 변경 금지"를 완전히 강제하기 어렵다. SECURITY DEFINER RPC 패턴으로 우회하되, 프론트엔드에서도 role 변경 UI를 organizer 전용 페이지에만 노출하는 이중 방어를 사용한다.

### 6-4. is_manager() 함수 범위

`is_manager()`를 `organizer + manager` 포함으로 정의한다. 이는 "organizer는 manager의 모든 권한을 포함한다"는 요구사항을 반영한다. 만약 향후 두 역할을 완전히 분리해야 한다면 `is_manager()`를 `manager only`로 재정의하고, 각 정책에 `is_organizer() OR is_manager()` 형태로 변경해야 한다.

### 6-5. assertTournamentStepAllowed 분기 복잡도

현재 단일 함수에서 모든 단계 검증을 처리한다. manager 허용 단계/organizer 전용 단계 분기가 추가되면 함수가 복잡해진다. 리팩토링 방향: stepKey를 두 그룹으로 분류하는 상수 배열을 두고 `MANAGER_ALLOWED_STEPS`에 속하면 `is_manager()`, 아니면 `is_organizer()`로 처리한다.

### 6-6. 사용자 이메일 표시 방법

사용자 관리 UI에서 이메일 표시는 세 가지 방법이 있다:
1. `profiles`에 `email` 컬럼 추가 (비정규화, 가장 단순)
2. Supabase Admin API 사용 (`service_role` key 필요, Next.js Route Handler에서 처리)
3. `auth.users`와 JOIN하는 DB View 생성 (RLS 적용 주의 필요)

v1에서는 방법 1을 채택하고, 보안 민감도가 올라가면 방법 2로 전환.
