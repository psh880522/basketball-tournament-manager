## 1. 구현 기능 상세 설명

### 1-1. 현재 로그인 액션의 이동 로직

`app/(auth)/login/actions.ts` 현재 흐름:

```
signIn() → getCurrentUserRole() → isOperationRole(role) ? "/admin" : "/dashboard"
```

역할 분기만 있고 프로필 완료 여부는 확인하지 않는다.

---

### 1-2. 변경 후 분기 순서

```
signIn() 성공
  ↓
1. getCurrentUserRole() → role 확인
  ↓
2. isOperationRole(role) → true → redirect "/admin"   ← 운영자/매니저 여기서 끝
  ↓
3. (player 대상만) getMyProfile() 호출
  ↓
4. isProfileCompleted(profile) → false → redirect "/onboarding/profile"
  ↓
5. (프로필 완료) → redirect "/dashboard"
  
  [VS-5 이후 추가 위치: 4와 5 사이에 본인인증 분기 삽입]
```

**분기 정책 요약**:
- `organizer` / `manager`: 프로필 완료 여부 확인 없이 바로 `/admin`
- `player` (기본): 프로필 미완료 → `/onboarding/profile`, 완료 → `/dashboard`

운영자/매니저는 시스템 운영 역할이므로 프로필 온보딩 흐름을 강제하지 않는다. 필요하다면 `/admin` 진입 후 별도 배너나 설정 페이지에서 유도한다.

---

### 1-3. 프로필 완료 여부 판단 기준

`isProfileCompleted(profile)` 함수 (VS-1에서 `lib/api/profiles.ts`에 이미 구현):
- `display_name`이 비어 있지 않고
- `phone`이 비어 있지 않으면 → 완료
- `birth_date`는 선택 항목 — 판단 기준에 포함하지 않음

---

### 1-4. 프로필 미완료 시 처리

`redirect("/onboarding/profile")` — VS-2에서 이미 구현된 페이지.  
이 redirect는 Server Action에서 `redirect()`를 직접 호출한다 (응답을 클라이언트에 반환하기 전에 처리).

---

### 1-5. `profiles` 행이 없거나 비정상 상태일 때

`getMyProfile()` 반환 케이스별 처리:

| 케이스 | 조건 | 처리 |
|--------|------|------|
| 정상 (신규 가입) | `data.display_name === null` | `isProfileCompleted()` → false → `/onboarding/profile` |
| 정상 (기존 완료) | `data.display_name`, `data.phone` 모두 존재 | `/dashboard` |
| `data === null` | `maybeSingle()` 결과 없음 (`empty` 상태) | `isProfileCompleted(null)` → false → `/onboarding/profile` |
| `error !== null` | DB 오류 | `{ ok: false, error: "..." }` 반환 — redirect 없이 로그인 폼에 오류 표시 |

> `profiles` 행 없는 `empty` 케이스: `on_auth_user_created` 트리거가 정상 동작하면 이론적으로 발생하지 않는다. 그러나 에러로 처리하지 않고 온보딩 대상으로 취급하는 것이 사용자 경험상 더 안전하다.

---

### 1-6. 향후 본인인증 분기(VS-5) 삽입 위치

```typescript
// VS-5 이후 추가될 위치 (코드 주석으로 명시)
// [VS-5] if (!profile.is_identity_verified) redirect("/identity");
redirect("/dashboard");
```

프로필 완료 확인 직후, `/dashboard` redirect 직전에 삽입하면 된다.  
이번 VS-3에서는 해당 라인을 주석으로 예약 표시하지 않고, 구조적으로 삽입 가능한 형태로만 작성한다.

## 4. 파일별 핵심 로직 구조

### 4-1. `app/(auth)/login/actions.ts`

**역할**: 로그인 성공 후 사용자 상태에 따라 이동 경로를 분기하는 Server Action

**현재 import** (변경 없음):
```typescript
import { getCurrentUserRole, signInWithPassword as signIn } from "@/lib/api/auth";
import { isOperationRole } from "@/src/lib/auth/roles";
```

**추가할 import**:
```typescript
import { getMyProfile, isProfileCompleted } from "@/lib/api/profiles";
```

**핵심 로직 구조 (pseudocode)**:

```typescript
export async function signInWithPassword(input): Promise<ActionResult> {
  // 1. 입력 검증 (기존 유지)
  if (!input.email.trim() || !input.password.trim()) { ... }

  // 2. Supabase 로그인 (기존 유지)
  const result = await signIn(input.email.trim(), input.password);
  if (result.error) { return { ok: false, error: translateAuthError(result.error) }; }

  // 3. role 조회 (기존 유지)
  const roleResult = await getCurrentUserRole();
  if (roleResult.error) { return { ok: false, error: "사용자 정보를 불러오지 못했습니다." }; }

  // 4. 운영자/매니저는 바로 /admin (기존 유지)
  if (isOperationRole(roleResult.role)) {
    redirect("/admin");
  }

  // 5. player: 프로필 완료 여부 확인 (신규 추가)
  const profileResult = await getMyProfile();
  
  if (profileResult.error) {
    // DB 오류 — redirect 없이 에러 반환 (로그인 폼에 표시)
    return { ok: false, error: "프로필 정보를 불러오지 못했습니다." };
  }

  // 6. 프로필 미완료 (data=null 포함) → 온보딩
  if (!isProfileCompleted(profileResult.data)) {
    redirect("/onboarding/profile");
  }

  // 7. 프로필 완료 → 대시보드
  // [VS-5 삽입 위치: 본인인증 분기]
  redirect("/dashboard");
}
```

**`getCurrentUserRole()` vs `getMyProfile()` 분리 이유**:
- `getCurrentUserRole()`은 `profiles`에서 `role`만 SELECT → 경량, 기존 모든 페이지에서 재사용
- `getMyProfile()`은 `display_name`, `phone` 등 추가 컬럼 SELECT → player에게만 필요
- 두 호출을 합치지 않는 이유: `getUserWithRole()`의 단일 책임을 유지하고, player가 아닌 경우 `getMyProfile()`을 아예 호출하지 않는다 (불필요한 DB 쿼리 방지)

**`Promise.all` 병렬 호출을 쓰지 않는 이유**:
- `getMyProfile()` 호출 여부는 role 결과에 의존한다 (player일 때만 호출)
- 순차 처리가 맞다

## 5. 트레이드오프 및 고려사항

### 5-1. 분기 로직을 login/actions.ts에 직접 둘지, 헬퍼로 분리할지

**결정**: `login/actions.ts` 내부에 직접 작성.

헬퍼(`getLoginRedirectPath()` 등)로 분리하면 테스트는 쉬워지지만:
- 현재 프로젝트에 유닛 테스트 구조가 없음
- 이 로직은 `login/actions.ts` 단 하나에서만 사용됨
- 분리하면 호출부와 로직이 흩어져 오히려 파악이 어렵다

VS-5 이후 분기가 3개를 초과하면 그때 헬퍼 분리를 검토한다.

---

### 5-2. `empty` 상태를 에러로 볼지, 온보딩 대상으로 볼지

**결정**: 온보딩 대상으로 취급.

- `getMyProfile()` 결과 `data === null` (profiles 행 없음) → `isProfileCompleted(null)` → false → `/onboarding/profile`
- 에러 메시지로 처리하면 사용자가 아무 것도 할 수 없게 된다
- `/onboarding/profile`에는 이미 `empty` 케이스 대비가 있으므로 안전하게 보낼 수 있다

---

### 5-3. 운영자/매니저에도 프로필 완료 분기를 적용할지

**결정**: 적용하지 않는다.

- 운영자/매니저는 대회 운영 목적으로 등록된 계정이며, 프로필 입력 온보딩은 선수(player) 중심 흐름이다
- `isOperationRole()` 분기가 먼저 나오므로 코드상으로도 자연스럽게 분리된다
- 향후 운영자용 프로필 설정이 필요하다면 `/admin/settings/profile` 별도 경로로 설계한다

---

### 5-4. 프로필 완료 기준을 이번 VS에서 변경해야 하는지

**결정**: VS-1에서 확정된 기준(`display_name` + `phone`)을 그대로 따른다.

이번 VS-3에서 기준을 바꾸면 VS-1, VS-2에서 이미 구현된 `isProfileCompleted()`와 온보딩 폼 필드 정책이 충돌한다.

---

### 5-5. `getMyProfile()` 오류 시 처리

`profileResult.error`가 존재하는 경우: `{ ok: false, error: "..." }` 반환.
- 로그인은 성공했으나 redirect 전 DB 조회가 실패한 케이스
- 사용자를 `/dashboard`로 보내지 않고, 로그인 폼에 에러 메시지를 표시한다
- 재시도하면 대부분 해결되는 일시적 오류를 상정

---

### 5-6. 이후 `returnTo` 처리와의 연결

현재 `returnTo` 미구현 (VS-10에서 처리 예정). 이번 VS-3에서는 영향 없음.  
나중에 추가 시 `/dashboard` redirect 직전 `searchParams`에서 `returnTo` 꺼내서 처리하면 된다. 구조적으로 분기 끝에만 삽입하면 되므로 충돌 없음.

---

### 5-7. admin 계정의 `/admin` 우선 정책 유지

기존 `isOperationRole()` 우선 분기를 변경하지 않는다. 운영 계정 흐름은 건드리지 않는다는 정책 유지.

---

### 5-8. 로그인 액션의 책임 증가 문제

현재 `login/actions.ts`의 책임:
1. 입력 검증
2. Supabase 로그인
3. role 조회 + 분기
4. (추가) 프로필 완료 조회 + 분기

2개 → 3개 DB 호출이지만, 각 호출은 독립적이고 순서가 있다. 수용 가능 범위.  
VS-5에서 본인인증 분기까지 추가되면 4개 → 그 시점에 헬퍼 분리를 재검토한다.
