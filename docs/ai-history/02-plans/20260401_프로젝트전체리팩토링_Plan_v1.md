## 0. 플랜 개요

리서치에서 발견한 문제들을 리스크와 영향도 기준으로 4개 Phase로 분류한다.
각 Phase는 독립적으로 실행 가능하며 순서대로 진행한다.

| Phase | 이름 | 리스크 | 파일 수 | 핵심 목표 |
|-------|------|--------|---------|-----------|
| **1** | 즉시 버그/누락 수정 | 최소 | 4개 | 데이터 무결성, 보안 누락 수정 |
| **2** | 공통 인증 가드 추출 | 낮음 | ~6개 | `requireOrganizer()` 3중 정의 통합 |
| **3** | 공통 타입 추출 | 낮음 | ~15개 | `ApiResult<T>` / `ActionResult` 통일 |
| **4** | 성능 개선 | 중간 | 1개 | 순차 15회 쿼리 → 병렬 실행 |

**Phase 5 (scope 외):** `result/` vs `results/` 통합, `schedule-slots.ts` vs `scheduleSlots.ts` 통합은 기능 변경 리스크가 있어 별도 플랜으로 진행.

### Phase 2: 공통 인증 가드 추출

#### 배경

현재 `requireOrganizer()` 함수가 **3개 파일에 각각 private으로 선언**되어 있다:

```typescript
// lib/api/bracket.ts:80 - requireOrganizer()
async function requireOrganizer(): Promise<{ ok: true } | { ok: false; error: string }> {
  const auth = await getUserWithRole()
  if (auth.status !== "ready" || auth.role !== "organizer") {
    return { ok: false, error: "권한이 없습니다." }
  }
  return { ok: true }
}

// lib/api/schedule-slots.ts:434 - requireOrganizer() (15회 이상 내부 호출)
async function requireOrganizer(): Promise<{ ok: true } | { ok: false; error: string }> {
  const auth = await getUserWithRole()
  if (auth.status !== "ready" || auth.role !== "organizer") {
    return { ok: false, error: "권한이 없습니다." }
  }
  return { ok: true }
}

// lib/api/tournaments.ts:163 - ensureOrganizer() (이름이 다름)
async function ensureOrganizer(): Promise<ActionResult> {
  const result = await getUserWithRole()
  if (result.status === "unauthenticated") return { ok: false, error: "로그인이 필요합니다." }
  if (result.status === "error") return { ok: false, error: result.error ?? "..." }
  if (result.status === "empty") return { ok: false, error: "프로필이 없습니다." }
  if (result.role !== "organizer") return { ok: false, error: "권한이 없습니다." }
  return { ok: true }
}

// lib/api/results.ts:120 - requireOperationRole()
async function requireOperationRole(): Promise<ActionResult> {
  const auth = await getUserWithRole()
  if (auth.status !== "ready" || !isOperationRole(auth.role)) {
    return { ok: false, error: "권한이 없습니다." }
  }
  return { ok: true }
}
```

**주의:** 각 파일마다 에러 메시지와 상태 체크 방식이 미묘하게 다름.

#### 신규 파일 생성: `src/lib/auth/guards.ts`

공통 인증 가드를 한 곳에 정의하고, 각 파일의 private 함수를 이 공용 함수로 대체한다.

**설계:**

```typescript
// src/lib/auth/guards.ts (pseudo)

import { getUserWithRole, isOperationRole, Role } from "./roles"

type GuardSuccess = { ok: true; userId: string; role: Role }
type GuardFailure = { ok: false; error: string }
type GuardResult = GuardSuccess | GuardFailure

// organizer 전용 작업에 사용
export async function requireOrganizer(): Promise<GuardResult>
// → status 체크 → role === "organizer" 확인 → { ok: true, userId, role }

// organizer + manager 허용 작업에 사용
export async function requireOperationRole(): Promise<GuardResult>
// → status 체크 → isOperationRole() 확인 → { ok: true, userId, role }

// 에러 메시지 매핑 (내부 상수)
const AUTH_ERRORS = {
  unauthenticated: "로그인이 필요합니다.",
  error: "인증 오류가 발생했습니다.",
  empty: "프로필이 없습니다.",
  forbidden: "권한이 없습니다.",
}
```

**변경될 import 관계:**
```
before: lib/api/bracket.ts → (내부 private requireOrganizer)
after:  lib/api/bracket.ts → src/lib/auth/guards.ts (requireOrganizer)

before: lib/api/schedule-slots.ts → (내부 private requireOrganizer x15+)
after:  lib/api/schedule-slots.ts → src/lib/auth/guards.ts (requireOrganizer)

before: lib/api/results.ts → (내부 private requireOperationRole x5)
after:  lib/api/results.ts → src/lib/auth/guards.ts (requireOperationRole)

before: lib/api/tournaments.ts → (내부 private ensureOrganizer)
after:  lib/api/tournaments.ts → src/lib/auth/guards.ts (requireOrganizer)
```

**⚠️ 주의:** `tournaments.ts`의 `ensureOrganizer()`는 상태별 에러 메시지가 더 세분화되어 있다(`empty` 케이스를 별도 처리). 통합 시 이를 guards.ts에 반영해야 한다.

**페이지 레벨의 `getUserWithRole()` 직접 호출은 그대로 유지:**
페이지에서 `getUserWithRole()`을 직접 호출하는 경우는 role 값을 prop으로 전달하거나 redirect 처리를 하는 것으로, guard 패턴과 다른 용도다. 이들은 변경하지 않는다.

```typescript
// 예시: 변경하지 않는 케이스 (페이지 레벨)
const userResult = await getUserWithRole()
if (!isOperationRole(userResult.role)) redirect("/dashboard")
// → props로 role 전달하거나 redirect 처리가 목적
```

### Phase 4: `tournamentProgress.ts` 쿼리 병렬화

#### 배경

`getTournamentProgressState()`는 대회 상세 페이지 로드 시 호출되며, **15회의 순차적 Supabase 쿼리**를 실행한다. 각 쿼리가 순서대로 `await`되어 대기 시간이 누적된다.

#### 현재 실행 구조

```
1. tournaments 조회 (다른 쿼리에서 tournament_id 사용하므로 필수 선행)
2. divisions 조회 (divisionIds 추출 후 groups 쿼리에 필요하므로 선행)
─── 위 2개 완료 후 아래 13개가 순차 실행 ───
3.  tournament_team_applications COUNT (approved)
4.  tournament_team_applications COUNT (total)
5.  groups COUNT
6.  matches COUNT (league type)
7.  matches COUNT (completed league)
8.  standings COUNT
9.  matches COUNT (total)
10. matches COUNT (completed)
11. matches COUNT (tournament type)
12. matches COUNT (final completed)
13. matches COUNT (final exists)
14. courts COUNT
15. matches COUNT (scheduled)
```

3~15번은 서로 의존성이 없다. tournament와 divisions 조회 후 나머지를 `Promise.all`로 병렬화한다.

#### 개선 후 구조 (pseudo)

```typescript
// 1단계: 대회 기본 정보 조회 (다른 모든 쿼리의 선행 조건)
const tournament = await supabase.from("tournaments")...

// 2단계: divisions 조회 (groups 쿼리의 선행 조건)
const divisions = await supabase.from("divisions")...
const divisionIds = divisions.map(d => d.id)

// 3단계: 나머지 13개 쿼리 병렬 실행
const [
  approvedTeamsResult,
  totalTeamsResult,
  groupsCount,
  groupMatchesResult,
  completedGroupMatchesResult,
  standingsResult,
  totalMatchesResult,
  completedMatchesResult,
  tournamentMatchesResult,
  finalCompletedResult,
  finalExistsResult,
  courtsCountResult,
  scheduledMatchesResult,
] = await Promise.all([
  supabase.from("tournament_team_applications").select(...).eq("status", "approved"),
  supabase.from("tournament_team_applications").select(...),
  supabase.from("groups").select(...).in("division_id", divisionIds),
  // ... 나머지 10개
])
```

**예상 성능 개선:** 15번 순차 → 3단계 실행 (1+1+병렬13)으로 전체 대기 시간 대폭 감소.
레이턴시 환경에 따라 다르지만 이론적으로 ~80% 단축 가능.

### Phase 2: `src/lib/auth/guards.ts` 신규 생성

```typescript
// src/lib/auth/guards.ts
import { getUserWithRole, isOperationRole, type Role } from "./roles"

type GuardSuccess = { ok: true; userId: string; role: Role }
type GuardFailure = { ok: false; error: string }
export type GuardResult = GuardSuccess | GuardFailure

const AUTH_ERRORS = {
  unauthenticated: "로그인이 필요합니다.",
  error: "인증 오류가 발생했습니다.",
  empty: "프로필이 없습니다.",
} as const

function buildAuthError(status: string): GuardFailure {
  return {
    ok: false,
    error: AUTH_ERRORS[status as keyof typeof AUTH_ERRORS] ?? "인증 오류가 발생했습니다.",
  }
}

/**
 * organizer 전용 작업에서 사용.
 * lib/api 파일 내부 뮤테이션 함수 상단에서 호출한다.
 */
export async function requireOrganizer(): Promise<GuardResult> {
  const result = await getUserWithRole()

  if (result.status !== "ready") {
    return buildAuthError(result.status)
  }
  if (result.role !== "organizer") {
    return { ok: false, error: "권한이 없습니다." }
  }
  return { ok: true, userId: result.user!.id, role: result.role }
}

/**
 * organizer + manager 허용 작업에서 사용.
 * 결과 입력, 순위 계산 등에 사용한다.
 */
export async function requireOperationRole(): Promise<GuardResult> {
  const result = await getUserWithRole()

  if (result.status !== "ready") {
    return buildAuthError(result.status)
  }
  if (!isOperationRole(result.role)) {
    return { ok: false, error: "권한이 없습니다." }
  }
  return { ok: true, userId: result.user!.id, role: result.role! }
}
```

  // ── 2단계: divisions 조회 (groups 쿼리의 선행 조건) ──
  const { data: divisions, error: divisionsError } = await supabase
    .from("divisions")
    .select("id")
    .eq("tournament_id", tournamentId)
  if (divisionsError) return { data: null, error: divisionsError.message }
  const divisionIds = (divisions ?? []).map((d) => d.id)

  // ── 3단계: 나머지 13개 COUNT 쿼리 병렬 실행 ──
  const [
    approvedTeamsResult,
    totalTeamsResult,
    groupsCountResult,

    completedGroupMatchesResult,
    standingsResult,
    totalMatchesResult,
    completedMatchesResult,
    tournamentMatchesResult,
    finalCompletedResult,
    finalExistsResult,
    courtsCountResult,
    scheduledMatchesResult,
  ] = await Promise.all([
    supabase
      .from("tournament_team_applications")
      .select("id", { count: "exact", head: true })
      .eq("tournament_id", tournamentId)
      .eq("status", "approved"),
    supabase
      .from("tournament_team_applications")
      .select("id", { count: "exact", head: true })
      .eq("tournament_id", tournamentId),
    divisionIds.length === 0
      ? Promise.resolve({ count: 0, error: null })
      : supabase
          .from("groups")
          .select("id", { count: "exact", head: true })
          .in("division_id", divisionIds),
    // ... 나머지 10개 쿼리
  ])

  // 에러 체크
  if (approvedTeamsResult.error) return { data: null, error: approvedTeamsResult.error.message }
  // ... 나머지 에러 체크

  // summary 구성 및 반환 (기존 로직 동일)
  const summary: ProgressSummary = {
    approvedTeams: approvedTeamsResult.count ?? 0,
    // ...
  }
  // ...

## 5. 트레이드오프 및 고려 사항

### Phase 1: 버그/누락 수정

**트레이드오프 없음.** 순수한 수정이고 동작 변경이 없다.

**고려 사항:**
- `getAdminTournaments` 제거 시 혹시 모를 외부 사용처(예: API 라우트)를 grep으로 한 번 더 확인
- `TournamentAdminRow` 타입도 함께 사용되는 곳이 있는지 확인 후 제거

### Phase 2: 공통 인증 가드 추출

**트레이드오프:**
- `tournaments.ts`의 `ensureOrganizer()`는 `status === "empty"`를 별도 에러 메시지로 처리하는 반면, `bracket.ts`와 `schedule-slots.ts`의 `requireOrganizer()`는 `status !== "ready"`를 단순히 "권한이 없습니다"로 처리한다. 새로운 `guards.ts`에서는 상태별 세분화된 메시지를 기준으로 통일하는 것이 더 사용자 친화적이다.

**고려 사항:**
- `schedule-slots.ts`는 15회 이상 `requireOrganizer()`를 호출하므로 교체 후 전체 함수를 재검토해야 한다. 기계적인 치환이므로 로직 변경은 없다.
- `GuardResult`의 `userId`, `role` 필드는 현재 호출처에서 사용하지 않을 수 있다. 단, 나중에 "누가 변경했는지" 로깅 추가 시 유용하다.

### Phase 3: 공통 타입 추출

**트레이드오프:**
- `tournamentGuards.ts`의 `GuardResult`는 `ActionResult`와 구조가 동일하다. Phase 2에서 `src/lib/auth/guards.ts`에 `GuardResult`를 이미 정의하므로 혼선이 있을 수 있다. 두 타입의 관계를 명확히 해야 한다:
  - `lib/api/tournamentGuards.ts`의 `GuardResult` → `ActionResult`로 교체 (구조 동일)
  - `src/lib/auth/guards.ts`의 `GuardResult` → 별도 (userId, role 포함)

**고려 사항:**
- `results.ts`에서 현재 `ActionResult`와 `ActionResult<message>` 두 가지를 혼용한다. `ActionResultWithMessage`로 명확히 분리하면 타입 안전성이 향상된다.
- 파일 수가 많으므로 한 번에 모두 변경하지 말고 파일 단위로 커밋하는 것이 권장된다.

### Phase 4: 성능 개선

**트레이드오프:**
- `Promise.all` 실행 시 하나라도 실패하면 나머지 결과도 버려진다. 현재 순차 실행은 실패 지점을 정확히 파악할 수 있지만, 병렬화 후에도 에러 체크를 개별로 하면 동일한 수준의 디버깅이 가능하다.
- `Promise.all`에서 개별 쿼리 에러는 `rejected`가 아닌 `{ error }` 형태로 반환되므로 `Promise.allSettled`가 아닌 `Promise.all`로도 안전하다.

**고려 사항:**
- `divisionIds`가 비어 있을 때 `groups` 쿼리의 `.in("division_id", [])` 처리를 단락해야 한다 (기존 `countGroupsByDivisionIds` 헬퍼가 이미 처리하던 부분).
- 병렬화 후 쿼리 결과의 에러 체크 순서가 변경되어도 로직 결과에 영향 없음.
- Vercel의 서버리스 환경에서 동시 Supabase 연결 수 제한을 확인해야 한다. 13개의 병렬 연결은 일반적으로 문제없지만 프로젝트 tier에 따라 확인 필요.
