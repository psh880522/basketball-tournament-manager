## 1. 구현 범위 요약

### 1-1. 이번에 구현할 것

| 항목 | 설명 |
|---|---|

| 편집 page.tsx — 신청 현황 데이터 로드 | server component에서 집계 데이터 로드 후 Form에 props 전달 |

#### 3-2. `app/(app)/admin/tournaments/[id]/edit/actions.ts` — `updateTournamentStatusAction` 추가

`admin/actions.ts`의 `changeTournamentStatusAction`을 edit 편집 페이지에서 직접 호출하지 않고, `edit/actions.ts`에 래퍼 함수를 추가하여 revalidatePath를 편집 경로 기준으로 처리:
```
// edit/actions.ts에 추가
import { changeTournamentStatus, type TournamentStatus } from "@/lib/api/tournaments";

export async function updateTournamentStatusAction(
  tournamentId: string,
  nextStatus: TournamentStatus
): Promise<ActionResult> {
  const result = await changeTournamentStatus(tournamentId, nextStatus);
  if (result.ok) {
    revalidatePath(`/admin/tournaments/${tournamentId}/edit`);
    revalidatePath(`/admin`);
  }
  return result;
}
```

open 전환 가능 조건: 디비전이 1개 이상 존재 → 클라이언트에서 props 기반으로 버튼 disabled 처리.

## 6. 리스크 및 주의사항

### 1. `new/Form.tsx` 완전 재작성 시 기존 동작 회귀

- 현재 폼 로직(날짜 범위 선택, 포스터 파일 처리, useTransition 패턴)이 재작성 과정에서 누락될 수 있음
- **대응**: Step 0(기본정보)은 현재 코드 내용을 최대한 그대로 복사 후 배치. 스텝 위저드 구조만 감쌈.

### 2. `edit/page.tsx` props 전달 구조 변경

- 현재 `page.tsx`는 `PosterSection`, `SettingsSection`을 직접 렌더링하고, `TournamentEditForm`은 기본정보만 받음
- 변경 후 `TournamentEditForm`이 모든 sections를 담으므로, `PosterSection`, `SettingsSection`의 export를 제거하고 Form 내부로 이동 시 import 충돌 가능
- **대응**: `page.tsx`에서 `PosterSection`, `SettingsSection` import 제거를 명시적으로 처리. `Form.tsx` 파일에서 기존 export를 tabs 파일로 이동.

### 3. `getDivisionApplicationCounts` — DB 조회 최적화

- `listTournamentApplications`를 재사용하면 전체 row를 fetch 후 클라이언트 집계 → 대규모 데이터에서 비효율
- `tournament_team_applications`에서 `status IN ('confirmed', 'waitlisted')` + `tournament_id` 조건으로 필터링한 후 `division_id`별 클라이언트 집계는 현재 규모에서 허용 범위
- **대응**: 클라이언트 측 집계로 구현. 추후 DB 뷰나 RPC로 최적화 가능.

### 4. `application_open_at` / `application_close_at` 시간대 처리

- `datetime-local` input은 로컬 시간(KST) 반환
- `new/actions.ts`에서 `schedule_start_at` 처리 패턴(`new Date(\`${date}T${time}:00+09:00\`)`)을 재사용
- ISO 형식으로 변환 후 저장. 편집 폼에서는 이미 동일 처리 중임을 확인 완료.
- **대응**: `new/actions.ts`의 DivisionPayload에서 datetime-local 값을 받아 UTC ISO로 변환하는 헬퍼 인라인 작성.

### 5. 생성 폼 리다이렉트 경로 변경

- 현재 생성 완료 후 `router.push("/admin")` → 편집 페이지로 변경 시 `result.id`를 활용
- `createTournamentAction`이 `{ ok: true; id: string }`을 반환하므로 `router.push(\`/admin/tournaments/${result.id}/edit\`)`로 변경 가능
- **대응**: `new/Form.tsx`의 `handleSubmit` 내 `router.push` 경로를 `/admin/tournaments/${result.id}/edit`로 수정.

### 6. 편집 탭 상태 초기화

- `edit/Form.tsx`의 탭 상태를 `useState`로 관리하면 새로고침 시 탭1(기본정보)로 초기화됨
- URL 파라미터(`?tab=divisions`) 방식은 v1에서 생략
- **대응**: 단순 `useState`로 구현. 리서치 문서에서도 `useState`로 관리해도 기능상 문제 없음을 명시.

### 7. `max_teams` 이동 — SettingsSection 제거

- 현재 `max_teams`는 `SettingsSection`에 있고 `page.tsx`에서 `initialMaxTeams` prop으로 전달됨
- 공개설정 탭으로 이동 후 `SettingsSection` 컴포넌트 export를 제거할 때 `page.tsx`에서 import 오류 발생 가능
- **대응**: `edit/page.tsx` 수정 시 `SettingsSection` import 라인(현재 line 9의 named export)을 동시에 제거.

### 8. 탭 분리 시 서버 액션 import 분산

- 각 `tabs/*.tsx`에서 필요한 액션을 `edit/actions.ts`에서 직접 import
- 서버 액션 파일에 `"use server"` 지시어가 있으므로 클라이언트 컴포넌트에서 import 시 Next.js가 자동으로 처리함 — 패턴은 기존 `edit/Form.tsx`와 동일

## 7. 결론

이번 구현의 핵심은 **생성 폼과 편집 폼 간의 기능 갭 해소**와 **편집 페이지 UX 개선**이다.

**우선순위 순서**:
1. `lib/api/applications.ts` 신규 함수 추가 (의존성 선행)
2. `new/actions.ts` + `DivisionDraft` 타입 확장 (이중 작업 해소의 핵심)
3. `new/Form.tsx` 3단계 스텝 위저드 (생성 UX 개선)
4. `edit/tabs/*.tsx` + `edit/Form.tsx` 탭 기반 재구조화 (편집 UX 개선)
5. `edit/page.tsx` 신청 현황 데이터 로드 + 디비전 카드 표시

**DB 마이그레이션 없음**. 프론트엔드와 서버 액션 수정만으로 전체 구현 완료 가능.
