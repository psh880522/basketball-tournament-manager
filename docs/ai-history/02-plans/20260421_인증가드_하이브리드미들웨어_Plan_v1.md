## 1. 구현 범위 요약

### 1-1. 이번에 구현할 것

1. **`middleware.ts` 신규 생성** (프로젝트 루트)
   - Supabase 세션 유무만 확인하는 1차 인증 가드
   - 미인증 상태로 보호 경로 접근 시 `/login?next={경로}` 리다이렉트
   - 세션 토큰 자동 갱신 처리 (Supabase SSR 필수 패턴)
   - 이미 로그인된 사용자의 `/login`, `/signup` 접근 시 `/dashboard` 리다이렉트

2. **`app/(app)` 하위 보호 페이지 정리** (21개)
   - 미들웨어가 처리하게 되는 중복 세션 체크 구문 제거
   - 역할 가드(`isOperationRole`, `isPlayerRole` 등)는 유지

3. **`app/(auth)` 페이지 정리** (login, signup)
   - 미들웨어에서 이미 로그인된 사용자를 처리하므로 page.tsx의 중복 리다이렉트 제거

### 1-2. 제외할 것

- DB 스키마 변경 없음 (새 테이블/컬럼 없음)
- RLS 정책 변경 없음
- 역할 기반 세부 가드 변경 없음 (page.tsx 그대로 유지)
- 팀 멤버십 가드(`getMyRoleInTeam`) 변경 없음 (page.tsx 그대로 유지)
- `src/lib/auth/guards.ts` 변경 없음 (서버 액션 전용)

### Phase 4. 프론트엔드

이 기능의 핵심 Phase.

#### Phase 4-1. `middleware.ts` 신규 생성

**파일**: `middleware.ts` (프로젝트 루트, `app/` 및 `src/`와 동일 레벨)

**핵심 로직 (의사코드)**:

```
함수 middleware(request: NextRequest):
  1. NextResponse.next({ request }) 으로 response 초기화
     → 쿠키를 request와 response 양쪽에 전달하기 위해 필요

  2. createServerClient(SUPABASE_URL, SUPABASE_ANON_KEY, {
       cookies.getAll: () => request.cookies.getAll(),
       cookies.setAll: (cookiesToSet) => {
         각 쿠키를 request.cookies.set() AND response.cookies.set() 에 모두 설정
       }
     })
     → 이 패턴이 세션 갱신 쿠키를 response에 담는 Supabase 공식 미들웨어 패턴

  3. supabase.auth.getUser() 호출
     → 이 호출 자체가 만료된 access_token을 refresh_token으로 자동 갱신 트리거

  4. 현재 경로(pathname) 확인:
     
     a. 이미 로그인된 사용자가 AUTH 경로(/login, /signup)에 접근:
        - user가 존재하면 → redirect("/dashboard")
        - user가 없으면 → response 반환 (통과)

     b. 보호 경로(PROTECTED_PATHS)에 접근:
        - user가 없으면 → redirect("/login?next={pathname}")
        - user가 있으면 → response 반환 (통과)

     c. 그 외 경로:
        → response 반환 (통과)

  5. response 반환
     → 갱신된 세션 쿠키가 담긴 response

상수 PROTECTED_PATHS: 미들웨어가 세션 체크를 실행할 경로 패턴 목록
상수 AUTH_PATHS: 이미 로그인된 사용자를 리다이렉트할 경로 목록
```

**`config.matcher` 설계**:

```
포함 (미들웨어 실행):
  - /admin/:path*
  - /dashboard
  - /teams/:path*
  - /my-applications/:path*
  - /onboarding/:path*
  - /team/:path*
  - /tournament/:id/apply
  - /login
  - /signup

제외 (미들웨어 건너뜀, Next.js 정적 파일):
  - /_next/static/:path*
  - /_next/image/:path*
  - /favicon.ico
  - /api/:path* (API 라우트는 자체 인증 처리)
```

**재사용 패턴**: `@supabase/ssr`의 `createServerClient` — 기존 `src/lib/supabase/server.ts`의 `createSupabaseServerClient`와 동일한 패키지이나, 미들웨어는 `cookies()` (next/headers) 대신 `request/response` 쿠키를 직접 다뤄야 하므로 별도 구현.

**주의**: `src/lib/supabase/server.ts`의 `createSupabaseServerClient`는 `React cache()`와 `next/headers`의 `cookies()`에 의존하므로 미들웨어에서 import하면 런타임 에러 발생. 미들웨어 전용 클라이언트 생성 로직을 `middleware.ts` 안에 직접 작성한다.

#### Phase 4-2. `app/(auth)` 페이지 정리

미들웨어에서 이미 로그인된 사용자의 `/login`, `/signup` 접근을 `/dashboard`로 리다이렉트하므로, page.tsx 상단의 중복 세션 체크 구문을 제거한다.

**파일**: `app/(auth)/login/page.tsx`

```
제거 대상 (미들웨어가 대신 처리):
  const result = await getUserWithRole();
  if (result.status === "ready") {
    if (isOperationRole(result.role)) redirect("/admin");
    if (isPlayerRole(result.role)) redirect("/dashboard");
    redirect("/");
  }

유지:
  - LoginForm 렌더링
  - getUserWithRole import 제거 (더 이상 사용 안 함)
```

**파일**: `app/(auth)/signup/page.tsx`

```
제거 대상:
  const result = await getUserWithRole();
  if (result.status === "ready") {
    redirect(isOperationRole(result.role) ? "/admin" : "/dashboard");
  }

유지:
  - SignupForm, OnboardingStepIndicator 렌더링
  - getUserWithRole import 제거
```

**주의**: 미들웨어에서는 역할 조회 없이 `/dashboard`로 단순 리다이렉트한다. 현재 login/page.tsx는 역할에 따라 `/admin` vs `/dashboard`를 분기하는데, 이 분기를 미들웨어에서 재현하려면 DB 조회가 필요하다. 미들웨어에서는 DB 조회 없이 `/dashboard`로만 보내고, `/dashboard/page.tsx`에서 기존 역할 분기(`isOperationRole` → `/admin`) 로직이 처리한다. 따라서 로그인된 organizer가 `/login`에 접근하면:
  - 미들웨어 → `/dashboard`
  - `/dashboard/page.tsx`의 역할 가드 → `/admin`
  리다이렉트가 2회 발생하지만 기능적으로 올바르다.

**`status === "empty"` 처리 방침**:

현재 `getUserWithRole()`에서 `"empty"`는 "auth user는 있으나 profiles 행이 없는" 예외 상태다. 미들웨어는 `auth.getUser()`만 호출하므로 `"empty"` 케이스는 미들웨어를 통과한다. 따라서 page.tsx에서 `"empty"` 분기를 유지할지 결정해야 한다.

**결정**: `"empty"`는 데이터 정합성 오류(profiles 행 누락)이므로, 발생 자체가 비정상이다. 미들웨어 통과 후 page.tsx에 도달했을 때 `"empty"` 케이스를 `/login`으로 보내거나 에러 UI를 보여주는 기존 로직을 **유지**한다.

```
수정 후 page.tsx 공통 구조:
  const result = await getUserWithRole();

  // ① "empty" (profiles 행 누락) — 비정상 케이스, 유지
  if (result.status === "empty") redirect("/login");

  // ② DB 에러 → 에러 UI, 유지
  if (result.status === "error") return <ErrorUI />;

  // ③ 역할 가드 — 유지
  if (!isOperationRole(result.role)) redirect("/dashboard");
```

## 5. 구현 순서 및 의존 관계

```
[Step 1] middleware.ts 작성
    │
    │  ← 미들웨어가 세션 보장을 하기 전까지는 page.tsx를 수정하면 안 됨
    │
    ▼
[Step 2] 미들웨어 동작 검증
    - 비인증 상태로 /admin, /dashboard 등 직접 URL 접근 → /login 리다이렉트 확인
    - 인증 상태로 /login 접근 → /dashboard 리다이렉트 확인
    - 세션 만료 시 쿠키 갱신 후 정상 접근 확인
    │
    ▼
[Step 3] app/(auth) 페이지 정리 (login, signup)
    - getUserWithRole() 호출 및 import 제거
    │
    ▼
[Step 4] app/(app) 보호 페이지 정리 (21개)
    - status === "unauthenticated" redirect 구문 제거
    - status === "empty" redirect 구문 중 unauthenticated 관련만 제거
      (empty → login 리다이렉트는 유지)
    │
    ▼
[Step 5] 전체 흐름 재검증
    - 역할별 접근 시나리오 확인
    - 세션 만료 시나리오 확인
```

**의존 관계 요약**:
- Step 1이 완료되어야 Step 3, 4 작업 가능 (미들웨어 없이 page.tsx를 먼저 수정하면 인증 가드 공백 발생)
- Step 2 검증 없이 Step 3, 4로 진행하면 보안 홀 발생 위험
- Step 3과 Step 4는 서로 독립적 (병렬 작업 가능하나, Step 1 완료 후 진행)

## 6. 리스크 및 주의사항

### 6-1. 미들웨어 쿠키 갱신 누락 시 세션 루프 위험

**상황**: 미들웨어에서 `auth.getUser()`를 호출하되 response 쿠키에 갱신된 토큰을 담지 않으면, 후속 Server Component에서 만료된 토큰으로 Supabase를 호출해 세션 없음으로 판단 → 무한 리다이렉트 루프.

**대응**: `createServerClient` 초기화 시 `setAll`에서 `response.cookies.set()`을 반드시 실행.

### 6-2. `status === "empty"` 처리 공백

**상황**: 미들웨어는 `auth.getUser()`만 사용하므로 `profiles` 행 누락 케이스를 탐지하지 못한다. page.tsx에서 `"empty"` → `/login` 분기를 제거하면, `profiles` 행이 없는 사용자가 보호 페이지에 도달해 역할 가드에서 `null` 역할로 잘못 처리될 수 있다.

**대응**: `"empty"` → `/login` 분기는 **제거하지 않는다** (Phase 4-3에서 명시).

### 6-3. `/tournament/:id/apply` 경로 matcher 누락

**상황**: `app/(app)/tournament/[id]/apply/page.tsx`가 인증 필요 페이지인데 matcher에서 빠지면 미들웨어 미적용.

**대응**: `matcher`에 `/tournament/:id/apply` 패턴 명시적 포함.

### 6-4. `app/(app)/team/page.tsx` 경로 확인 필요

리서치 시 `app/(app)/team/page.tsx`(단수)가 발견됨. `/teams/*`(복수) matcher로 커버되지 않는다.

**대응**: `middleware.ts`의 `matcher`에 `/team/:path*`를 별도 추가.

### 6-5. login 페이지의 역할별 분기 제거로 인한 UX 변화

**상황**: 현재 organizer가 `/login` 접근 시 `/admin`으로 바로 이동. 미들웨어 도입 후에는 `/dashboard` → `/admin` 2단 리다이렉트 발생.

**영향**: 기능적으로 동일하나 네트워크 왕복 1회 추가.

**대응**: 허용 범위로 판단. 단, organizer + manager 사용자가 많아지면 미들웨어에서 profile 역할 조회를 추가하는 방안을 고려할 수 있으나 현재 MVP에서는 제외.

### 6-6. `forgot-password`, `reset-password` 경로

`app/(auth)/forgot-password/page.tsx`, `app/(auth)/reset-password/page.tsx`는 비인증 사용자도 접근해야 한다. matcher에서 `/login`, `/signup`만 AUTH 경로로 처리하고, 이 두 경로는 그냥 통과.

**대응**: AUTH_PATHS 상수에 `/login`, `/signup`만 포함.

### 6-7. Edge Runtime 제약

미들웨어는 Edge Runtime에서 실행된다. `@supabase/ssr`의 `createServerClient`는 Edge 호환이지만, `src/lib/auth/roles.ts`의 `getUserWithRole()`은 `React cache()` + `next/headers`에 의존해 Edge에서 실행 불가.

**대응**: 미들웨어에서는 절대 `getUserWithRole()`을 import하지 않는다. 미들웨어 전용 클라이언트를 `middleware.ts` 내부에 직접 구현.

### 달성 목표

```
Before:
  21개 page.tsx에 중복된 세션 체크 → 새 페이지 추가 시 가드 누락 위험

After:
  middleware.ts 1곳에서 세션 유무 보장
  page.tsx는 역할 기반 세부 가드에만 집중
  새 보호 경로 추가 시 matcher에만 추가하면 자동 보호
```

### 최종 인증 흐름

```
HTTP 요청 (비인증, 보호 경로)
    │
    ▼
middleware.ts
    └─ auth.getUser() → null → redirect("/login?next=/admin/...")
                                               ↑ 로그인 후 원래 경로로 복귀 가능

HTTP 요청 (인증, 보호 경로)
    │
    ▼
middleware.ts
    └─ auth.getUser() → user → NextResponse.next() + 갱신된 세션 쿠키 담김
         │
         ▼
    (app)/layout.tsx
         └─ getUserWithRole() → Sidebar / GlobalHeader 결정
              │
              ▼
         page.tsx
              ├─ getUserWithRole() (cache 재사용)
              ├─ status === "empty" → /login   ← DB 정합성 오류 방어
              ├─ status === "error" → <ErrorUI>
              └─ 역할/팀멤버십 가드 → 세부 리다이렉트
```
