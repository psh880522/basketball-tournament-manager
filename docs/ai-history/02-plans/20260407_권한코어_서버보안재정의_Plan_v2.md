## 1. 기능 요약

### 이번 범위에 포함되는 것

- `app_role` enum에 `user` 추가 → 4종 (`organizer | manager | user | player`)
- 신규 가입 기본 role을 `user`로 변경 (DEFAULT 변경)
- 기존 `player` 데이터 중 프로필 미완료자를 `user`로 downgrade (backfill)
- DB 함수 `is_player()` 신규 추가
- TypeScript `Role` 타입에 `"user"` 추가 + `isPlayerRole()`, `isUserRole()` 유틸 추가
- `requirePlayer()` 서버 가드 추가 (`src/lib/auth/guards.ts`)
- 로그인 후 redirect 분기 재설계 (`user` → `/`, `player` → `/dashboard`)
- 회원가입 후 redirect 재설계 (`user` → `/`)
- `teams` INSERT RLS에 `is_player()` 조건 추가 → `user`는 팀 생성 불가
- `tournament_team_applications` INSERT RLS에 `is_player()` 조건 추가
- `update_user_role` RPC에 `user` 값 허용 추가
- admin/users 서버 액션의 허용 role 목록에 `user` 추가
- **신규 추가**: `src/lib/config/env.ts` — 환경 플래그 유틸 파일 생성

### 이번 범위에서 제외

### `supabase/migrations/0215_role_add_user.sql`

**역할**: `app_role` enum 확장 + 기본값/backfill/DB 함수 처리  
**수정 이유**: 현재 enum에 `user`가 없어 신규 가입자와 선수를 구분 불가

**핵심 구조 및 pseudocode**:

```sql
-- 1) storage 정책 임시 삭제 (app_role 타입 참조 의존성 제거)
DROP POLICY IF EXISTS poster_update_organizer ON storage.objects;
DROP POLICY IF EXISTS poster_delete_organizer ON storage.objects;

-- 2) 신규 enum 생성 (4종)
CREATE TYPE public.app_role_v3 AS ENUM ('organizer', 'manager', 'user', 'player');

-- 3) profiles.role DEFAULT 제거 후 타입 교체
ALTER TABLE public.profiles ALTER COLUMN role DROP DEFAULT;
ALTER TABLE public.profiles
  ALTER COLUMN role TYPE public.app_role_v3
  USING (role::text::public.app_role_v3);

-- 4) backfill: 기존 player 중 프로필 미완료자 → user로 downgrade
--    기준: display_name IS NULL OR phone IS NULL
UPDATE public.profiles
  SET role = 'user'
  WHERE role = 'player'
    AND (display_name IS NULL OR phone IS NULL);

-- 5) 신규 기본값 설정
ALTER TABLE public.profiles ALTER COLUMN role SET DEFAULT 'user';

-- 6) enum 이름 교체
DROP TYPE public.app_role;
ALTER TYPE public.app_role_v3 RENAME TO app_role;

-- 7) storage 정책 재생성
CREATE POLICY poster_update_organizer ON storage.objects ...;
CREATE POLICY poster_delete_organizer ON storage.objects ...;

-- 8) is_organizer(), is_manager() — 변경 없음 (재생성 불필요)

-- 9) is_player() 신규 추가
CREATE OR REPLACE FUNCTION public.is_player() RETURNS boolean
LANGUAGE sql SECURITY DEFINER SET search_path = public AS $$
  SELECT EXISTS (
    SELECT 1 FROM profiles WHERE id = auth.uid() AND role = 'player'
  )
$$;
```

**주의**:
- `on_auth_user_created` 트리거는 `INSERT INTO profiles (id)`만 실행 → DEFAULT 변경으로 별도 수정 불필요
- `is_organizer()`, `is_manager()` 함수는 `player`, `user` 모두 false를 반환하므로 변경 없음

### `supabase/migrations/0218_update_user_role_add_user.sql`

**역할**: `update_user_role` RPC가 `user` 값을 허용하도록 수정  
**수정 이유**: 현재 RPC가 `player | manager`만 허용 → admin에서 `user`로 역할 변경 불가

**핵심 pseudocode**:

```sql
CREATE OR REPLACE FUNCTION public.update_user_role(
  target_user_id uuid,
  new_role text
)
RETURNS void LANGUAGE plpgsql SECURITY DEFINER ...
AS $$
BEGIN
  IF NOT is_organizer() THEN RAISE EXCEPTION 'Forbidden'; END IF;

  -- 변경: 'user'도 허용
  IF new_role NOT IN ('user', 'player', 'manager') THEN
    RAISE EXCEPTION 'Invalid role';
  END IF;

  -- organizer 보호 유지
  IF EXISTS (SELECT 1 FROM profiles WHERE id = target_user_id AND role = 'organizer') THEN
    RAISE EXCEPTION 'Cannot change organizer role';
  END IF;

  UPDATE profiles SET role = new_role::app_role WHERE id = target_user_id;
END;
$$;
```

### `src/lib/config/env.ts` ← 신규 파일

**역할**: 환경 플래그 유틸 — 현재 실행 환경(production/non-production) 판별  
**추가 이유**: 본인인증 UI 구현 시(VS-R8, VS-R9) 비-프로덕션 환경에서 실제 인증 로직을 스킵하기 위한 공통 플래그 필요. 이 파일은 이번 단계에서 생성만 하고, 실제 활용은 VS-R8/VS-R9 구현 시점에 이루어짐.

**환경 판별 근거**:
- Vercel 배포 환경: `VERCEL_ENV` = `"production"` | `"preview"` | `"development"`
- 로컬 개발: `VERCEL_ENV` 미설정 → `NODE_ENV = "development"`
- 프로덕션 판별 조건: `VERCEL_ENV === "production"`만 프로덕션으로 간주

**핵심 구조**:

```typescript
// src/lib/config/env.ts

/** Vercel 프로덕션 배포인지 여부 */
export const isProduction = process.env.VERCEL_ENV === "production";

/** 본인인증 로직을 실제로 호출해야 하는지 여부 */
export const requireIdentityVerification = isProduction;

/**
 * 현재 앱 환경 이름
 * - "production": Vercel 프로덕션 배포
 * - "preview": Vercel preview(staging) 배포
 * - "development": 로컬 개발 또는 Vercel development 환경
 */
export const appEnv: "production" | "preview" | "development" =
  (process.env.VERCEL_ENV as "production" | "preview" | "development") ??
  "development";
```

**VS-R8/VS-R9 구현 시 활용 방식 (참고용)**:

```typescript
// 본인인증 서버 액션 (VS-R8/R9 구현 시 참고)
import { requireIdentityVerification } from "@/src/lib/config/env";

async function confirmIdentityVerification(payload: ...) {
  if (!requireIdentityVerification) {
    // 비-프로덕션: 실제 인증 API 호출 스킵, 바로 승격 처리
    await promoteToPlayer(userId);
    redirect("/dashboard");
    return;
  }

  // 프로덕션: 실제 외부 인증 업체 API 검증 후 승격
  const verified = await callIdentityVerificationAPI(payload);
  if (!verified) return { ok: false, error: "본인인증에 실패했습니다." };
  await promoteToPlayer(userId);
  redirect("/dashboard");
}
```

### `src/lib/auth/guards.ts`

**역할**: 서버 액션용 권한 가드 함수  
**수정 이유**: `player` 전용 기능 서버 보호를 위한 `requirePlayer()` 추가

**핵심 구조**:

```typescript
// 신규 추가
export async function requirePlayer(): Promise<ActionResult> {
  const result = await getUserWithRole();

  if (result.status === "unauthenticated") {
    return { ok: false, error: "로그인이 필요합니다." };
  }
  if (result.status === "error") {
    return { ok: false, error: result.error ?? "사용자 정보를 불러오지 못했습니다." };
  }
  if (result.status === "empty") {
    return { ok: false, error: "프로필이 없습니다." };
  }
  if (result.role !== "player") {
    return { ok: false, error: "선수 등록이 필요합니다." };
  }

  return { ok: true };
}

// 기존 requireOrganizer(), requireOperationRole() 유지
```

## 5. 구현 시 고려사항

### 기존 구조 유지 vs 공통화

- `getUserWithRole()` / `isOperationRole()` 패턴 그대로 유지
- 신규 `isPlayerRole()`, `requirePlayer()` 추가만으로 충분 — 공통 추상화 불필요

### 클라이언트/서버 검증 위치

- **서버 우선**: `requirePlayer()` 가드를 player 전용 서버 액션(팀 생성, 대회 신청)에 먼저 적용
- **RLS 이중 보호**: 서버 액션 + DB RLS 두 계층에서 `user` 차단
- 클라이언트 UI 분기(버튼 숨김 등)는 이번 범위 제외

### 상태 처리 / 에러 처리

- `requirePlayer()` 반환값이 `ok: false`이면 서버 액션에서 그대로 에러 반환
- `"선수 등록이 필요합니다."` 에러 메시지를 프론트에서 선수 등록 페이지로 유도하는 UI는 후속 단계(VS-R11, VS-R12)에서 처리

### 기존 player 데이터 backfill 기준

- **기준**: `display_name IS NULL OR phone IS NULL` → `user`로 downgrade
- **근거**: 프로필 미완료 player는 실제 선수 등록을 완료하지 않은 상태
- **주의**: 마이그레이션 전 `SELECT count(*) FROM profiles WHERE role='player' AND (display_name IS NULL OR phone IS NULL)` 로 영향 규모 파악 권장

### status = "empty" 예외 처리

- `status="empty"`: `profiles` 행이 없는 상태 (on_auth_user_created 트리거 실패 등 예외)
- 기존 가드 패턴 그대로 유지 — `empty` → `"프로필이 없습니다."` 에러 반환

### user가 팀 생성 / 대회 신청 / dashboard 접근 시도 시

| 시도 | 차단 레이어 | 응답 |
|------|------------|------|
| 팀 생성 (서버 액션) | `requirePlayer()` | `ok: false, error: "선수 등록이 필요합니다."` |
| 팀 생성 (RLS) | `is_player()` 조건 | DB 레벨 INSERT 거부 |
| 대회 신청 (서버 액션) | `requirePlayer()` | 동일 |
| 대회 신청 (RLS) | `is_player()` 조건 | DB 레벨 INSERT 거부 |
| `/dashboard` 접근 | 이번 범위 제외 (VS-R11) | 후속 단계에서 redirect 처리 |

### admin/users 기능과 충돌 가능성

- `update_user_role` RPC: `user` 값 추가 → admin에서 `user → player` 수동 승격 가능
- 본인인증 미구현 기간 동안 admin 수동 승격이 유일한 승격 경로
- admin UI(UserRoleForm) 변경은 이번 범위 최소화 — TypeScript 타입만 확장

### 본인인증 도입 전 임시 승격 정책 + 환경 플래그 계획

**현재 단계 (이번 범위)**:
- admin에서 수동 `user → player` 승격 가능하도록 RPC + 서버 액션 수정
- `src/lib/config/env.ts` 파일 생성 — 환경 플래그 기반 구조 마련

**VS-R8/VS-R9 (본인인증 UI 구현 시 적용 필수)**:
- UI는 프로덕션/비-프로덕션 구분 없이 동일하게 모두 구현
- 단, **실제 외부 인증 API 호출은 `requireIdentityVerification` 플래그로 분기**
- 비-프로덕션(`VERCEL_ENV !== "production"`)에서는 API 호출 스킵 → 바로 `user → player` 승격 처리
- 플래그 판별: `VERCEL_ENV` 환경변수 사용 (Vercel 배포 시 자동 주입, 로컬에서는 `.env.local`에 수동 설정 가능)

```
VERCEL_ENV=production   → 본인인증 API 실제 호출 필요
VERCEL_ENV=preview      → 스킵 (staging 환경)
VERCEL_ENV=development  → 스킵 (로컬 또는 Vercel dev 환경)
VERCEL_ENV 미설정        → 스킵 (로컬 개발, VERCEL_ENV는 Vercel 외부에서 미설정)
```

### 향후 확장 시 안전한 구조인지

- `is_player()` DB 함수를 기준으로 RLS 확장이 단순함
- `requirePlayer()` 가드를 서버 액션에 추가하는 패턴이 일관적
- `isPlayerRole()` 유틸을 통해 프론트엔드 분기도 일관된 방식으로 추가 가능
- `promote_to_player()` SECURITY DEFINER RPC 추가 시 이 구조와 자연스럽게 연결됨
- `src/lib/config/env.ts`는 본인인증 외에도 향후 환경별 기능 분기에 재사용 가능
