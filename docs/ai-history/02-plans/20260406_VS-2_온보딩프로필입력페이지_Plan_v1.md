## 1. 구현 기능 상세 설명

### 1-1. `/onboarding/profile/page.tsx` 역할

서버 컴포넌트. 온보딩 진입 시 다음 두 가지를 수행한다.

1. **인증 가드**: `getUserWithRole()` 호출 → 비로그인이면 `/login` redirect
2. **프로필 로딩**: `getMyProfile()` 호출 → 초기값을 Form 컴포넌트에 props로 전달

page.tsx 자체는 UI를 최소화하고, 레이아웃 컨테이너 + `ProfileForm` 렌더링이 전부다.

---

### 1-2. 프로필 입력 폼 필드

| 필드 | 타입 | 필수 여부 | 비고 |
|------|------|----------|------|
| `display_name` | text | 필수 | 표시 이름 (닉네임 또는 실명) |
| `phone` | text | 필수 | 연락처 (형식 검증은 최소화) |
| `birth_date` | date | 선택 | 생년월일 (YYYY-MM-DD) |

- `isProfileCompleted()` 기준: `display_name` + `phone` 두 필드 모두 비어 있지 않으면 완료
- `birth_date`는 선택이므로 입력 없이도 저장 가능
- 초기값은 page.tsx에서 전달 (기존 입력값 있으면 미리 채워줌)

---

### 1-3. 초기값 로딩 방식

```
page.tsx (서버 컴포넌트)
  → getMyProfile() 호출
  → data가 있으면 Form에 initialValues로 전달
  → data가 null / error이면 빈 값으로 전달
```

`getMyProfile()`은 RLS 의존 — 비로그인 상태면 error를 반환하지만, 이 경우 page.tsx의 인증 가드에서 이미 redirect되므로 Form에는 항상 로그인 사용자 데이터가 들어온다.

---

### 1-4. 저장 처리 방식

```
Form.tsx (클라이언트 컴포넌트)
  → handleSubmit: useState로 loading/error 관리
  → saveOnboardingProfile(input) Server Action 호출
  → 성공 시 router.push("/dashboard")
  → 실패 시 error 메시지 표시
```

Server Action (`actions.ts`):
- `getUserWithRole()` 재검증 (action 단에서도 인증 확인)
- `display_name`, `phone` 유효성 검사 (빈 값 차단)
- `updateMyProfile(input)` 호출
- 성공 시 `{ ok: true }` 반환 → 클라이언트에서 `router.push("/dashboard")`

> **redirect 위치**: Server Action에서 `redirect()`를 직접 호출하지 않고 `{ ok: true }` 반환 후 클라이언트가 `router.push()`. 이유: 폼 제출 흐름에서 서버 액션 내 redirect()는 클라이언트 전환 애니메이션이 없고, 에러 처리 분기가 복잡해진다. 기존 apply/Form.tsx 패턴 (`router.refresh()`) 과 일관성 유지.

---

### 1-5. 저장 성공/실패 시 사용자 흐름

| 케이스 | 처리 |
|--------|------|
| 저장 성공 | `router.push("/dashboard")` |
| 저장 실패 (서버 오류) | 폼 하단에 에러 메시지 표시, 페이지 유지 |
| 필수 필드 미입력 | 클라이언트 검증으로 즉시 차단, 에러 메시지 표시 |
| 인증 만료 중 저장 시도 | Server Action에서 `{ ok: false, error: "로그인이 필요합니다." }` 반환 |

---

### 1-6. 로그인 상태가 아닐 때

```typescript
// page.tsx
const userResult = await getUserWithRole();
if (userResult.status === "unauthenticated") redirect("/login");
```

page.tsx 서버 컴포넌트에서 즉시 redirect. Form 렌더링 전에 처리된다.

---

### 1-7. `profiles` 행이 없거나 비정상일 때

| 상태 | 처리 방식 |
|------|---------|
| `status === "empty"` | `/login` redirect (트리거 미동작 이상 케이스) |
| `getMyProfile()` error | Form에 빈 초기값으로 렌더링. 저장 시 action에서 재차 실패하면 에러 표시 |
| `getMyProfile()` data = null | Form에 빈 초기값으로 렌더링 (신규 가입 직후 정상 케이스) |

> `status === "empty"`: `getUserWithRole()`이 profiles 행 없는 상태를 반환하는 이론적 케이스. 현재 트리거가 정상 동작하면 발생하지 않는다. 안전하게 `/login`으로 보낸다.

## 2. 추가 라이브러리 필요 여부

**결론: 추가 라이브러리 불필요.**

| 필요 기능 | 현재 방식으로 충분한 이유 |
|----------|----------------------|
| 폼 상태 관리 | `useState` (loading, error, 각 필드값) |
| 비동기 처리 | `useState(loading)` + async handleSubmit |
| 서버 통신 | Server Action 직접 호출 |
| 유효성 검사 | 클라이언트: 빈 값 체크, 서버: trim 후 빈 값 체크 |
| 페이지 이동 | `useRouter().push()` |

기존 프로젝트의 모든 폼 (login, signup, apply, players)이 이 패턴을 사용한다. `react-hook-form`이나 `zod` 같은 라이브러리는 이번 VS에서 도입하지 않는다.

> 필드가 3개이고 복잡한 중첩 검증이 없으므로 현재 패턴으로 충분하다. `useTransition` 대신 `useState(loading)`을 사용하는 것도 apply/Form.tsx와 동일 패턴이므로 일관성 있다.

## 4. 파일별 핵심 로직 구조

### 4-1. `app/(app)/onboarding/profile/page.tsx`

**역할**: 인증 가드 + 프로필 초기값 로딩 후 Form 렌더링

**핵심 로직**:
```typescript
// 서버 컴포넌트
export const dynamic = "force-dynamic";

export default async function OnboardingProfilePage() {
  // 1. 인증 가드
  const userResult = await getUserWithRole();
  if (userResult.status === "unauthenticated") redirect("/login");
  if (userResult.status === "empty") redirect("/login");
  if (userResult.status === "error") return <ErrorUI />;

  // 2. 프로필 초기값 로드 (실패해도 빈 값으로 렌더링)
  const profileResult = await getMyProfile();
  const initialValues = profileResult.data ?? null;

  // 3. Form 렌더링
  return (
    <main className="...">
      <header>...</header>
      <ProfileForm initialValues={initialValues} />
    </main>
  );
}
```

**설계 포인트**:
- `force-dynamic`: 프로필 상태는 매 요청마다 최신값이어야 한다.
- `getMyProfile()` 에러는 Form에 null 전달로 처리. 치명적 에러로 취급하지 않는다. (신규 가입 직후 profiles 행은 있지만 display_name이 null인 상태가 정상)
- `(app)/layout.tsx`는 로그인 상태면 Sidebar를 렌더링한다. 온보딩 페이지도 Sidebar가 표시된다. **별도 layout 처리 없음** (이유: 5-1 참조).

---

### 4-2. `app/(app)/onboarding/profile/Form.tsx`

**역할**: display_name, phone, birth_date 입력 폼 + 저장 처리

**핵심 로직**:
```typescript
"use client";

// Props
type Props = {
  initialValues: Profile | null;
};

// 상태
const [displayName, setDisplayName] = useState(initialValues?.display_name ?? "");
const [phone, setPhone] = useState(initialValues?.phone ?? "");
const [birthDate, setBirthDate] = useState(initialValues?.birth_date ?? "");
const [loading, setLoading] = useState(false);
const [error, setError] = useState<string | null>(null);
const router = useRouter();

// 제출 핸들러
async function handleSubmit(e: React.FormEvent) {
  e.preventDefault();

  // 클라이언트 검증
  if (!displayName.trim()) { setError("이름을 입력해주세요."); return; }
  if (!phone.trim()) { setError("연락처를 입력해주세요."); return; }

  setLoading(true);
  setError(null);

  const result = await saveOnboardingProfile({
    display_name: displayName,
    phone,
    birth_date: birthDate || undefined,
  });

  if (!result.ok) {
    setError(result.error);
    setLoading(false);
    return;
  }

  router.push("/dashboard");
}
```

**UI 구조**:
```
<Card>
  <form>
    <Input label="이름" value={displayName} ... />
    <Input label="연락처" value={phone} ... />
    <Input label="생년월일 (선택)" type="date" value={birthDate} ... />
    {error && <p className="text-red-600">...</p>}
    <Button type="submit" disabled={loading}>
      {loading ? "저장 중…" : "저장하고 계속"}
    </Button>
  </form>
</Card>
```

**기존 컴포넌트 재사용**: `Card`, `Button`, `Input` (components/ui/)

### 4-3. `app/(app)/onboarding/profile/actions.ts`

**역할**: 프로필 저장 Server Action — 인증 재검증 + 유효성 검사 + `updateMyProfile()` 호출

**핵심 로직**:
```typescript
"use server";

export async function saveOnboardingProfile(
  input: ProfileUpdateInput
): Promise<ActionResult> {
  // 1. 인증 확인 (action 단에서 재검증)
  const userResult = await getUserWithRole();
  if (userResult.status !== "ready") {
    return { ok: false, error: "로그인이 필요합니다." };
  }

  // 2. 필수 필드 검증
  if (!input.display_name?.trim()) {
    return { ok: false, error: "이름을 입력해주세요." };
  }
  if (!input.phone?.trim()) {
    return { ok: false, error: "연락처를 입력해주세요." };
  }

  // 3. 저장 (lib/api/profiles.ts 위임)
  return updateMyProfile(input);
}
```

**설계 포인트**:
- `updateMyProfile()`에 인증 확인이 이미 있지만, action 단에서도 `getUserWithRole()` 재검증. 이유: 에러 메시지를 한국어로 통일하기 위해.
- `birth_date` 미입력 시 `undefined` 전달 → `updateMyProfile()`에서 `null`로 처리됨.
- redirect는 Server Action에서 하지 않는다. `{ ok: true }` 반환 후 Form에서 `router.push()`.

### 4-4. `lib/api/profiles.ts` 연결 방식


`isProfileCompleted()`는 이번 VS-2에서 직접 사용하지 않는다. (VS-3에서 login/actions.ts에서 사용)

## 5. 트레이드오프 및 고려 사항

### 5-1. 온보딩 페이지를 `(app)` 그룹에 둘지

**결정**: `(app)` 그룹에 둔다.

| 기준 | 판단 |
|------|------|
| 인증 필요 여부 | 필요 (로그인 사용자만 접근) |
| 기존 `(app)` layout 재사용 | 가능 (Sidebar + GlobalHeader) |
| 별도 레이아웃 필요 여부 | 불필요 — 온보딩 단계에서도 sidebar 표시는 자연스럽다 |

`(app)/layout.tsx`는 로그인 상태 기준으로 Sidebar를 렌더링한다. 온보딩 단계에서 Sidebar가 보이는 것은 어색할 수 있으나, 이번 VS-2 범위에서는 layout 커스터마이징을 하지 않는다. 향후 UX 개선 시 별도 `(onboarding)` 그룹으로 분리하거나 `(app)/onboarding/layout.tsx`를 추가할 수 있다.

---

### 5-2. 서버 컴포넌트에서 가드할지, 액션에서만 검증할지

**결정**: 양쪽 모두 검증한다.

- **page.tsx**: 비로그인 사용자 redirect (UX 목적)
- **actions.ts**: 인증 만료된 채 제출 시 에러 반환 (보안 목적)

`updateMyProfile()`에도 내부적으로 `getUser()` 호출이 있으므로 3중 검증이지만, 각 계층의 책임이 다르므로 중복이 아니다.

---

### 5-3. 프로필 완료 여부 판단을 이번 VS에서 어디까지 반영할지

이번 VS-2는 **입력 UI와 저장 흐름**에만 집중한다.

- 저장 완료 후 `isProfileCompleted()` 결과와 관계없이 `/dashboard`로 이동
- "완료 여부를 체크해서 온보딩을 강제하는 로직"은 VS-3에서 처리
- VS-2 단계에서 사용자는 필드를 비우고 저장할 수 없다 (필수 필드 검증으로 차단)

---

### 5-4. 저장 후 `/identity`로 보낼지 `/dashboard`로 보낼지

**결정**: 이번 VS-2에서는 `/dashboard`로 이동.

- VS-5(본인인증)가 아직 구현되지 않으므로 `/identity`로 보낼 수 없다.
- VS-5 구현 후, `/dashboard`에서 배너로 본인인증을 유도하거나 `login/actions.ts`에서 분기하는 방식으로 자연스럽게 연결된다.
- redirect 목적지를 나중에 바꾸기 쉽게 Form.tsx에서 `router.push("/dashboard")`를 상수로 관리하거나 actions.ts 반환값에 `redirectTo` 포함 검토 → **이번 VS에서는 단순하게 하드코딩** (상수화는 과도한 설계).

---

### 5-5. 기존 로그인/회원가입 흐름과 충돌 가능성

| 충돌 지점 | 현황 | 영향 |
|----------|------|------|
| `signup/actions.ts`가 `/dashboard`로 redirect | 이번 VS-2에서 수정 안 함 | 회원가입 후 온보딩 자동 연결은 VS-4까지 미루어짐 |
| `login/actions.ts`가 프로필 여부 무관 redirect | 이번 VS-2에서 수정 안 함 | 로그인 후 온보딩 유도는 VS-3에서 처리 |
| `/onboarding/profile` URL 직접 접근 | page.tsx에서 인증 가드 | 비로그인이면 `/login` redirect — 충돌 없음 |

**결론**: 기존 흐름에 영향 없음.

---

### 5-6. `/settings/profile`로 재사용 가능한 구조인지

**가능하다.** VS-3에서 `/settings/profile` 구현 시:

- `Form.tsx`의 props 인터페이스 (`initialValues: Profile | null`)는 그대로 재사용 가능
- `actions.ts`의 `saveOnboardingProfile`과 동일한 로직을 `updateProfileSettings`로 이름만 바꿔서 재사용
- 혹은 `Form.tsx`를 공통 컴포넌트(`components/profile/ProfileForm.tsx`)로 추출하여 양쪽에서 사용

이번 VS-2에서는 `app/(app)/onboarding/profile/` 디렉토리에 위치시키고, VS-3에서 재사용 구조를 판단한다.
