### 1-3. `is_identity_verified`를 이번 VS에 포함하지 않는 이유

`is_identity_verified` 컬럼을 이번에 추가하지 않는다. 이유는 다음과 같다.

1. **본인인증 흐름 없이 추가하는 컬럼은 의미가 없다.** 이 컬럼은 외부 인증 업체 연동(Server Action, 콜백 처리, SECURITY DEFINER 함수)과 함께 도입해야 의미가 있다. 데이터 구조만 미리 넣으면 관리 범위만 늘어난다.
2. **RLS 설계가 달라진다.** `is_identity_verified`는 사용자가 직접 UPDATE할 수 없어야 하므로, 이번 VS의 `profiles_update_own` 정책 검토와 방향이 다르다. 동시에 설계하면 복잡도가 급격히 올라간다.
3. **VS-1의 목표는 "사용자가 입력하고 수정 가능한 기본 정보 구조 정비"** 이다. `is_identity_verified`는 "시스템이 관리하는 검증 상태"로 성격이 다르다.

결론: VS-5에서 `identity_verifications` 테이블 + `profiles.is_identity_verified` 컬럼을 함께 설계하는 것이 더 일관성 있다.

## 4. 파일별 핵심 로직 설계

### 4-1. `supabase/migrations/0214_profiles_personal_info.sql`

**역할**: `profiles` 테이블에 사용자 기본 정보 컬럼 추가 + RLS 정책 검토

**핵심 결정사항**:

1. **컬럼 nullable 정책**: 모든 신규 컬럼은 `DEFAULT NULL`로 추가한다.
   - `on_auth_user_created` 트리거는 `INSERT INTO profiles (id) VALUES (new.id)`만 실행하므로, 신규 컬럼에 `NOT NULL`을 붙이면 트리거가 실패한다.
   - `display_name`, `phone`, `birth_date` 모두 `NULLABLE` 필수.

2. **RLS 정책 `profiles_update_own` 현황 및 판단**:
   - 현재 정책: `USING (id = auth.uid()) WITH CHECK (id = auth.uid())`
   - 이 정책은 row-level이지 column-level이 아니다. 즉, 사용자가 이론적으로 `role` 컬럼도 직접 UPDATE할 수 있다.
   - 단, 현재 앱 코드는 사용자가 `role`을 업데이트하는 경로를 노출하지 않으며, `update_user_role()` RPC는 `SECURITY DEFINER`로 별도 보호된다.
   - **VS-1에서 취할 방향**: `profiles_update_own` 정책을 유지하되, `updateMyProfile()` 함수 내에서 명시적으로 허용 컬럼만 업데이트하도록 API 레이어에서 제한한다. 이 제한은 이미 RPC 패턴으로 검증된 방식이다.
   - 추후 `is_identity_verified` 추가 시에는 SECURITY DEFINER 함수를 통해서만 업데이트하도록 설계할 예정 (VS-5).

3. **`profiles_select_own_or_organizer` 정책 검토**:
   - organizer는 현재 모든 `profiles` 행을 조회할 수 있다. 신규 컬럼(`display_name`, `phone`, `birth_date`) 추가 후에도 이 정책이 그대로 적용된다.
   - 이 앱의 context(대회 운영)에서 organizer가 참가자 연락처를 확인할 수 있는 것은 자연스럽다. 정책 변경 없이 유지한다.

**SQL 스키마 (pseudocode)**:

```sql
-- 0214_profiles_personal_info.sql
-- profiles 테이블에 사용자 기본 정보 컬럼 추가
-- 신규 컬럼은 모두 nullable (on_auth_user_created 트리거와 충돌 방지)

ALTER TABLE public.profiles
  ADD COLUMN IF NOT EXISTS display_name text,
  ADD COLUMN IF NOT EXISTS phone text,
  ADD COLUMN IF NOT EXISTS birth_date date;

-- 기존 RLS 정책 검토 결과: 변경 없이 유지
-- profiles_select_own_or_organizer: 본인 또는 organizer 조회 → 유지
-- profiles_update_own: 본인만 업데이트 (row-level) → 유지
--   단, API 레이어(updateMyProfile)에서 허용 컬럼을 명시적으로 제한

-- 코멘트 추가 (선택)
COMMENT ON COLUMN public.profiles.display_name IS '사용자 표시 이름 (사용자 수정 가능)';
COMMENT ON COLUMN public.profiles.phone IS '연락처 (사용자 수정 가능)';
COMMENT ON COLUMN public.profiles.birth_date IS '생년월일 (사용자 수정 가능)';
```

### 4-2. `lib/api/profiles.ts` (신규)

**역할**: `profiles` 테이블의 개인 정보 조회 및 업데이트 API 함수 모음

**포함 내용**:
- `Profile` 타입 정의
- `ProfileUpdateInput` 타입 정의 (업데이트 가능한 필드만)
- `getMyProfile()` — 현재 로그인 사용자의 전체 프로필 조회
- `getProfileById(userId)` — 특정 사용자 프로필 조회 (organizer용, RLS 의존)
- `updateMyProfile(input)` — 현재 사용자의 프로필 수정 (허용 컬럼만)
- `isProfileCompleted(profile)` — 프로필 완료 여부 판단 헬퍼

**타입 설계**:

```typescript
// 타입: DB에서 읽어오는 전체 프로필 구조
export type Profile = {
  id: string;
  role: Role;
  display_name: string | null;
  phone: string | null;
  birth_date: string | null;  // ISO 8601 date string (YYYY-MM-DD)
  created_at: string;
};

// 타입: 사용자가 업데이트 가능한 필드만
export type ProfileUpdateInput = {
  display_name?: string;
  phone?: string;
  birth_date?: string;  // YYYY-MM-DD
};
```

**`getMyProfile()` 설계**:

```typescript
export async function getMyProfile(): Promise<ApiResult<Profile>> {
  const supabase = await createSupabaseServerClient();
  const { data: { user } } = await supabase.auth.getUser();

  if (!user) return { data: null, error: "로그인이 필요합니다." };

  const { data, error } = await supabase
    .from("profiles")
    .select("id, role, display_name, phone, birth_date, created_at")
    .eq("id", user.id)
    .single();

  if (error) return { data: null, error: error.message };
  return { data: data as Profile, error: null };
}
```

**`updateMyProfile()` 설계**:

```typescript
export async function updateMyProfile(
  input: ProfileUpdateInput
): Promise<ActionResult> {
  // 1. 입력값 trim 처리
  const display_name = input.display_name?.trim() || null;
  const phone = input.phone?.trim() || null;
  const birth_date = input.birth_date?.trim() || null;

  // 2. 인증 확인
  const supabase = await createSupabaseServerClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) return { ok: false, error: "로그인이 필요합니다." };

  // 3. 허용된 컬럼만 명시적으로 UPDATE
  //    role, created_at 등 시스템 필드는 포함하지 않음
  const { error } = await supabase
    .from("profiles")
    .update({ display_name, phone, birth_date })
    .eq("id", user.id);

  if (error) return { ok: false, error: error.message };
  return { ok: true };
}
```

**`getProfileById()` 설계**:

```typescript
export async function getProfileById(
  userId: string
): Promise<ApiResult<Profile>> {
  const supabase = await createSupabaseServerClient();

  const { data, error } = await supabase
    .from("profiles")
    .select("id, role, display_name, phone, birth_date, created_at")
    .eq("id", userId)
    .maybeSingle();

  if (error) return { data: null, error: error.message };
  return { data: data as Profile | null, error: null };
}
// RLS 의존: organizer는 모든 프로필 조회 가능, 일반 사용자는 본인만
```

**`isProfileCompleted()` 설계**:

```typescript
// 프로필 완료 여부 판단
// 기준: display_name과 phone이 모두 채워져 있으면 완료
// birth_date는 선택 항목으로 취급 (향후 본인인증 시 필수화 가능)
export function isProfileCompleted(profile: Profile | null): boolean {
  if (!profile) return false;
  return (
    !!profile.display_name?.trim() &&
    !!profile.phone?.trim()
  );
}
```

**`isProfileCompleted()` 위치 결정 근거**:
- `src/lib/auth/roles.ts`에 두지 않는 이유: 역할(role) 판단과 프로필 완료 판단은 관심사가 다르다. `roles.ts`는 auth/role 계층을 담당하고, `profiles.ts`는 프로필 데이터 계층을 담당한다.
- 같은 파일에 배치하면 VS-3에서 `login/actions.ts`가 `profiles.ts` 하나만 import해도 된다.

## 5. 트레이드오프 및 고려사항

### 5-1. `profiles`에 모든 상태를 몰아넣을지, 일부는 별도 테이블로 분리할지

**결정**: `profiles`에 기본 정보만 추가, 본인인증 이력은 별도 테이블로 분리

| 방식 | 장점 | 단점 |
|------|------|------|
| profiles에 모두 추가 | JOIN 없이 단일 조회 | 인증 이력 관리 불가, 재인증 불가, RLS 복잡도 증가 |
| 별도 테이블 분리 | 인증 이력 보존, 재인증 처리, 상태 머신 구현 용이 | JOIN 필요, 설계 복잡도 증가 |

`display_name`, `phone`, `birth_date`는 단순 프로필 정보이므로 `profiles`에 추가. `is_identity_verified`, 인증 이력은 VS-5에서 `identity_verifications` 별도 테이블로 설계. `profiles.is_identity_verified`는 별도 테이블의 캐시 컬럼으로 추가 예정.

---

### 5-2. `is_profile_completed`를 DB 컬럼으로 둘지, TypeScript 계산 함수로 둘지

**결정**: TypeScript 계산 함수 (`isProfileCompleted(profile): boolean`)

**이유**:
- "완료 기준"이 향후 변경될 수 있다 (예: birth_date 필수화). DB 컬럼이면 마이그레이션이 필요하고, TypeScript 함수면 코드 한 줄 수정으로 끝난다.
- 현재 프로젝트의 `isOperationRole(role)` 패턴과 일관성이 있다.
- DB에 computed boolean을 별도로 관리하면 데이터 정합성 유지 비용이 발생한다 (display_name 업데이트 시 is_profile_completed도 UPDATE해야 하는 문제).

---

### 5-3. `profiles_update_own` 정책이 너무 넓게 열려 있지 않은지

**현황**: 현재 RLS 정책은 row-level 보호만 한다. 사용자가 이론적으로 `role`, `created_at` 등 모든 컬럼을 UPDATE할 수 있다.

**위험 평가**:
- `role` 변경: `update_user_role()` RPC는 SECURITY DEFINER로 보호되지만, 사용자가 직접 `supabase.from("profiles").update({role: "organizer"}).eq("id", user.id)`를 호출하면 RLS 정책 상 막히지 않는다.
- 즉, 현재 구조는 클라이언트 SDK를 직접 조작하는 공격에 취약하다.

**VS-1에서 취할 방향**:

**옵션 A**: RLS를 컬럼 단위로 제어하는 DB 트리거 추가
- PostgreSQL은 column-level RLS를 지원하지 않는다. 트리거로 구현해야 한다.
- 복잡도가 높고 유지보수가 어렵다.

**옵션 B**: `profiles`의 UPDATE를 SECURITY DEFINER 함수를 통해서만 허용
- `profiles_update_own` 정책을 제거하고, `update_my_profile(display_name, phone, birth_date)` RPC만 허용.
- 안전하지만 모든 프로필 수정을 RPC로 강제해야 하는 제약이 생긴다.

**옵션 C**: 현재 `profiles_update_own` 유지, API 레이어에서 허용 필드만 전달
- `updateMyProfile(input: ProfileUpdateInput)`에서 명시적으로 `{ display_name, phone, birth_date }`만 UPDATE.
- 클라이언트 SDK 직접 조작은 막지 못하지만, 서버 API를 통한 경로는 안전.

**결정**: **옵션 C**를 우선 적용. 이유:
1. 현재 앱은 Server Components + Server Actions 중심이라 클라이언트가 Supabase SDK를 직접 호출하지 않는다.
2. 브라우저 환경에서 `NEXT_PUBLIC_SUPABASE_ANON_KEY`로 직접 API 호출하는 것은 막을 수 없지만, 이것은 anon key 자체의 한계이며 현재 프로젝트 구조의 한계이기도 하다.
3. 나중에 `role` 컬럼 보호가 필요하다면 별도 마이그레이션에서 트리거 방식으로 추가한다.

> **주의 사항 기록**: `profiles.role` 직접 UPDATE 취약점이 존재함. 현재 범위에서는 Server Action 레이어로 보호하고, 추후 DB 트리거 방식으로 보강하는 것을 기술 부채로 기록.

### 5-4. `on_auth_user_created` 트리거와 신규 컬럼 충돌 방지

```sql
-- 현재 트리거 (변경 없음)
CREATE TRIGGER on_auth_user_created
AFTER INSERT ON auth.users
FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();

-- handle_new_user() 함수 (변경 없음)
INSERT INTO public.profiles (id)
VALUES (new.id)
ON CONFLICT (id) DO NOTHING;
```

- `INSERT INTO profiles (id)`는 `id`만 명시적으로 삽입한다.
- 나머지 컬럼은 각자의 DEFAULT 값이 적용된다.
- `role DEFAULT 'player'` → 정상 적용
- `display_name DEFAULT NULL` → 정상 적용 (NULL 허용)
- `phone DEFAULT NULL` → 정상 적용
- `birth_date DEFAULT NULL` → 정상 적용

**결론**: `DEFAULT NULL`로 추가하면 트리거와 충돌 없다.

### 5-5. `getUserWithRole()` 구조를 계속 유지하는 것이 좋은지

**결정**: 유지한다.

현재 `getUserWithRole()`은 다음 목적에 최적화되어 있다:
- "이 사용자가 로그인했는가?" (status: unauthenticated)
- "이 사용자의 역할은 무엇인가?" (role: organizer/manager/player)

프로필 완료 여부는 다른 관심사이므로 별도 함수로 분리한다. VS-2, VS-3에서 필요한 페이지는 두 함수를 각각 호출한다:

```typescript
// 예: 로그인 후 분기가 필요한 VS-3에서
const [roleResult, profileResult] = await Promise.all([
  getUserWithRole(),
  getMyProfile(),
]);
```

`Promise.all`로 병렬 호출하므로 성능 차이는 미미하다.
