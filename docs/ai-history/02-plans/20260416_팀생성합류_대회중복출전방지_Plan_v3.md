### Phase 1. DB 스키마 / 마이그레이션 [완료]

#### 1-1. `supabase/migrations/0246_rosters.sql` ← v3에서 테이블 구조 변경

**`rosters` 테이블 (v3 간소화)**:
```sql
컬럼:
  id               uuid PK DEFAULT gen_random_uuid()
  application_id   uuid NOT NULL REFERENCES tournament_team_applications(id) ON DELETE CASCADE
  team_id          uuid NOT NULL REFERENCES teams(id) ON DELETE RESTRICT
  tournament_id    uuid NOT NULL REFERENCES tournaments(id) ON DELETE RESTRICT
  created_at       timestamptz NOT NULL DEFAULT now()
  updated_at       timestamptz NOT NULL DEFAULT now()

제약:
  UNIQUE(application_id)  -- 신청 1건당 로스터 1개
```

**v2 대비 제거된 컬럼**: `status`, `submitted_at`, `confirmed_at`, `confirmed_by`, `rejection_reason`

**설계 결정 이유**:
- 로스터는 "선수 명단" 그 자체. 상태 머신 불필요
- 편집 가능 여부는 `tournaments.start_date > now()` 조건으로 런타임 판단
- `application_id` UNIQUE: 신청 1건 = 로스터 1개 (1:1)
- `team_id`, `tournament_id` denormalize: 중복 출전 검사 JOIN 최소화

**`roster_members` 테이블 (v2와 동일)**:
```sql
컬럼:
  id          uuid PK DEFAULT gen_random_uuid()
  roster_id   uuid NOT NULL REFERENCES rosters(id) ON DELETE CASCADE
  user_id     uuid NOT NULL REFERENCES auth.users(id) ON DELETE RESTRICT
  created_at  timestamptz NOT NULL DEFAULT now()

제약:
  UNIQUE(roster_id, user_id)   -- 같은 로스터 내 중복 선수 방지
```

**인덱스**:
```sql
-- 중복 출전 검사용
CREATE INDEX idx_roster_members_user ON roster_members(user_id) INCLUDE (roster_id);

-- 로스터 조회용
CREATE INDEX idx_rosters_tournament ON rosters(tournament_id);
```

**롤백**:
```sql
DROP TABLE IF EXISTS roster_members;
DROP TABLE IF EXISTS rosters;
```

#### 1-2. `supabase/migrations/0247_rosters_rls.sql` ← v3에서 status 조건 제거

**`rosters` 테이블 정책**:
```
ENABLE ROW LEVEL SECURITY;

SELECT: is_team_member_for_team(team_id) OR is_organizer()
INSERT: is_team_manager_for_team(team_id)   ← status = 'draft' 조건 제거
UPDATE: is_team_manager_for_team(team_id) OR is_organizer()
DELETE: is_team_manager_for_team(team_id) OR is_organizer()
```

**`roster_members` 테이블 정책**:
```
ENABLE ROW LEVEL SECURITY;

SELECT:
  - EXISTS(SELECT 1 FROM rosters r WHERE r.id = roster_id AND is_team_member_for_team(r.team_id))
  - OR is_organizer()

INSERT:
  - EXISTS(SELECT 1 FROM rosters r WHERE r.id = roster_id AND is_team_manager_for_team(r.team_id))
  ← status 조건 제거 (tournament start 체크는 RPC에서)

DELETE:
  - EXISTS(SELECT 1 FROM rosters r WHERE r.id = roster_id AND is_team_manager_for_team(r.team_id))
```

**재사용**: `public.is_team_member_for_team()`, `public.is_team_manager_for_team()`, `public.is_organizer()`

**롤백**:
```sql
DROP POLICY IF EXISTS ... ON rosters;
DROP POLICY IF EXISTS ... ON roster_members;
ALTER TABLE rosters DISABLE ROW LEVEL SECURITY;
ALTER TABLE roster_members DISABLE ROW LEVEL SECURITY;
```

#### 1-3. `supabase/migrations/0248_roster_rpcs.sql` ← v3에서 RPC 3개로 축소

**v2 대비 변경**: `submit_roster`, `confirm_roster` 제거. RPC 5개 → 3개.

**중복 출전 방지 기준 이동**: `confirm_roster` 삭제 → `add_roster_member` 시점에 즉시 검사.

---

**1) `upsert_roster(p_application_id uuid) RETURNS jsonb`**

```
목적: application_id에 대한 로스터가 없으면 생성, 있으면 기존 반환
호출자: team captain

로직:
  1. application의 team captain 확인 (team_members.role_in_team = 'captain')
  2. application.status IN ACTIVE_STATUSES 확인
     → cancelled, expired면 에러: "취소되거나 만료된 신청에는 로스터를 생성할 수 없습니다"
     → ACTIVE_STATUSES = ('payment_pending','paid_pending_approval','confirmed','waitlisted')
  3. SELECT * FROM rosters WHERE application_id = p_application_id
  4. 있으면 RETURN {ok: true, roster_id: existing.id, created: false}
  5. 없으면:
     - application에서 team_id, tournament_id 조회
     - INSERT INTO rosters (application_id, team_id, tournament_id)
     - RETURN {ok: true, roster_id: new.id, created: true}
```

---

**2) `add_roster_member(p_roster_id uuid, p_user_id uuid) RETURNS jsonb`** ← 핵심 변경

```
목적: 로스터에 선수 추가 + 중복 출전 방지 검사 (즉시)
호출자: team captain

로직:
  1. roster 조회 + team의 captain 확인
  2. tournament.start_date > now() 확인
     → 대회 시작 후면 에러: "대회가 이미 시작되어 로스터를 수정할 수 없습니다"
     (roster.tournament_id → tournaments.start_date 조회)
  3. p_user_id가 해당 팀의 team_members에 있는지 확인
     → 팀원이 아니면 에러: "팀 소속 선수만 로스터에 추가할 수 있습니다"
  4. ===== 핵심: 중복 출전 방지 검사 =====
     SELECT rm.user_id
     FROM roster_members rm
     JOIN rosters r ON r.id = rm.roster_id
     WHERE r.tournament_id = v_tournament_id
       AND r.team_id != v_team_id
       AND rm.user_id = p_user_id
     → 결과가 있으면:
       RETURN {ok: false, error: "이미 같은 대회 다른 팀 로스터에 포함된 선수입니다"}
  5. INSERT INTO roster_members (roster_id, user_id)
     (UNIQUE 제약으로 같은 로스터 내 중복 자동 방지)
  6. RETURN {ok: true}
```

---

**3) `remove_roster_member(p_roster_id uuid, p_user_id uuid) RETURNS jsonb`**

```
목적: 로스터에서 선수 제거
호출자: team captain

로직:
  1. roster 조회 + captain 확인
  2. tournament.start_date > now() 확인
     → 대회 시작 후면 에러: "대회가 이미 시작되어 로스터를 수정할 수 없습니다"
  3. DELETE FROM roster_members WHERE roster_id = p_roster_id AND user_id = p_user_id
  4. RETURN {ok: true}
```

#### 4-5. `app/(app)/my-applications/[applicationId]/page.tsx` (신규) ← v3에서 UI 대폭 단순화

**목적**: 특정 대회 신청 상세 + 로스터 편집 통합 페이지.

**페이지 데이터 로딩**:

```
3. getMyRoleInTeam(teamId) → captain/player 구분
4. getOrCreateRoster(applicationId) → ACTIVE_STATUSES면 자동 생성

6. 팀 전체 멤버 목록 (team_members) → 로스터 미포함 멤버 구분용
7. getTournament(tournamentId) → tournament.start_date (잠금 여부 판단)
```

**UI 구조 (v3 간소화)**:
```
상단: 신청 현황 섹션
  - 대회명 + 부문명
  - 팀명
  - 신청 상태 배지 + 상태별 안내 (payment_due_at 카운트다운 등)
  - 입금 표시 버튼 (payment_pending이고 captain인 경우)

하단: 로스터 섹션 (application.status IN ACTIVE_STATUSES면 표시)

  잠금 상태 분기:
    - tournament.start_date <= now():
      "대회가 시작되어 로스터를 수정할 수 없습니다." + 멤버 목록 표시 (읽기 전용)
    - tournament.start_date > now(): 편집 가능

  로스터 내용 (상태 배지 없음 ← v2 대비 제거):
    - 등록된 선수 목록:
      - 선수명, 포지션
      - "제거" 버튼 (편집 가능 상태이고 captain인 경우)
    - 팀 멤버 중 로스터 미포함 목록:
      - "추가" 버튼 (편집 가능 상태이고 captain인 경우)
      - 클릭 시 → addMemberAction() → 중복 출전이면 에러 인라인 표시:
        "이미 같은 대회 다른 팀 로스터에 포함된 선수입니다"

  로스터 섹션 비표시 조건:
    - application.status IN ('cancelled', 'expired')
    → "신청이 취소 또는 만료되었습니다." 메시지만 표시

  ← v2 대비 제거: 로스터 상태 배지, "로스터 제출" 버튼 없음
```

## 6. 리스크 및 주의사항

### 6-1. `getUserTeamStatus` 다중 팀 호환성

v2와 동일. `app/(app)/layout.tsx`에서 `listMyTeams()` 기반으로 교체 시 `getUserTeamStatus`의 모든 사용처 확인 필요.

### 6-2. RLS의 roster_members 정책 복잡도

v2와 동일. RPC(SECURITY DEFINER)를 통한 INSERT/DELETE 권장. RLS는 SELECT 정책 중심.

### 6-3. `add_roster_member` 중복 출전 검사 성능

**문제**: `add_roster_member` RPC의 중복 검사 쿼리:
```sql
SELECT rm.user_id
FROM roster_members rm
JOIN rosters r ON r.id = rm.roster_id
WHERE r.tournament_id = v_tournament_id
  AND r.team_id != v_team_id
  AND rm.user_id = p_user_id
```

**v2와의 차이**: v2에서는 `confirm_roster`에서 한 번만 실행되던 쿼리가, v3에서는 멤버 추가 때마다 실행됨.

**대응**: `idx_roster_members_user (user_id) INCLUDE (roster_id)` + `idx_rosters_tournament (tournament_id)` 인덱스로 Index Scan 보장. 대회당 팀 수가 수십~수백 수준이므로 성능 이슈 없음.

### 6-4. 신청 취소/만료 시 로스터 자동 삭제

**문제**: application이 cancelled/expired가 되면 `ON DELETE CASCADE`로 roster + roster_members 모두 자동 삭제. 이미 구성한 로스터가 사라짐.

**v2 대비**: v3에서는 로스터를 payment_pending 시점부터 구성할 수 있으므로, 더 이른 시점에 데이터 손실 리스크 발생.

**대응**: 
- 취소/만료 시 UI에서 "로스터 정보가 함께 삭제됩니다" 경고 표시
- expire_overdue_applications Edge Function 실행 시 관련 알림 포함 (기존 notification 흐름 활용)

### 6-5. 팀 멤버 변경과 로스터 정합성

v2와 동일. 팀 탈퇴 기능 미구현이므로 MVP에서는 고려하지 않음.

### 6-6. 신청 폼의 초기 로스터 구성 UI 범위

v2와 동일. 신청 폼의 선수 선택은 UX 미리보기. 실제 저장은 `/my-applications/[applicationId]` 진입 후 `getOrCreateRoster()` 시점. MVP에서는 신청 폼 체크박스 선택 결과 전달 없이 사용자가 직접 추가하는 방식(옵션 C) 권장.

### 6-7. Sidebar `teamId` 파라미터 제거 영향

v2와 동일. `SidebarProps`에서 `teamRole`, `teamId` → `hasTeam`, `isCaptain`.

### 6-8. `is_team_manager_for_team` 함수명 주의

v2와 동일. 내부는 `role_in_team = 'captain'` 체크이나 함수명 그대로 유지.

### 6-9. division vs tournament 단위 중복 방지

v2와 동일. `add_roster_member`의 중복 검사는 `tournament_id` 단위. UI에서 명확히 안내 필요.

## 7. 결론

**v3 핵심 변경**: 로스터 status 전면 제거 → 단순 멤버 명단으로 설계 단순화.

**v3 확정 로스터 정책**:
```
신청 제출 (ACTIVE_STATUSES) → 로스터 생성 + 멤버 추가/제거 가능
  ↓ (대회 시작 전까지 자유롭게 편집)
tournament.start_date 도달 → 자동 잠금 (편집 불가, 읽기만)
  ↓
신청 취소/만료 → roster + roster_members cascade 삭제
```

**중복 출전 방지**: `add_roster_member` 시점에 즉시 검사 → 즉각적 피드백 제공.

**총 신규 파일**: 9개 (v2와 동일)
**총 수정 파일**: 9개 (v2와 동일)
**DB 마이그레이션**: 3개 (0246, 0247, 0248)
**로스터 RPC**: 3개 (v2 5개 → v3 3개로 축소)
**추가 패키지**: 없음
