## 3. Phase별 구현 계획

### Phase 1. DB 스키마 / 마이그레이션

#### 신규 마이그레이션 파일

**`supabase/migrations/0241_team_join_applications.sql`**

- `team_join_applications` 테이블 신규 생성
  - 컬럼: `id uuid PK`, `team_id uuid FK→teams`, `applicant_id uuid FK→auth.users`, `status text CHECK('pending'|'approved'|'rejected')`, `created_at timestamptz`, `updated_at timestamptz`
  - 유니크 제약: `(team_id, applicant_id)` — 같은 팀에 중복 신청 방지
  - 인덱스: `team_id`, `applicant_id`, `(team_id, status)`
  - RLS 활성화

- 핵심 RPC 추가: `apply_for_team(p_team_id uuid) RETURNS jsonb`
  - `SECURITY DEFINER`, `row_security = off`
  - 이미 팀 멤버인지 확인 → 이미 신청 중인지 확인 → `team_join_applications` INSERT
  - 반환: `{ ok: true }` 또는 `{ ok: false, error: '...' }`

- 핵심 RPC 추가: `approve_team_application(p_application_id uuid) RETURNS jsonb`
  - 호출자가 해당 팀의 captain인지 확인
  - `team_join_applications.status = 'approved'` UPDATE
  - `team_members` INSERT (`role_in_team = 'player'`)
  - 반환: `{ ok: true }` 또는 `{ ok: false, error: '...' }`

- 핵심 RPC 추가: `reject_team_application(p_application_id uuid) RETURNS jsonb`
  - 호출자가 해당 팀의 captain인지 확인
  - `team_join_applications.status = 'rejected'` UPDATE
  - 반환: `{ ok: true }` 또는 `{ ok: false, error: '...' }`

**롤백 방법**: `DROP TABLE IF EXISTS public.team_join_applications CASCADE;` + `DROP FUNCTION IF EXISTS public.apply_for_team(uuid);` 등

---

**`supabase/migrations/0242_team_join_applications_rls.sql`**

- `team_join_applications` RLS 정책:
  - SELECT: 본인 신청 OR 해당 팀 captain (`is_team_manager_for_team(team_id)`)
  - INSERT: RPC를 통해서만 (직접 INSERT 불허 — policy 없음, RPC SECURITY DEFINER)
  - UPDATE: RPC를 통해서만
  - DELETE: 신청자 본인만 (status = 'pending'일 때)

---

**`supabase/migrations/0243_fix_createteam_rpc_call.sql`**

- 코드 레벨에서 `create_team_with_captain` 호출로 전환 시, 이미 0207 마이그레이션에서 `create_team_with_captain` RPC가 존재하므로 DB 마이그레이션 불필요.
- 단, `create_team_with_manager` wrapper를 명시적으로 deprecated 표시하는 코멘트 마이그레이션만 생성.

### Phase 5. 권한 정책 (RLS)

**`supabase/migrations/0242_team_join_applications_rls.sql`** 에 포함:

```sql
-- team_join_applications SELECT:
-- 본인 신청 or 해당 팀 captain
CREATE POLICY "team_join_applications_select"
ON public.team_join_applications FOR SELECT
USING (
  applicant_id = auth.uid()
  OR public.is_team_manager_for_team(team_id)
);

-- team_join_applications INSERT:
-- RPC를 통해서만 처리 (SECURITY DEFINER RPC)
-- 단, 직접 INSERT도 허용 시: applicant_id = auth.uid() AND status = 'pending'
CREATE POLICY "team_join_applications_insert"
ON public.team_join_applications FOR INSERT
WITH CHECK (applicant_id = auth.uid() AND status = 'pending');

-- team_join_applications UPDATE:
-- RPC SECURITY DEFINER로만 처리 → 직접 UPDATE 정책 없음

-- team_join_applications DELETE:
-- 본인이 pending 신청만 철회 가능
CREATE POLICY "team_join_applications_delete"
ON public.team_join_applications FOR DELETE
USING (applicant_id = auth.uid() AND status = 'pending');
```

**teams 테이블 기존 정책 확인**:
- `0208_teams_rls_player.sql`, `0216_teams_rls_player_guard.sql` 정책이 player role의 SELECT를 허용하는지 확인 필요
- `/teams/find` 페이지에서 팀 목록 조회 시 player가 teams를 SELECT할 수 있어야 함

## 6. 리스크 및 주의사항

### 6-1. `lib/api/teams.ts`의 `createTeam()` RPC 호출 불일치

**문제**: `createTeam()`이 현재 `create_team_with_manager` RPC를 호출하지만, DB에서는 0207 마이그레이션 이후 `create_team_with_manager`가 내부적으로 `create_team_with_captain`으로 위임하므로 기능은 동일함.  
**그러나**: 코드 가독성을 위해 `create_team_with_captain` 직접 호출로 변경.  
**주의**: 변경 후 기존 `createDummyTeam()`은 RPC를 사용하지 않으므로 영향 없음.

### 6-2. `teams/[teamId]/page.tsx`의 `isManager` 버그

**문제**: `const isManager = roleResult.role === "manager"` — 실제 DB 값은 `'captain'`이므로 항상 false.  
**영향**: 캡틴이 팀 상세에서 선수 추가/삭제 등 관리 기능을 사용할 수 없음.  
**수정**: `"manager"` → `"captain"` 으로 즉시 수정.

### 6-3. `team_join_applications` 유니크 제약 처리

**문제**: `(team_id, applicant_id)` 유니크 제약으로 인해 rejected 후 재신청 불가.  
**권장**: status = 'rejected'인 경우 기존 레코드를 'pending'으로 UPDATE하거나, 유니크 제약을 `(team_id, applicant_id, status = 'pending')` partial index로 변경.  
**결정**: RPC `apply_for_team` 내부에서 rejected 레코드가 있으면 pending으로 UPDATE하는 upsert 방식 채택.

### 6-4. `/teams/find` 페이지의 teams SELECT RLS

**문제**: player role이 자신이 속하지 않은 팀도 조회해야 함.  
**현재 RLS**: `teams_select_members` 정책이 `is_team_member_for_team(id)` 기반이므로, 비멤버 팀은 SELECT 불가.  
**해결**: `getTeamsForJoin()` 함수를 SECURITY DEFINER RPC로 구현하거나, teams 테이블에 "player는 모든 팀 SELECT 가능" 정책 추가.  
**권장**: RPC(`get_teams_for_join`) SECURITY DEFINER 방식 채택 — 기존 RLS 정책 구조 보존.  
**추가 마이그레이션 필요**: `0241`에 `get_teams_for_join` RPC 포함.

### 6-5. 온보딩 플로우 중간 이탈 처리

**문제**: 사용자가 `/onboarding/team-choice`에서 나중에 하기를 눌러 대시보드로 이동한 후 재진입 경로.  
**구현**: 대시보드 배너에 "팀 없음 상태" UI로 `/teams/find` 또는 `/teams/new` 링크 제공.  
**주의**: `getUserTeamStatus()`가 대시보드 로드마다 호출되므로 성능 영향 최소화 (단순 SELECT 쿼리 2개).

### 6-6. 동시 합류 신청 방지

**문제**: 한 팀에 동시에 여러 승인이 일어나거나, 이미 다른 팀 멤버가 된 사용자가 승인될 수 있음.  
**해결**: `approve_team_application` RPC 내부에서:
1. `team_join_applications.status = 'pending'` 재확인 (FOR UPDATE 잠금)
2. 신청자가 이미 다른 팀 멤버인지 확인 (MVP에서는 멀티팀 허용 여부 결정 필요)
**MVP 결정**: 멀티팀 허용 (한 선수가 여러 팀 소속 가능) — 제약 없음.

### 6-7. 회원가입 완료 후 온보딩 completion 페이지 진입 경로

**문제**: 현재 `/signup` 완료 후 이메일 인증 대기 화면이 표시됨.  
**현재 구조 확인 필요**: `app/(auth)/signup/Form.tsx`에서 성공 후 어디로 이동하는지 확인 후, `/onboarding/completion` 리디렉트 추가.  
**주의**: 이메일 인증이 필요한 경우, 인증 완료 후 `/onboarding/completion`으로 이동해야 하므로 Supabase Auth redirect URL 설정 확인 필요.
