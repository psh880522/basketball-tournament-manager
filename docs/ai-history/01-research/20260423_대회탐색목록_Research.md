## 3. 현재 구현 상태

### `app/(app)/tournaments/page.tsx`

현재 구현은 **매우 단순**하며 기능 요구사항과 큰 격차가 있다.

- **데이터 로딩**: `getOpenTournaments()` + `getMyParticipatedTournaments()` 두 함수만 사용
- **UI 구조**: 고정된 3개 섹션 — "모집중인 대회", "참가중인 대회", "참가했던 대회"
- **`TournamentCard`**: 이름, 날짜 범위, 장소, 팀명 뱃지만 표시
- **없는 것**: 검색, 필터, 탭 전환, 페이지네이션, 참가비/정원/마감일 표시, 카드별 신청 상태 표시

### `app/(public)/page.tsx` (랜딩)

- open + closed 대회를 별도 섹션으로 가로 스크롤 카드로 표시
- `DragScroll` 래퍼 사용
- Suspense 스켈레톤 패턴 사용 중
- 비인증 사용자는 카드 클릭 시 "/login"으로 리다이렉트

### `lib/api/tournaments.ts` 현재 함수 목록

| 함수명 | 동작 |
|---|---|
| `getOpenTournaments()` | `status='open'`, `deleted_at IS NULL`, `start_date ASC` |
| `getInProgressTournaments()` | `status='closed'` + 미완료 match 있는 대회 |
| `getMyParticipatedTournaments()` | team_members → confirmed 신청 → closed/finished 대회 (분리 반환) |
| `getPublicTournamentById(id)` | 단건 (open/closed/finished 상태) |
| `listAdminTournaments({includeDeleted})` | admin용 전체 |
| `getTournamentForEdit(id)` | edit용 단건 |

## 4. 재사용 가능한 패턴

### 4-1. URL searchParams 기반 서버 필터 (`matches/Filters.tsx`)

```tsx
// URL에 ?division=...&court=... 반영 후 서버 컴포넌트가 재렌더링
const handleChange = (key: string, value: string) => {
  const params = new URLSearchParams(searchParams)
  params.set(key, value)
  router.push(`?${params.toString()}`)
}
```

→ 상태별 탭 전환, 지역/날짜/형식 필터에 동일하게 적용 가능

### 4-2. 클라이언트 실시간 검색 (`teams/find/SearchSection.tsx`)

```tsx
const [query, setQuery] = useState('')
const filtered = teams.filter(t => t.team_name.toLowerCase().includes(query))
```

→ 대회 이름 키워드 검색에 유사하게 적용 가능 (소규모 목록인 경우)  
→ 대규모 목록이면 서버 `.ilike()` 쿼리로 대체 필요

### 4-3. Suspense 스켈레톤 (`app/(public)/page.tsx`)

```tsx
<Suspense fallback={<Skeleton />}>
  <TournamentSection />
</Suspense>
```

→ 필터 변경 시 서버 재요청 중 스켈레톤 표시에 재사용 가능

### 4-4. `DragScroll` 컴포넌트

→ 모바일에서 대회 카드 가로 스크롤에 이미 검증된 방식

### 4-5. `getDivisionApplicationCounts(tournamentId)`

```typescript
// 반환: Record<divisionId, count>
// 'payment_pending' | 'paid_pending_approval' | 'confirmed' 상태 집계
```

→ 대회 카드의 "현재 신청 수" 표시에 활용 가능 (단, 대회 목록 단위에서는 N+1 쿼리 문제 주의)

### 4-6. `listAllMyTeamApplications()`

→ 내 모든 팀의 모든 신청 목록 반환 → 카드별 신청 상태 오버레이에 활용 (서버에서 한번 조회 후 프론트 매핑)

### 5-5. 권한 정책 (RLS)

**추가 마이그레이션 불필요** — 기존 RLS로 충분

| 테이블 | 정책 | 확인 사항 |
|---|---|---|
| `tournaments` | `status IN ('open','closed','finished')` → 전체 공개 | 대회 목록은 비인증 사용자도 조회 가능 |
| `divisions` | `true` → 전체 공개 | 마감일 포함 조회 가능 |
| `tournament_team_applications` | `is_organizer() OR is_team_member_for_team(team_id)` | 내 팀 신청만 SELECT 가능 → 목록에서 신청상태 조회는 반드시 인증 사용자 기준 |

**주의**: 비인증 사용자에게도 대회 목록은 보여야 하므로, 신청 상태 표시 부분만 조건부 렌더링 필요

## 7. 구현 시 주의사항

### 7-1. `closed` 상태 한국어 표기 불일치

현재 코드베이스 내에 `closed` 상태가 혼용됨:
- 관리자 화면: **"진행중"**
- 일반 사용자 화면: **"모집마감"**

대회 탐색 목록에서 `closed`를 어떤 이름으로 표시할지 정책 결정 필요. 사용자 관점에서 `closed`는 "신청은 마감됐지만 경기는 진행중"이므로 "진행중"이 더 자연스러울 수 있음.

### 7-2. N+1 쿼리 문제 (대회 목록 + 신청 수)

대회 카드마다 신청 수를 표시하려면 각 대회에 대해 `getDivisionApplicationCounts()` 호출 시 N+1 발생.  
해결 방안:
- Supabase 관계형 select + count 집계를 한 쿼리로 구성
- 또는 별도 Supabase RPC 함수로 집계
- 또는 초기 MVP에서는 capacity만 표시 (신청 수 생략)

### 7-3. 사용자 신청 상태 표시 RLS 제약

`tournament_team_applications`의 SELECT는 내 팀 소속 멤버만 가능.  
목록 페이지에서 각 카드에 신청 상태를 표시하려면 `listAllMyTeamApplications()`로 한 번 가져와서 프론트에서 매핑하는 방식이 효율적.  
비인증 사용자는 신청 상태 표시 영역 자체를 렌더링하지 않아야 함.

### 7-4. 필터 상태 관리 방식 결정

두 가지 전략 중 선택 필요:

| 전략 | 장점 | 단점 |
|---|---|---|
| URL searchParams | 새로고침/공유 가능, SSR 활용 | router.push() 비용, 초기 구현 복잡도 |
| useState (클라이언트) | 단순 구현, 즉각 반응 | URL 공유 불가, 새로고침 시 리셋 |

→ 상태 탭(모집중/진행중/종료/내 신청)은 URL param 권장 (SEO, 공유, 뒤로 가기 고려)  
→ 실시간 키워드 검색은 debounce + useState 권장

### 7-5. `divisions.application_close_at` 누락

현재 `getDivisionsByTournament()` 함수의 Supabase select 쿼리에 `application_open_at`, `application_close_at`가 포함되어 있지 않음. 대회 카드에 신청 마감일을 표시하려면 반드시 이 컬럼을 select에 추가해야 함.

### 7-6. 개인화 데이터 한계

사용자 `profiles`에 지역(region) 정보 없음 → 지역 기반 개인화 추천 불가.  
팀의 `region` 필드는 있으나 사용자 본인 지역이 아닌 팀 지역이므로 부정확할 수 있음.  
→ MVP에서는 지역 필터를 "사용자 선택" 방식으로만 구현, 자동 추천은 제외.

### 7-7. 바텀시트 컴포넌트 부재

현재 프로젝트에 바텀시트(BottomSheet/Sheet) 전용 컴포넌트 없음.  
구현 옵션:
- Radix UI Dialog를 활용하여 모바일 환경에서 하단 고정 스타일로 커스터마이징
- 직접 구현 (fixed + transform slide-up 애니메이션)
- Tailwind CSS만으로 조건부 `translate-y` 제어

## 9. 결론

### 현재 상태 요약

`/tournaments` 페이지는 골격만 있는 상태로, 기능 요구사항 대비 **대부분이 미구현**이다.  
DB 스키마, RLS 정책, 공통 UI 컴포넌트는 이미 잘 갖춰져 있어 프론트엔드 레이어 위주의 작업이 될 것이다.

### 핵심 GAP

| 요구사항 | 현재 상태 | 필요한 작업 |
|---|---|---|
| 상태별 탭 (모집중/진행중/종료/내 신청) | 고정 3섹션 | FilterBar + searchParams |
| 키워드 검색 | 없음 | SearchBar + `.ilike()` 쿼리 |
| 지역/날짜/형식 필터 | 없음 | FilterBottomSheet + 쿼리 |
| 카드에 참가비/마감일/정원 표시 | 없음 | API 확장 + TournamentCard 재작성 |
| 카드별 내 팀 신청 상태 표시 | 없음 | `listAllMyTeamApplications()` 매핑 |
| 모바일 필터 바텀시트 | 없음 | 신규 컴포넌트 |
| 카드형/리스트형 전환 | 없음 | view mode state |

### 작업 규모 예측

- Phase 1~3 (필수): **4~6개 파일 신규, 3개 파일 수정** — 중간 규모  
- Phase 4 (선택): 추가 2~3개 컴포넌트

### 리스크 포인트

1. 목록 + 신청 수 집계 N+1 쿼리 — MVP에서는 capacity만 표시하고 신청 수는 생략 권장
2. 바텀시트 컴포넌트 부재 — 별도 구현 비용 발생
3. `closed` 상태 레이블 불일치 — 정책 결정 후 코드베이스 전체 통일 필요
