## 3. 현재 구현 상태

### 3-1. 팀 테이블 구조 (현재)

```sql
CREATE TABLE teams (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  team_name text NOT NULL,
  contact text,
  region text NULLABLE,
  bio text NULLABLE,
  created_by uuid NOT NULL REFERENCES auth.users(id) ON DELETE RESTRICT,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  is_dummy boolean NOT NULL DEFAULT false,

  -- 구 모델 흔적 (현재 미사용)
  tournament_id uuid NULLABLE,
  captain_user_id uuid NULLABLE,
  status text NOT NULL DEFAULT 'pending'
);
```

**중요 관찰:**
- 팀은 tournament-independent (독립 실체). 하나의 팀이 여러 대회에 신청 가능
- `tournament_id`, `captain_user_id`, `status`는 구 설계의 흔적으로 현재 사용 안 함
- `created_by`는 생성자지만, 팀 captain 관리는 `team_members` 테이블로 분리됨
- `is_dummy`: organizer가 더미 팀을 만들어 대회 채울 때 사용 (실제 신청 없이 자동 confirmed)

### 3-2. team_members 테이블 구조 (현재)

```sql
CREATE TABLE team_members (
  team_id uuid NOT NULL REFERENCES teams(id) ON DELETE CASCADE,
  user_id uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  role_in_team text NOT NULL CHECK (role_in_team IN ('captain', 'player')),
  created_at timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (team_id, user_id)
);
```

**중요 관찰:**
- `PRIMARY KEY (team_id, user_id)` → 한 유저는 한 팀에 1번만 소속 가능
- 여러 팀에 소속은 가능 (팀 A captain + 팀 B player 동시 가능)
- `role_in_team = 'captain'` 이 팀 관리 권한의 핵심 (대회 신청, 합류 승인 등)
- 한 팀에 captain이 여러 명 가능한지는 DB 레벨에서 제약 없음

### 3-3. team_join_applications 테이블 구조 (현재)

```sql
CREATE TABLE team_join_applications (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  team_id uuid NOT NULL REFERENCES teams(id) ON DELETE CASCADE,
  applicant_id uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  status text NOT NULL DEFAULT 'pending' CHECK (status IN ('pending', 'approved', 'rejected')),
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE (team_id, applicant_id)
);
```

**중요 관찰:**
- `UNIQUE(team_id, applicant_id)`: 같은 팀에 동시에 최대 1개 신청만 가능
- `rejected` 후 재신청: RPC `apply_for_team`에서 status를 다시 `pending`으로 update
- 승인 시 `team_members`에 player로 추가

### 3-4. tournament_team_applications 테이블 구조 (현재)

```sql
CREATE TABLE tournament_team_applications (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tournament_id uuid NOT NULL REFERENCES tournaments(id) ON DELETE CASCADE,
  team_id uuid NOT NULL REFERENCES teams(id) ON DELETE CASCADE,
  division_id uuid NOT NULL REFERENCES divisions(id) ON DELETE RESTRICT,
  applied_by uuid NOT NULL REFERENCES auth.users(id) ON DELETE RESTRICT,
  status text NOT NULL CHECK (status IN (
    'payment_pending', 'paid_pending_approval', 'confirmed',
    'waitlisted', 'expired', 'cancelled'
  )),

  -- 비용
  base_entry_fee integer NOT NULL DEFAULT 0,
  discount_amount integer NOT NULL DEFAULT 0,
  final_amount integer NOT NULL DEFAULT 0,

  -- 대기열
  waitlist_position integer,

  -- 입금
  payment_due_at timestamptz,
  depositor_name text,
  depositor_note text,
  paid_marked_at timestamptz,

  -- 운영자
  approved_by uuid REFERENCES auth.users(id) ON DELETE SET NULL,
  approved_at timestamptz,
  admin_memo text,

  -- 타임스탬프
  submitted_at timestamptz NOT NULL DEFAULT now(),
  confirmed_at timestamptz,
  cancelled_at timestamptz,
  expired_at timestamptz
);

-- 핵심 인덱스 (활성 신청 중복 방지)
CREATE UNIQUE INDEX applications_active_unique
  ON tournament_team_applications (team_id, division_id)
  WHERE status IN ('payment_pending', 'paid_pending_approval', 'confirmed', 'waitlisted');
```

**중요 관찰:**
- `(tournament_id, team_id, division_id)` PK 없음 → 같은 팀이 취소 후 재신청 가능
- `PARTIAL UNIQUE INDEX`: 활성 상태(`ACTIVE_STATUSES`)에 한해서만 중복 방지
- `cancelled`, `expired` 상태는 제약 대상 외 → 재신청 가능
- **로스터 관련 테이블 없음**: 현재 로스터 확정 기능 미구현 상태

### 3-5. 팀 생성 현재 구현

**경로**: `app/(app)/teams/new/page.tsx` + `Form.tsx`

**흐름:**
1. 사용자가 `player` role이어야 접근 가능 (page.tsx 서버에서 redirect 처리)
2. 이미 captain인 팀 있으면 `/dashboard`로 redirect (UI 레벨 제약)
3. 폼 제출 → `createTeamAction()` Server Action → `createTeam(name, region, bio, contact)` API → RPC `create_team_with_captain`
4. RPC 내부: `teams` INSERT + `team_members` INSERT (`role_in_team = 'captain'`)

**현재 제약:**
- 팀 생성 = captain이 없으면 생성 가능 (UI 레벨 체크)
- DB 레벨에서 captain 여러 팀 생성 방지 인덱스/제약 없음 → **DB 레벨 제약 없음**

### 3-6. 팀 합류 현재 구현

**경로**: `app/(app)/teams/find/page.tsx`

**흐름:**
1. 사용자가 `player` role이어야 접근
2. RPC `get_teams_for_join` 호출: 자신이 미소속인 팀 목록 (더미팀 제외)
3. 팀 선택 → `applyForTeam(teamId)` → RPC `apply_for_team`
4. RPC 내부 검증:
   - 이미 `team_members`에 있으면 에러
   - 이미 `pending` 신청 있으면 에러
   - 이미 `approved` 신청 있으면 에러 (이미 멤버여야 하지만 방어)
   - `rejected` 신청 있으면 `pending`으로 UPDATE (재신청)
   - 신규면 INSERT

**현재 제약:**
- 합류 개수 제한 없음 (여러 팀에 순차적으로 신청 가능)
- 한 팀에 동시 중복 신청 방지 (UNIQUE 제약)
- 이미 팀 member인 경우에도 해당 팀에 재신청 불가

### 3-7. 대회 신청 현재 구현

**경로**: `app/(app)/tournament/[id]/apply/page.tsx`

**흐름:**
1. 사용자가 captain인 팀 목록 로드 (`listMyManagedTeams`)
2. 팀 + division 선택 폼
3. 제출 → `applyTeamToTournament()` Server Action → `applyToTournament(tournamentId, teamId, divisionId)` → RPC `submit_tournament_application`
4. RPC 내부:
   - `tournament.status = 'open'` 확인
   - 호출자가 해당 팀의 captain 확인
   - division 신청 기간 확인
   - FOR UPDATE로 division 행 잠금
   - 정원(`capacity`) 확인 → `payment_pending` 또는 `waitlisted`

**현재 제약:**
- 같은 팀이 같은 division에 활성 신청 1개 (PARTIAL UNIQUE INDEX)
- 같은 팀이 다른 division에는 신청 가능 (동일 대회 내 다른 부문)
- **같은 유저가 여러 팀 captain이라면 각각 대회 신청 가능** → 현재 로스터 중복 방지 로직 없음

### 3-8. 로스터 현재 구현

**현재 없음**: `rosters` 테이블이나 roster 관련 마이그레이션 파일이 존재하지 않음. 로스터 확정 기능은 미구현 상태.

## 4. 재사용 가능한 패턴

### 4-1. Server Action / API 패턴

**에러 처리:**
```typescript
// lib/api/*.ts 공통 패턴
if (!user) return { data: null, error: "로그인이 필요합니다." };
if (error) return { data: null, error: error.message };

// RPC 결과 파싱
const result = data as { ok: boolean; error?: string };
if (!result.ok) return { ok: false, error: result.error ?? "알 수 없는 오류" };
return { ok: true };
```

**Server Action 패턴:**
```typescript
// app/**/actions.ts 공통 구조
"use server";
export async function someAction(input: SomeInput): Promise<ActionResult> {
  const supabase = await createClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) return { ok: false, error: "로그인이 필요합니다." };
  // ...
}
```

### 5-4. DB / 마이그레이션

**현재 없는 테이블 (구현 필요):**

```sql
-- rosters: 팀의 대회별 로스터
CREATE TABLE rosters (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  application_id uuid NOT NULL REFERENCES tournament_team_applications(id) ON DELETE CASCADE,
  team_id uuid NOT NULL REFERENCES teams(id),
  tournament_id uuid NOT NULL REFERENCES tournaments(id),
  status text NOT NULL DEFAULT 'draft' CHECK (status IN ('draft', 'submitted', 'confirmed', 'rejected')),
  submitted_at timestamptz,
  confirmed_at timestamptz,
  created_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE(application_id)  -- 신청 1건당 로스터 1개
);

-- roster_members: 로스터에 포함된 선수
CREATE TABLE roster_members (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  roster_id uuid NOT NULL REFERENCES rosters(id) ON DELETE CASCADE,
  user_id uuid NOT NULL REFERENCES auth.users(id),
  created_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE(roster_id, user_id)  -- 같은 로스터 내 중복 선수 방지
);
```

**중복 출전 방지를 위한 핵심 제약 (로스터 확정 시점):**

```sql
-- 같은 대회에 이미 다른 confirmed 로스터에 포함된 선수 등록 방지
-- → RPC 레벨에서 구현 (PARTIAL UNIQUE INDEX로 표현 어려움)
-- RPC confirm_roster 내부에서:
SELECT rm.user_id
FROM roster_members rm
JOIN rosters r ON r.id = rm.roster_id
JOIN tournament_team_applications tta ON tta.id = r.application_id
WHERE tta.tournament_id = v_tournament_id
  AND r.status = 'confirmed'
  AND rm.user_id = ANY(p_member_ids)
  AND r.team_id != p_team_id;
-- → 결과가 있으면 중복 출전 에러
```

**수정 필요한 기존 마이그레이션:**
- `teams` 테이블에 팀 생성 정책 변경 시 인덱스 추가 고려

```sql
-- roster_members
-- SELECT: team_member OR organizer
-- INSERT: team captain (draft/submitted 상태 roster에만)
-- DELETE: team captain (draft/submitted 상태 roster에만)
```

**기존 정책 영향:**
- `tournament_team_applications` RLS는 현재 `team_members` 기준이므로 roster 연계 시 동일 패턴 재사용 가능

## 7. 구현 시 주의사항

### 7-1. `UserTeamStatus` 타입의 한계

현재 `UserTeamStatus`는:
```typescript
type UserTeamStatus =
  | "no_team"       // 팀 없음
  | "join_pending"  // 합류 신청 중
  | "team_member"   // 팀원
  | "captain";      // 주장
```
이 타입은 사용자가 **하나의 팀에만 속한다는 가정**으로 설계됨. 다중 팀 소속 정책에서는 이 구조가 부적합하다.

**수정 방향**: 단일 상태값이 아닌 팀 목록 기반 상태 (각 팀별 role 반환)로 전환 필요.

### 7-2. captain 여러 팀 생성: DB vs UI 제약

현재 `teams` 생성 제약:
- UI 레벨: `listMyManagedTeams().length > 0`이면 팀 생성 불가
- DB 레벨: **제약 없음**

`create_team_with_captain` RPC 내부에도 captain 중복 체크 없음. 요구사항에서 "팀 생성 개수 제한 없음"이므로 **현재 구조 유지 가능**. UI 레벨 제약만 제거하면 됨.

### 7-3. 대회 신청 시 동일 대회 중복 참가 사전 검증

현재 `submit_tournament_application` RPC는:
- 같은 팀이 같은 division에 중복 신청 방지 (PARTIAL UNIQUE INDEX)
- **같은 유저가 다른 팀으로 같은 대회에 신청하는 것은 방지하지 않음**

이 검증은 로스터 확정 시점에서 하는 것이 정책 기준이나, 사전에 UI/API 레벨에서 경고를 보여줄 수 있음. 단, 강제 차단은 로스터 확정 RPC에서만 수행.

### 7-4. 로스터 확정 중복 검사 로직

`confirm_roster` RPC 내부에서:
```sql
-- 같은 tournament에서 이미 confirmed roster에 있는 user_id 검사
SELECT rm.user_id
FROM roster_members rm
JOIN rosters r ON r.id = rm.roster_id
JOIN tournament_team_applications tta ON tta.id = r.application_id
WHERE tta.tournament_id = v_tournament_id
  AND r.status = 'confirmed'
  AND rm.user_id = ANY(p_member_ids)
  AND r.team_id != v_team_id;
```
이 쿼리 결과가 있으면 중복 출전 에러. **성능 주의**: roster_members 규모에 따라 인덱스 필요 (`tournament_id`, `status` 복합 인덱스).

### 7-5. 기존 `is_team_manager_for_team` 함수명 주의

헬퍼 함수명이 `is_team_manager_for_team`이지만, 내부는 `role_in_team = 'captain'` 체크. 네이밍 혼동 주의. RLS 정책에서 직접 사용 중이므로 함수명 변경은 영향 범위가 큼 → 현재 이름 유지 권장.

### 7-6. division vs tournament 단위 중복 방지

현재 `ACTIVE_STATUSES` UNIQUE INDEX는 `(team_id, division_id)` 단위:
- 같은 팀이 동일 대회의 **다른 division**에는 신청 가능
- 이것이 정책에서 허용하는 범위인지 확인 필요 (예: 남성부 + 여성부 동시 참가)

로스터 중복 방지는 `tournament_id` 단위로 걸 예정이므로, 신청 단계와 로스터 확정 단계의 범위가 다르다. 이 차이를 UI에서 명확히 안내해야 함.

### 7-7. `promote_next_waitlisted` 연쇄 호출 주의

취소/만료 시 자동으로 대기열 승격. 로스터 확정 취소 기능 추가 시 이 흐름과 충돌 여부 검토 필요 (로스터 확정 ≠ 대회 신청 확정이므로 별도 상태 관리 필요).

## 8. 권장 구현 순서

1. **정책 확정** (현재 리서치 기반)
   - 팀 생성 개수 제한 없음 → UI 제약 제거
   - 여러 팀 합류 가능 → 현재 구조 유지
   - 활성 팀 없음 → 현재 구조 유지
   - 동일 대회 중복 참가 방지 = 로스터 확정 시점 기준

2. **DB 마이그레이션** (우선순위 높음)
   - `rosters` 테이블 생성
   - `roster_members` 테이블 생성
   - RLS 정책 추가

3. **RPC 구현**
   - `add_roster_member(roster_id, user_id)`
   - `remove_roster_member(roster_id, user_id)`
   - `submit_roster(roster_id)` — 로스터 제출 (draft → submitted)
   - `confirm_roster(roster_id)` — 로스터 확정 (중복 출전 방지 검사 포함)

4. **API 레이어**
   - `lib/api/rosters.ts` 작성

5. **타입 정의**
   - `lib/types/roster.ts` 작성
   - `UserTeamStatus` 재정의 (다중 팀 소속 반영)

6. **프론트엔드**
   - 팀 생성 페이지: captain 제약 제거
   - 팀 로스터 관리 페이지 신규
   - 대회 신청 폼: 동일 대회 다른 팀 신청 경고 UI 추가

## 9. 결론

### 현재 구조가 지원하는 것 (변경 불필요)

| 항목 | 상태 |
|------|------|
| 팀이 tournament-independent 실체 | ✅ 이미 지원 |
| 한 유저가 여러 팀에 소속 | ✅ `team_members`에서 지원 |
| 동일 팀의 동일 division 중복 신청 방지 | ✅ PARTIAL UNIQUE INDEX |
| 정원 관리 + 대기열 | ✅ `capacity` + `waitlisted` 구현됨 |
| 동시성 제어 (대회 신청) | ✅ FOR UPDATE |
| captain 권한 기반 신청 | ✅ RPC 내 captain 확인 |

### 현재 구조에서 변경/추가가 필요한 것

| 항목 | 현황 | 필요 작업 |
|------|------|-----------|
| 팀 생성 개수 제한 | UI 레벨 제약 있음 (단, 제한 없음이 정책) | UI 제약 제거 |
| 여러 팀 captain 가능 | DB 레벨 제약 없음 (정책과 일치) | 없음 |
| 활성 팀 개념 | 없음 (정책과 일치) | 없음 |
| `UserTeamStatus` 타입 | 단일 팀 가정 | 다중 팀 기반으로 재설계 |
| 동일 대회 중복 출전 방지 | **미구현** | 로스터 테이블 + RPC 신규 구현 |
| 로스터 관리 | **미구현** | DB + RPC + API + UI 전체 구현 |
| 대회 신청 시 사전 경고 | **미구현** | API 검증 추가 + UI 안내 |

**핵심 결론:** 팀 생성/합류 정책 자체는 현재 DB 구조와 대체로 부합한다. 가장 큰 미구현 영역은 **로스터 관리 기능 전체**이며, 이것이 동일 대회 중복 출전 방지의 최종 기준점이 된다. 이를 구현하기 위해 `rosters` + `roster_members` 테이블 신규 설계와 `confirm_roster` RPC에서의 중복 검사 로직이 핵심 구현 과제다.
