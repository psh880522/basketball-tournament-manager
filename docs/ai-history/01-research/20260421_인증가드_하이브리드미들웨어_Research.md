# 인증 가드 하이브리드 미들웨어 리서치

> **기능 개요**: 현재 각 `page.tsx`에 분산된 세션 인증 가드를 `middleware.ts`로 1차 집중화하고,
> 역할 기반 세부 가드는 `page.tsx`에 유지하는 하이브리드 방식으로 개선한다.

---

## 1. 기술 스택 요약

| 항목 | 버전 / 패키지 |
|------|--------------|
| Framework | Next.js 15 (App Router) |
| React | 19.x (Server Components) |
| Auth | `@supabase/ssr` 0.8.0 + `@supabase/supabase-js` 2.95.3 |
| 세션 관리 | 쿠키 기반 (Supabase SSR 패턴) |
| TypeScript | 5.x |

**핵심 아키텍처 특징**
- 모든 페이지가 **Server Component** 기반이며 `getUserWithRole()`을 직접 호출
- `createSupabaseServerClient`가 `React cache()`로 감싸져 있어 **요청당 1회만 Supabase 연결**
- 현재 `middleware.ts` 파일이 **존재하지 않음** (프로젝트 루트 및 `src/` 모두 없음)

## 3. 현재 구현 상태

### 3-1. `getUserWithRole()` 동작 흐름

```
supabase.auth.getUser()
  ├─ error (세션 없음)  → status: "unauthenticated"
  ├─ error (기타)       → status: "error"
  └─ user 있음
       └─ profiles 테이블 SELECT role WHERE id = user.id
            ├─ error          → status: "error"
            ├─ 행 없음        → status: "empty"
            └─ 정상           → status: "ready", role: Role
```

### 3-2. 현재 각 page.tsx의 인증 패턴 (보일러플레이트)

모든 보호 페이지에서 아래 구조가 반복된다:

```typescript
export const dynamic = "force-dynamic";
export const revalidate = 0;

export default async function SomePage() {
  const result = await getUserWithRole();

  // ① 세션 없음 → 로그인 페이지
  if (result.status === "unauthenticated") redirect("/login");

  // ② DB 에러 → 에러 UI 렌더링
  if (result.status === "error") return <ErrorUI />;

  // ③ 프로필 없음 → 로그인 페이지 (또는 온보딩)
  if (result.status === "empty") redirect("/login");

  // ④ 역할 검증 (페이지별 상이)
  if (!isOperationRole(result.role)) redirect("/dashboard");

  return <PageContent />;
}
```

**문제점**: ①③은 **세션 유무만 보는 공통 로직**인데도 23개 이상의 page.tsx에 중복 작성되어 있다. 새 페이지 추가 시 빠뜨릴 위험 있음.

### 3-3. `(app)/layout.tsx`의 역할

레이아웃에서도 `getUserWithRole()`을 호출하지만 **리다이렉트하지 않는다**. Sidebar vs GlobalHeader 선택에만 사용한다.

- `React cache()` 덕분에 layout → page 간 `getUserWithRole()` 호출이 중복 DB 조회를 유발하지 않음
- 단, **미들웨어**에서도 Supabase 세션을 확인하려면 별도 클라이언트 생성 필요 (아래 참조)

## 4. 재사용 가능한 패턴

### 4-1. `requireXxx()` 패턴 (guards.ts)

서버 액션에서 이미 사용 중인 패턴. 리다이렉트 없이 `{ ok, error }` 반환.

```typescript
// src/lib/auth/guards.ts
export async function requireOrganizer(): Promise<ActionResult>
export async function requireOperationRole(): Promise<ActionResult>
export async function requirePlayer(): Promise<ActionResult>
```

**미들웨어 구현과의 차이**: guards.ts 함수는 `createSupabaseServerClient`(`cookies()` 기반)를 사용하므로 미들웨어에서 그대로 쓸 수 없다. 미들웨어는 `@supabase/ssr`의 `createServerClient`를 `request/response` 기반으로 새로 생성해야 한다.

### 4-2. 역할 판별 헬퍼 (roles.ts)

```typescript
isOperationRole(role)   // organizer | manager
isOrganizerRole(role)   // organizer only
isPlayerRole(role)      // player
isUserRole(role)        // user (미등록)
```

미들웨어에서는 세션 유무만 판단하므로 이 헬퍼는 미들웨어에서 불필요. page.tsx에서 계속 사용.

## 5. 영향 범위

### 5-1. 프론트엔드

**신규 파일**: `middleware.ts` (프로젝트 루트)

**수정 파일 (선택)**: 미들웨어에서 1차 세션 가드를 처리하면 각 page.tsx의 ①③ 구문을 제거 가능.

```typescript
// 현재 각 page.tsx에서 제거 가능해지는 구문 (미들웨어 적용 후)
if (result.status === "unauthenticated") redirect("/login");
if (result.status === "empty") redirect("/login");
```

단, 역할 가드(④)는 그대로 page.tsx에 유지해야 한다.

영향받는 페이지 수: **21개 이상**

### 5-2. 서버 액션 / API

- `guards.ts`의 `requireXxx()` 함수는 변경 없음
- `app/actions/auth.ts`의 `logoutAction()`은 변경 없음

### 5-3. 타입

```typescript
// src/lib/auth/roles.ts — 변경 없음
export type Role = "organizer" | "manager" | "user" | "player";
type UserRoleStatus = "unauthenticated" | "error" | "empty" | "ready";
```

미들웨어용 Supabase 클라이언트 타입: `@supabase/ssr`의 `createServerClient` 반환 타입 그대로 사용.

### 5-4. DB / 마이그레이션

**없음.** 미들웨어는 기존 세션 쿠키와 `profiles` 테이블을 그대로 사용한다.

단, 미들웨어에서 `profiles` 조회를 하지 않는다면 세션 토큰 유효성만 확인하는 수준이므로 별도 마이그레이션 불필요.

### 5-5. 권한 정책 (RLS)

미들웨어는 Supabase Anon Key로 실행되며 `auth.getUser()`만 사용할 것이므로 기존 RLS 정책 변경 없음.

| 파일 | 설명 |

| `middleware.ts` (프로젝트 루트) | Next.js 미들웨어. 보호 경로 세션 유무 1차 검증, 미인증 시 `/login` 리다이렉트 |

## 7. 구현 시 주의사항

### 7-1. 미들웨어용 Supabase 클라이언트는 별도로 만들어야 한다

기존 `createSupabaseServerClient`는 `next/headers`의 `cookies()`를 사용하는데, 미들웨어 컨텍스트에서는 이를 사용할 수 없다. 미들웨어에서는 `NextRequest/NextResponse` 기반으로 쿠키를 읽고 써야 한다.

```typescript
// middleware.ts 내부 패턴
import { createServerClient } from "@supabase/ssr";
import { NextRequest, NextResponse } from "next/server";

function createMiddlewareSupabaseClient(request: NextRequest, response: NextResponse) {
  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll: () => request.cookies.getAll(),
        setAll: (cookiesToSet) => {
          cookiesToSet.forEach(({ name, value, options }) => {
            request.cookies.set(name, value);
            response.cookies.set(name, value, options);
          });
        },
      },
    }
  );
}
```

**중요**: Supabase SSR 문서에 따르면 미들웨어에서 쿠키를 갱신(refresh)하는 역할도 맡아야 한다. 세션 토큰이 만료되면 미들웨어에서 갱신하고 새 쿠키를 response에 담아야 후속 서버 컴포넌트가 정상 세션을 받는다.

### 7-2. `matcher`를 명확히 설정해야 한다

미들웨어가 모든 요청에 실행되면 성능 저하. 보호가 필요한 경로만 명시해야 한다.

보호 대상:
- `/admin/*`
- `/dashboard`
- `/teams/*`
- `/my-applications/*`
- `/onboarding/*`
- `/team/*`
- `/tournament/*/apply`

공개 유지 (미들웨어 제외):
- `/` (랜딩)
- `/tournaments/*` (대회 목록/상세 공개)
- `/tournament/*/result` (결과 공개)
- `/(auth)/*` → 단, 이미 로그인된 사용자 접근 처리 필요
- `/_next/*`, `/api/*`, `/favicon.ico`

### 7-3. `(auth)` 경로 처리

미인증 상태에서의 `/login`, `/signup`은 통과. 하지만 **이미 로그인된 사용자**가 `/login`에 접근하면 리다이렉트해야 한다. 현재 이 로직이 `login/page.tsx`에 있는데 미들웨어로 올릴지 여부를 결정해야 한다.

```typescript
// 현재 app/(auth)/login/page.tsx 상단
const result = await getUserWithRole();
if (result.status === "ready") {
  redirect(isOperationRole(result.role) ? "/admin" : "/dashboard");
}
```

미들웨어에서 세션 있는 사용자의 `/login` 접근을 처리하면 page.tsx의 이 로직을 제거할 수 있으나, 역할별 분기(`/admin` vs `/dashboard`)는 미들웨어에서 판단하기 어렵다(DB profiles 조회 필요). **미들웨어에서는 세션 유무만 판단하고 `/dashboard`로 보내는 것**이 현실적이다.

### 7-4. `React cache()`의 무효화 범위

`createSupabaseServerClient`는 `React cache()`로 감싸여 있어 **같은 요청 내**에서 중복 Supabase 클라이언트 생성을 방지한다. 미들웨어는 다른 요청 컨텍스트에서 실행되므로, 미들웨어에서 별도 Supabase 클라이언트를 만들어도 Server Component의 `cache()`와 충돌하지 않는다.

### 7-5. `profiles` DB 조회 없이 세션만 확인

미들웨어에서 `auth.getUser()`만 호출하면 충분하다 — 인증 세션 유무만 판별. `profiles.role` 조회는 page.tsx에서 담당. 이 방식이 미들웨어 지연을 최소화한다.

### 7-6. `force-dynamic` / `revalidate = 0` 유지 여부

현재 모든 보호 페이지에 `export const dynamic = "force-dynamic"` 설정되어 있다. 미들웨어와 무관하게 이 설정은 데이터 실시간성을 위해 유지한다.

## 8. 권장 구현 순서

1. **`middleware.ts` 신규 생성**
   - Supabase 미들웨어 클라이언트 생성 (`createServerClient` with request/response cookie 패턴)
   - `auth.getUser()` 호출 → 세션 없으면 `/login?next=경로` 리다이렉트
   - 세션 쿠키 갱신 처리 (`supabase.auth.getUser()` 호출이 자동 갱신 트리거)
   - `config.matcher`로 보호 경로 열거

2. **`(auth)` 경로 처리 방침 결정**
   - 로그인한 사용자의 `/login`, `/signup` 접근: 미들웨어에서 `/dashboard`로 단순 리다이렉트하거나 page.tsx 유지 중 선택

3. **page.tsx 정리 (선택, 단계적)**
   - 미들웨어 적용 후 각 page.tsx에서 `"unauthenticated"`/`"empty"` 분기 제거
   - 역할 가드(`isOperationRole`, `isPlayerRole` 등)는 그대로 유지
   - 팀 멤버십 가드(`getMyRoleInTeam`)는 반드시 page.tsx에 유지 (DB 쿼리 필요)

4. **통합 테스트**
   - 미인증 상태로 `/admin`, `/dashboard`, `/teams/X`, `/my-applications` 직접 URL 접근 확인
   - 세션 만료 후 자동 갱신 및 리다이렉트 확인
   - 로그인 후 인증 필요 경로 접근 정상 작동 확인

## 9. 결론

### 현재 상태 요약

- `middleware.ts` 파일 **없음**
- 세션 유무 검증이 21개 이상의 page.tsx에 중복
- `getUserWithRole()`에 세션 확인 + 역할 조회가 묶여 있음
- `(app)/layout.tsx`에서도 `getUserWithRole()` 호출하지만 리다이렉트 없음

### 하이브리드 방식 설계

```
HTTP 요청
    │
    ▼
middleware.ts (Edge Runtime)
    ├─ matcher에 포함된 경로만 실행
    ├─ Supabase auth.getUser() → 세션 유무만 확인
    ├─ 세션 없음 → redirect("/login?next=현재경로")
    └─ 세션 있음 → NextResponse.next() (쿠키 갱신 포함)
         │
         ▼
    (app)/layout.tsx (Server Component)
         │  getUserWithRole() — React cache() 덕분에 미들웨어와 별개
         └─ Sidebar / GlobalHeader 선택
              │
              ▼
         page.tsx (Server Component)
              ├─ getUserWithRole() — cache() 덕분에 layout과 동일 인스턴스 재사용
              ├─ 역할 가드: isOperationRole, isPlayerRole 등
              └─ 팀 멤버십 가드: getMyRoleInTeam() (DB 쿼리 필요)
```

### 이 방식의 이점

| 구분 | 현재 | 미들웨어 도입 후 |
