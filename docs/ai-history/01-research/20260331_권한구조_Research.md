## 3. 프로필 테이블 및 기본 역할

**파일:** `supabase/migrations/0002_profiles_roles.sql` (L18-L22)

```sql
create table if not exists public.profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  role public.app_role not null default 'spectator',
  created_at timestamptz not null default now()
);
```

**핵심:** `default 'spectator'` — 현재 모든 신규 가입자는 `spectator`로 생성된다.
**요구사항과 불일치:** 신규 요구사항은 기본 역할을 `player`로 고정해야 한다.

### 3.1 회원가입 트리거

**파일:** `supabase/migrations/0002_profiles_roles.sql` (L26-L43)

```sql
create or replace function public.handle_new_user()
returns trigger
language plpgsql
security definer
set search_path = public
as $$
begin
  insert into public.profiles (id)
  values (new.id)
  on conflict (id) do nothing;
  return new;
end;
$$;

create trigger on_auth_user_created
after insert on auth.users
for each row execute function public.handle_new_user();
```

트리거는 `id`만 삽입하고 `role`은 명시하지 않는다 → DB 컬럼 default(`spectator`)가 적용된다.
**수정 위치:** `default 'spectator'` → `default 'player'`로 변경하면 트리거 코드는 수정 불필요.

### 3.2 회원가입 서버 액션

**파일:** `app/signup/actions.ts`
역할 지정 코드 없음. `supabase.auth.signUp({ email, password })`만 호출하며, 역할은 전적으로 DB trigger + default에 의존한다.

## 5. RLS Helper 함수

### 5.1 is_organizer()

**파일:** `supabase/migrations/0002_profiles_roles.sql` (L46-L61)

```sql
create or replace function public.is_organizer()
returns boolean ...
begin
  return exists (
    select 1 from public.profiles p
    where p.id = auth.uid() and p.role = 'organizer'
  );
end;
```

`profiles.role = 'organizer'`를 직접 비교한다. 신규 구조에서도 `organizer`는 유지되므로 변경 불필요.

### 5.2 is_team_manager() — **제거 대상**

**파일:** `supabase/migrations/0020_team_apply.sql` (L49-L64)

```sql
create or replace function public.is_team_manager()
returns boolean ...
begin
  return exists (
    select 1 from public.profiles p
    where p.id = auth.uid() and p.role = 'team_manager'
  );
end;
```

`profiles.role = 'team_manager'`를 참조한다.
**영향:** `teams` 테이블의 SELECT, INSERT RLS 정책에서 사용.
**후속 마이그레이션(0088)에서 이미 대체 진행 중:** `is_team_manager_for_team(team_uuid)`으로 팀 멤버 테이블 기반으로 전환됨.

### 5.3 is_team_manager_for_team(team_uuid)

**파일:** `supabase/migrations/0088_team_members_applications.sql` (L76-L92)

```sql
create or replace function public.is_team_manager_for_team(team_uuid uuid)
returns boolean ...
begin
  return exists (
    select 1 from public.team_members tm
    where tm.team_id = team_uuid
      and tm.user_id = auth.uid()
      and tm.role_in_team = 'manager'
  );
end;
```

`profiles.role`과 무관하게 `team_members.role_in_team = 'manager'`를 기반으로 판단한다.
신규 권한 체계에서는 이 함수가 팀 내 관리 역할의 기준이 된다.

### 5.4 is_team_member_for_team(team_uuid)

**파일:** `supabase/migrations/0088_team_members_applications.sql` (L59-L74)

```sql
create or replace function public.is_team_member_for_team(team_uuid uuid)
returns boolean ...
begin
  return exists (
    select 1 from public.team_members tm
    where tm.team_id = team_uuid and tm.user_id = auth.uid()
  );
end;
```

### 5.5 can_manage_team(team_id)

**파일:** `supabase/migrations/0030_player_crud.sql` (L40-L55)

```sql
create or replace function public.can_manage_team(team_id uuid)
returns boolean ...
begin
  return exists (
    select 1 from public.teams t
    where t.id = team_id and t.captain_user_id = auth.uid()
  );
end;
$$;
```

`profiles.role`과 무관하게 `teams.captain_user_id` 기반.
`is_team_manager()`와 AND로 결합된 구 정책에서 사용됨. `0091_players_rls_team_members.sql`에서 `is_team_manager_for_team`으로 대체됨.

## 6. Supabase RLS 정책 전수 분석

### 6.1 profiles 테이블

| 정책명 | 동작 | 조건 |
|---|---|---|
| `profiles_select_own_or_organizer` | SELECT | `id = auth.uid() OR is_organizer()` |
| `profiles_update_own` | UPDATE | `id = auth.uid()` (USING+WITH CHECK) |

**문제점:** `profiles` UPDATE 정책에 역할 변경을 제한하는 로직 없음. 즉, 인증된 사용자가 자신의 `role` 컬럼을 직접 수정할 수 있는 보안 취약점이 존재한다. 신규 요구사항("organizer만 role 변경 가능")을 반영하려면 별도 정책 또는 트리거로 막아야 한다.

### 6.2 tournaments 테이블

| 정책명 | 파일 | 동작 | 조건 |
|---|---|---|---|
| `tournaments_select_public` | `0010_tournaments.sql` | SELECT | `status IN ('open','closed')` |
| `tournaments_select_organizer` | `0085_tournaments_select_organizer_all.sql` | SELECT | `is_organizer()` (본인 생성 포함 전체) |
| `tournaments_insert_organizer` | `0010_tournaments.sql` | INSERT | `is_organizer()` |

### 6.3 teams 테이블

| 정책명 | 파일 | 동작 | 조건 |
|---|---|---|---|
| `teams_select_organizer` | `0020_team_apply.sql` | SELECT | `is_organizer()` |
| `teams_select_team_manager` | `0020_team_apply.sql` | SELECT | `is_team_manager() AND captain_user_id = auth.uid()` — **team_manager role 의존** |
| `teams_insert_team_manager` | `0020_team_apply.sql` | INSERT | `is_team_manager() AND captain_user_id = auth.uid()` — **team_manager role 의존** |
| `teams_select_members` | `0088_team_members_applications.sql` | SELECT | `is_team_member_for_team(id)` — team_members 테이블 기반 |
| `teams_insert_members` | `0088_team_members_applications.sql` | INSERT | `created_by = auth.uid()` |
| `teams_update_manager` | `0088_team_members_applications.sql` | UPDATE | `is_team_manager_for_team(id) OR is_organizer()` |
| `teams_delete_manager` | `0088_team_members_applications.sql` | DELETE | `is_team_manager_for_team(id) OR is_organizer()` |

**중복/충돌 주의:** `teams_select_team_manager`(구)와 `teams_select_members`(신)가 공존한다. 구 정책이 `is_team_manager()` (profiles.role) 기반이라 `team_manager` 역할이 제거되면 해당 정책은 항상 false가 되며 사실상 비활성화된다. 단, 정책 자체가 남아 있으면 쿼리 플래너에 영향을 줄 수 있으므로 삭제해야 한다.

### 6.4 players 테이블

최신 상태는 `0091_players_rls_team_members.sql`에 의해 정책이 교체됨:

| 정책명 | 동작 | 조건 |
|---|---|---|
| `players_select_team_member` | SELECT | `is_team_member_for_team(team_id)` |
| `players_insert_team_mgr` | INSERT | `is_team_manager_for_team(team_id)` |
| `players_update_team_mgr` | UPDATE | `is_team_manager_for_team(team_id)` |
| `players_delete_team_mgr` | DELETE | `is_team_manager_for_team(team_id)` |

이전 정책 `players_select_team_manager` / `players_insert_team_manager` 등(구 `is_team_manager()` + `can_manage_team()` 기반)은 DROP 처리됨. 신규 권한 체계와 이미 호환 가능한 상태.

### 6.5 team_members 테이블

| 정책명 | 동작 | 조건 |
|---|---|---|
| `team_members_select` | SELECT | `is_team_member_for_team(team_id) OR is_organizer()` |
| `team_members_insert` | INSERT | `is_organizer() OR is_team_manager_for_team(team_id) OR (팀 created_by = auth.uid())` |
| `team_members_update` | UPDATE | `is_team_manager_for_team(team_id) OR is_organizer()` |
| `team_members_delete` | DELETE | `is_team_manager_for_team(team_id) OR is_organizer()` |

### 6.6 tournament_team_applications 테이블

| 정책명 | 동작 | 조건 |
|---|---|---|
| `tournament_team_applications_select` | SELECT | `is_organizer() OR is_team_member_for_team(team_id)` |
| `tournament_team_applications_insert` | INSERT | `is_organizer() OR (is_team_manager_for_team(team_id) AND applied_by = auth.uid())` |
| `tournament_team_applications_update` | UPDATE | `is_organizer()` |

최신 `0112_applications_insert_organizer.sql`에서 organizer도 INSERT 가능하도록 개정됨.

### 6.7 matches / standings (결과 공개 정책)

**파일:** `supabase/migrations/0078_tournament_result_public_rls.sql`

- `matches_select_public_finished`: 대회 status가 `closed` 또는 `finished`이면 누구나 SELECT 가능
- `standings_select_public_closed`: 동일 조건으로 순위 공개

역할 무관하게 tournament.status 기반으로 공개 — 신규 권한 체계와 충돌 없음.

### 6.8 divisions 테이블

## 9. Tournament Guard — assertTournamentStepAllowed

**파일:** `lib/api/tournamentGuards.ts` (L43-L45)

```typescript

```

경기 생성, 코트 배정, 결과 제출, 순위 계산, 브래킷 생성, 라운드 진행 등 모든 단계 전이(step transition)가 이 가드를 통과해야 한다. 현재 `organizer` 전용.
**영향:** `manager`도 경기 단계 전이를 수행할 수 있어야 한다면 이 가드 완화 필요.

## 10. team_members.role_in_team 과 profiles.role의 관계

**파일:** `supabase/migrations/0088_team_members_applications.sql` (L32)

```sql
role_in_team text not null check (role_in_team in ('manager','player'))
```

이는 `app_role` enum과 **별개**의 개념이다:
- `profiles.role` = 사이트 전역 권한 (`organizer` / `team_manager` / `player` / `spectator`)
- `team_members.role_in_team` = 특정 팀 내 역할 (`manager` / `player`)

신규 권한 체계에서:
- 전역 `manager` (= 현재의 `organizer`에서 권한 위임 받은 스태프 역할)와
- 팀 내 `role_in_team = 'manager'` (= 팀 대표/관리자)는 서로 다른 개념이다.

혼동 방지를 위해 팀 내 역할은 `role_in_team = 'captain'` / `'member'` 등으로 rename 검토가 필요하다.

## 11. 사용자 역할 변경 UI/API 현황

현재 프로젝트에 **사용자 역할 변경 UI 또는 API가 존재하지 않는다.**
역할은 DB에서 직접 수정하거나, `profiles_update_own` RLS 정책에 의해 사용자가 자신의 profile을 수정하는 것으로만 변경 가능하다.

**보안 문제:** `profiles_update_own` 정책의 `with check`는 `id = auth.uid()`만 확인하므로 `role` 컬럼 변경을 막지 않는다. 즉 현재 인증된 사용자는 자신의 역할을 임의로 `organizer`로 변경할 수 있는 취약점이 있다.

신규 요구사항 구현을 위해 필요한 작업:
1. **DB:** `profiles_update_own` 정책에 `role` 컬럼 변경 방지 조건 추가 (e.g., `with check (id = auth.uid() AND role = OLD.role)`) 또는 별도 RPC 함수로 역할 변경 처리
2. **API:** organizer가 특정 사용자의 역할을 `player` ↔ `manager`로 변경하는 서버 액션/API 라우트 생성
3. **UI:** Admin 페이지에 사용자 목록 + 역할 변경 기능 추가

## 13. 기존 구현과 신규 요구사항의 Gap 분석

### Gap 1: 기본 역할 불일치

| 항목 | 현재 | 요구사항 |
|---|---|---|
| 신규 가입 기본 역할 | `spectator` | `player` |

**수정:** `supabase/migrations/0002_profiles_roles.sql`의 `default 'spectator'` → `default 'player'`
신규 마이그레이션 파일 작성 필요: `ALTER TABLE public.profiles ALTER COLUMN role SET DEFAULT 'player';`

### Gap 2: 존재하지 않는 manager 역할

| 항목 | 현재 | 요구사항 |
|---|---|---|
| 운영 스태프 역할 | 없음 (`organizer`가 모두 담당) | `manager` 별도 역할 필요 |

**수정:**
- `app_role` enum에 `'manager'` 값 추가 (PostgreSQL enum에 값 추가 불가 → 새 enum 생성 후 컬럼 변경 또는 `ALTER TYPE ... ADD VALUE`)
- `src/lib/auth/roles.ts`의 `Role` 타입에 `"manager"` 추가
- 관련 RLS helper 함수 `is_manager()` 추가

### Gap 3: team_manager 역할 처리

현재 `team_manager`가 수행하는 기능(팀 생성, 선수 관리, 참가 신청)을 신규 권한 체계에서는 `player`가 수행해야 한다.

**수정 방향:**
- 팀 생성/관리 기능의 역할 체크를 `team_manager` → `player` (또는 `player | manager | organizer`)로 변경
- `is_team_manager()` RLS 함수는 `is_team_manager_for_team()` (팀 멤버 테이블 기반)으로 완전히 대체
- `teams_select_team_manager`, `teams_insert_team_manager` RLS 정책 삭제 (이미 0088에서 대체 정책 추가됨)

### Gap 4: manager의 운영 기능 접근 범위 미정의

현재 admin 하위 모든 서버 액션은 `organizer`만 허용한다. `manager` 허용 범위를 기능별로 결정해야 한다.

**제안 분류:**

| 기능 | 현재 | 제안 |
|---|---|---|
| 대회 생성/수정/삭제 | organizer | organizer 전용 유지 |
| 대회 상태 변경 | organizer | organizer 전용 유지 |
| 디비전 관리 | organizer | organizer 전용 유지 |
| 참가 승인/반려 | organizer | organizer 전용 유지 |
| 코트 관리 | organizer | organizer + manager 허용 |
| 경기 결과 입력 | organizer | organizer + manager 허용 |
| 경기 상태 변경 | organizer | organizer + manager 허용 |
| 순위 재계산 | organizer | organizer + manager 허용 |
| 브래킷 생성 | organizer | organizer 전용 유지 |
| 라운드 진행 | organizer | organizer + manager 허용 |
| 사용자 역할 변경 | 없음 | organizer 전용 신규 구현 |

### Gap 5: 역할 변경 UI/API 부재

신규 요구사항의 핵심 기능 중 하나인 "organizer가 player ↔ manager 역할 변경"이 현재 완전히 없다.
**수정:** Admin 페이지에 사용자 관리 섹션 신규 구현 필요.

### Gap 6: profiles_update_own 보안 취약점

현재 인증된 사용자가 자신의 `role`을 임의로 변경 가능하다.
**수정:** `profiles_update_own` 정책의 `with check`에 role 변경 방지 로직 추가 또는 role 업데이트는 별도 `SECURITY DEFINER` RPC로만 허용.

## 15. 테스트 영향 범위

현재 프로젝트에 역할 관련 자동화 테스트 파일이 별도 존재하지 않음 (검색 결과 `*.test.ts`, `*.spec.ts` 파일 없음). 수동 테스트 시나리오로 다음을 검증해야 한다:

1. 신규 가입 → `player` 역할로 생성되는지
2. `player` → 팀 생성/수정/참가신청 가능한지
3. `player` → `/admin` 경로 접근 불가한지
4. `manager` → 결과 입력/코트 배정/경기 상태 변경 가능한지
5. `manager` → 대회 생성/삭제/디비전 설정 불가한지
6. `manager` → 다른 사용자 역할 변경 불가한지
7. `organizer` → 모든 기능 사용 가능한지
8. `organizer` → `player` ↔ `manager` 역할 변경 가능한지
9. RLS 정책: `player`가 다른 팀 데이터 직접 쿼리 불가한지
10. RLS 정책: `manager`가 `profiles.role`을 직접 수정 불가한지
