## 3. 현재 구현 상태

### 3-1. `divisions` 테이블 현재 컬럼

```sql
id               UUID PRIMARY KEY
tournament_id    UUID FK → tournaments(id) ON DELETE CASCADE
name             text NOT NULL
group_size       integer
tournament_size  integer
include_tournament_slots boolean NOT NULL DEFAULT false
sort_order       int NOT NULL DEFAULT 0
standings_dirty  boolean
```

**신규 기능에서 필요한 컬럼이 전혀 없음**: `entry_fee`, `capacity`, `application_open_at`, `application_close_at` 모두 없음.

### 3-2. `tournament_team_applications` 테이블 현재 컬럼

```sql
id            UUID PRIMARY KEY
tournament_id UUID FK → tournaments(id) ON DELETE CASCADE
team_id       UUID FK → teams(id) ON DELETE CASCADE
division_id   UUID FK → divisions(id) ON DELETE RESTRICT
status        text NOT NULL DEFAULT 'pending'
              CHECK (status IN ('pending', 'approved', 'rejected'))
applied_by    UUID FK → auth.users(id) ON DELETE RESTRICT
created_at    timestamptz NOT NULL DEFAULT now()

UNIQUE(tournament_id, team_id)   ← 현재: 팀당 대회 1개만 신청 가능
```

**문제점**:
- status가 3개뿐 ('pending', 'approved', 'rejected') → 6개로 확장 필요
- 참가비, 대기열, 입금 관련 컬럼 전무
- UNIQUE 제약이 `(tournament_id, team_id)` — division 단위 중복 방지로 변경 필요

### 3-3. 현재 신청 흐름 (구현된 것)

```
팀장 → /tournament/[id]/apply → 팀+division 선택 → applyTeamToTournament()
→ tournament_team_applications INSERT (status: 'pending')
→ 관리자 → /admin/tournaments/[id]/applications → 승인/거절 버튼
→ setApplicationStatus('approved' | 'rejected')
```

단순 3단계 플로우. 정원, 대기열, 입금, 알림 개념 없음.

### 3-4. 알림 시스템

**없음.** 현재 프로젝트에 알림 테이블도, 알림 발송 로직도 전혀 존재하지 않음.

## 4. 재사용 가능한 패턴

### 4-1. 공통 반환 타입 (`lib/types/api.ts`)

```typescript
// 데이터 조회용
export type ApiResult<T> = {
  data: T | null;
  error: string | null;
};

// 액션(뮤테이션)용
export type ActionResult =
  | { ok: true; message?: string }
  | { ok: false; error: string };
```

모든 API 함수와 서버 액션이 이 타입을 따름. 신규 함수도 동일 패턴 사용.

### 4-2. 서버 액션 권한 가드 패턴 (`src/lib/auth/guards.ts`)

```typescript
// 서버 액션 최상단에서 호출
export async function someAction(...): Promise<ActionResult> {
  const guard = await requireOrganizer(); // 또는 requirePlayer()
  if (!guard.ok) return guard;           // 권한 없으면 즉시 반환

  // 이하 비즈니스 로직
}
```

`requireOrganizer()`, `requireOperationRole()`, `requirePlayer()` 이미 구현됨.

### 4-3. 서버 컴포넌트 데이터 로드 패턴

```typescript
// 병렬 데이터 로드
const [teamsResult, appResult, divisionsResult] = await Promise.all([
  listMyManagedTeams(),
  getMyApplicationStatus(tournamentId),
  getDivisionsByTournament(tournamentId),
]);

// 에러 표시는 컴포넌트 레벨에서 인라인으로
if (teamsResult.error) return <p className="text-red-600">...</p>;
```

### 4-4. 클라이언트 폼 에러 처리 패턴

```typescript
const [error, setError] = useState<string | null>(null);
const [loading, setLoading] = useState(false);

async function handleSubmit() {
  setLoading(true);
  setError(null);
  const result = await someServerAction(...);
  if (!result.ok) {
    setError(result.error);
    setLoading(false);
    return;
  }
  // 성공 처리
}
```

### 4-5. Supabase 에러 코드 처리 패턴

```typescript
const { error } = await supabase.from("table").insert({...});
if (error) {
  if (error.code === "23505") {        // unique constraint 위반
    return { ok: false, error: "이미 신청한 팀입니다." };
  }
  if (error.code === "23503") {        // foreign key 위반
    return { ok: false, error: "..." };
  }
  return { ok: false, error: error.message };
}
```

### 4-6. `revalidatePath` 패턴 (서버 액션 후 캐시 무효화)

```typescript
import { revalidatePath } from "next/cache";

revalidatePath(`/admin/tournaments/${tournamentId}/applications`);
revalidatePath(`/admin/tournaments/${tournamentId}`);
```

### 4-7. Supabase SECURITY DEFINER RPC 패턴

기존 `create_team_with_manager()` RPC처럼, 복잡한 원자적 트랜잭션(정원 확인 + 신청 INSERT + waitlist 처리)은 Supabase RPC(PL/pgSQL) + `SECURITY DEFINER`로 구현하는 패턴이 이미 있음.

### 4-8. 기존 SQL 헬퍼 함수 (재사용 가능)

```sql
is_organizer()                     -- profiles.role = 'organizer' 확인
```

**마이그레이션 4: `0233_applications_submit_rpc.sql`** (핵심)

```sql
-- 대회 신청 RPC: 정원 확인 + 신청 생성 (동시성 처리)
CREATE OR REPLACE FUNCTION submit_tournament_application(
  p_tournament_id uuid,
  p_team_id       uuid,
  p_division_id   uuid,
  p_user_id       uuid
) RETURNS jsonb
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
SET row_security = off
AS $$
DECLARE
  v_division       divisions%ROWTYPE;
  v_occupied       integer;
  v_available      integer;
  v_position       integer;
  v_status         text;
  v_due_at         timestamptz;
  v_app_id         uuid;
BEGIN
  -- 1) division 조회 + 행 잠금 (동시 신청 방지)
  SELECT * INTO v_division
  FROM divisions
  WHERE id = p_division_id AND tournament_id = p_tournament_id
  FOR UPDATE;

  IF NOT FOUND THEN
    RETURN jsonb_build_object('ok', false, 'error', '유효하지 않은 division입니다.');
  END IF;

  -- 2) 현재 점유 수 계산
  SELECT COUNT(*) INTO v_occupied
  FROM tournament_team_applications
  WHERE division_id = p_division_id
    AND status IN ('payment_pending', 'paid_pending_approval', 'confirmed');

  -- 3) 자리 여부 판단
  IF v_division.capacity IS NULL OR v_occupied < v_division.capacity THEN
    v_status := 'payment_pending';
    v_due_at := now() + interval '24 hours';
    v_position := NULL;
  ELSE
    -- 대기열: 다음 순번
    SELECT COALESCE(MAX(waitlist_position), 0) + 1 INTO v_position
    FROM tournament_team_applications
    WHERE division_id = p_division_id AND status = 'waitlisted';
    v_status := 'waitlisted';
    v_due_at := NULL;
  END IF;

  -- 4) 신청 INSERT
  INSERT INTO tournament_team_applications (
    tournament_id, team_id, division_id, applied_by,
    status, base_entry_fee, final_amount,
    waitlist_position, payment_due_at, submitted_at
  ) VALUES (
    p_tournament_id, p_team_id, p_division_id, p_user_id,
    v_status,
    v_division.entry_fee,
    v_division.entry_fee,  -- discount_amount = 0이므로 final = base
    v_position,
    v_due_at,
    now()
  )
  RETURNING id INTO v_app_id;

  -- 5) 이력 기록
  INSERT INTO application_status_history (application_id, from_status, to_status, changed_by)
  VALUES (v_app_id, NULL, v_status, p_user_id);

  RETURN jsonb_build_object('ok', true, 'status', v_status, 'application_id', v_app_id);

EXCEPTION
  WHEN unique_violation THEN
    RETURN jsonb_build_object('ok', false, 'error', '이미 이 디비전에 신청이 존재합니다.');
END;
$$;
```

```sql
-- 상태 전환 RPC (취소, 승격, 만료 처리 등 원자적 처리)
-- promote_next_waitlisted: 대기열 상위 팀 승격
CREATE OR REPLACE FUNCTION promote_next_waitlisted(p_division_id uuid)
RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
SET row_security = off
AS $$
DECLARE
  v_app_id  uuid;
  v_user_id uuid;
BEGIN
  SELECT id, applied_by INTO v_app_id, v_user_id
  FROM tournament_team_applications
  WHERE division_id = p_division_id AND status = 'waitlisted'
  ORDER BY waitlist_position ASC
  LIMIT 1
  FOR UPDATE SKIP LOCKED;

  IF NOT FOUND THEN RETURN; END IF;

  UPDATE tournament_team_applications
  SET
    status = 'payment_pending',
    waitlist_position = NULL,
    payment_due_at = now() + interval '12 hours'
  WHERE id = v_app_id;

  INSERT INTO application_status_history (application_id, from_status, to_status, changed_by)
  VALUES (v_app_id, 'waitlisted', 'payment_pending', v_user_id);
END;
$$;
```

### 5-5. 권한 정책 (RLS)

#### 현재 RLS 현황

| 대상 | 정책 | 내용 |
|---|---|---|
| `tournament_team_applications` SELECT | `tournament_team_applications_select` | organizer 또는 해당 팀 멤버 |
| `tournament_team_applications` INSERT | `tournament_team_applications_insert` | organizer 또는 (team_manager + applied_by = auth.uid()) |
| `tournament_team_applications` UPDATE | `tournament_team_applications_update` | organizer만 |
| `tournament_team_applications` DELETE | 없음 | 삭제 불가 (soft delete 방식) |

#### 변경 필요 사항

1. **사용자의 입금 완료 표시**: 현재 사용자는 UPDATE 불가 → `mark_payment_done` RPC(SECURITY DEFINER)로 우회 처리
2. **사용자의 신청 취소**: `cancel_application` RPC(SECURITY DEFINER)로 처리
3. **새로운 status 값들**: CHECK 제약에 새 값 추가 (마이그레이션 2에서 처리)
4. **`application_status_history`**: SELECT는 organizer + 해당 팀 멤버, INSERT는 RPC 내부에서만

#### RPC 기반 상태 전환 권장 이유

직접 UPDATE RLS로 상태 전환을 허용하면 클라이언트가 임의 상태 값을 넣을 수 있음. 모든 상태 전환을 `SECURITY DEFINER` RPC로 래핑하면:
- 상태 전환 유효성 검증을 DB 레벨에서 강제
- 이력 기록을 원자적으로 처리
- 부수 효과(대기열 승격, 알림)도 트랜잭션 내 처리 가능

## 7. 구현 시 주의사항

### 7-1. UNIQUE 제약 변경의 위험성

**현재**: `UNIQUE(tournament_id, team_id)` → 팀당 대회 1개 신청만 가능  
**변경 후**: `Partial UNIQUE INDEX (team_id, division_id) WHERE status IN (...)` → division 단위로 활성 신청 중복 방지

이 변경은 기존 데이터에 영향 없으나, **기존 코드가 "(tournament_id, team_id)"로 uniqueness를 가정하는 부분을 모두 확인해야 함**.  
특히 `getMyApplicationStatus()`가 현재 `.limit(1)`으로 하나만 가져오는데, 변경 후에는 팀이 여러 division에 신청했을 수 있으므로 특정 division 기준으로 조회해야 함.

### 7-2. 동시성 처리 (정원 초과 방지)

단순 `SELECT count(*) → INSERT` 패턴은 경쟁 조건에 취약. 반드시 **`submit_tournament_application` RPC 내에서 `SELECT ... FOR UPDATE`로 divisions 행을 잠근 후** 정원 계산 및 INSERT를 수행해야 함.

클라이언트 측에서 직접 Supabase 클라이언트로 INSERT하면 절대 안 됨 → 반드시 서버 액션 → RPC 경유.

### 7-3. 상태 전환 유효성 검증

잘못된 상태 전환 방지 필요 (예: `confirmed → payment_pending`은 불가):

```typescript
const VALID_TRANSITIONS: Record<ApplicationStatus, ApplicationStatus[]> = {
  payment_pending:       ["paid_pending_approval", "expired", "cancelled"],
  paid_pending_approval: ["confirmed", "cancelled"],
  confirmed:             ["cancelled"],
  waitlisted:            ["payment_pending", "cancelled"],
  expired:               [],  // 종료 상태
  cancelled:             [],  // 종료 상태
};
```

### 7-4. 마이그레이션 2의 기존 데이터 변환

현재 'approved' 상태 신청들은 'confirmed'로 변환됨. 기존 approved 팀들이 bracket, matches 등 하위 기능과 연결되어 있는지 확인 필요. `listApprovedTeamsByDivision()` 등 downstream 함수들이 `status = 'approved'`를 하드코딩하고 있으므로 모두 `status = 'confirmed'`로 변경해야 함.

**영향 받는 파일들:**
- `lib/api/applications.ts`: `listApprovedTeamsForTournament()`, `listApprovedTeamsByDivision()` — 'approved' → 'confirmed'
- `lib/api/divisions.ts`: `listDivisionsWithStats()` — `.eq("status", "approved")` → `.eq("status", "confirmed")`
- `app/(app)/admin/tournaments/[id]/applications/Form.tsx` — 상태 배지
- `app/(app)/admin/tournaments/[id]/bracket/`, `matches/`, `standings/` 등 approved 팀 기준 로직

### 7-5. capacity = NULL 처리

`capacity IS NULL`은 "무제한"을 의미. `getAvailableSlots()`에서 NULL 체크 필수:
```typescript
if (division.capacity === null) return Infinity; // 항상 자리 있음
```

### 7-6. 입금 기한 만료 처리의 정확도

Edge Function 주기(10분 간격 권장)로 인해 만료 처리에 최대 10분 지연 발생. UI에서 `payment_due_at`를 클라이언트에서 실시간 카운트다운으로 표시하되, 실제 만료는 Edge Function이 처리함을 안내해야 함. (만료 직후 UI에서 "만료됨"처럼 보여도 DB는 아직 payment_pending일 수 있음)

### 7-7. 알림 시스템의 점진적 구현

알림 테이블만 먼저 만들고, 서버 액션에서 `INSERT INTO notifications` 직접 호출로 시작. Edge Function, 이메일, 카카오 알림톡은 추후 확장. 모든 상태 전환 함수에서 알림 INSERT를 빠뜨리지 않도록 공통 헬퍼 `createNotification(userId, type, title, body, data)` 함수로 추상화.

### 7-8. 대기열 순번 정렬 유지

취소/만료로 자리가 생길 때 `promote_next_waitlisted()`를 호출해야 함. 이를 누락하면 대기자가 영원히 대기하게 됨. 자리를 반환하는 모든 경로(사용자 취소, 운영자 취소, 만료 처리)에서 반드시 호출해야 함.

### 7-9. 정원 감소 유효성 검증

`updateDivision(capacity)` 호출 시, 현재 점유 수(`payment_pending + paid_pending_approval + confirmed` 수)보다 낮은 값으로 감소시키는 것을 방지해야 함. DB 레벨 CHECK로는 처리 어려우므로 서버 액션/API 레이어에서 검증:

```typescript
const occupied = await getOccupiedCount(divisionId);
if (newCapacity < occupied) {
  return { ok: false, error: `현재 ${occupied}팀이 자리를 차지하고 있습니다.` };
}
```

## 9. 결론

### 현재 구현된 것 vs 신규 구현 필요 항목

| 항목 | 현재 상태 | 필요 작업 |
|---|---|---|
| 신청 기본 플로우 | pending → approved/rejected 3단계 | 6단계 상태로 전면 확장 |
| 정원 관리 | 없음 | divisions.capacity + 점유 수 계산 |
| 대기열 | 없음 | waitlisted + waitlist_position + 승격 RPC |
| 참가비 관리 | 없음 | entry_fee 스냅샷, discount, final_amount |
| 입금 기한 | 없음 | payment_due_at + Edge Function 만료 처리 |
| 입금 완료 표시 | 없음 | depositor_name, paid_marked_at, RPC |
| 운영자 수동 확인 | approved 버튼 1개 | confirmed 전환 + approved_by, approved_at |
| 상태 이력 | 없음 | application_status_history 테이블 신규 |
| 알림 시스템 | 없음 | notifications 테이블 + 발송 로직 신규 |
| 동시성 처리 | 없음 (race condition 존재) | SELECT FOR UPDATE 기반 RPC |
| 취소 플로우 | rejected만 (운영자 전용) | 사용자/운영자 취소 + 대기열 승격 |
| 재신청 | 불가 (UNIQUE 제약) | Partial index 변경으로 가능하게 |

### 구현 규모 요약

- **마이그레이션**: 7개 신규 파일 (약 350줄 SQL)
- **API 함수**: 기존 5개 수정 + 신규 약 8개 함수
- **서버 액션**: 기존 2개 + 신규 약 5개 액션
- **프론트엔드**: 기존 4개 파일 대폭 수정 + 신규 2~3개 컴포넌트
- **Edge Function**: 1개 신규 (만료 처리)

### 후속 확장 포인트

1. **PG 결제 연동**: `final_amount` 필드와 결제 provider를 연결하는 `payments` 테이블 추가
2. **환불 자동화**: `cancelled_at` + 결제 provider webhook → 자동 환불 트리거
3. **알림 채널 확장**: notifications 테이블의 `type` + `data` 필드를 이용해 이메일/카카오 발송 라우팅
4. **대회별 환불 정책**: `tournaments` 또는 별도 `refund_policies` 테이블로 분리
5. **입금 기한 정책 커스터마이징**: `tournaments` 또는 `divisions`에 `payment_deadline_hours` 필드 추가
