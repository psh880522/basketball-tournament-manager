## 3. 핵심 데이터 모델 (DB 스키마)

### 3.1 주요 테이블과 관계

```
profiles ──────────────────── auth.users (1:1)
  - id (uuid)
  - role: "organizer" | "manager" | "player"

tournaments ────────────────── profiles (n:1 via created_by)
  - status: "draft" | "open" | "closed" | "finished"
  - deleted_at: NULL 기반 soft delete

divisions ──────────────────── tournaments (n:1)
  - group_size, tournament_size
  - standings_dirty: boolean (순위 재계산 트리거)
  - sort_order

groups ─────────────────────── divisions (n:1)
  - type: "league" | "tournament"
  - order: int

teams ──────────────────────── tournaments (n:1)
  - status: "pending" | "approved" | "rejected"
  - is_dummy: boolean
  - captain_user_id

team_members ───────────────── teams, auth.users (n:1 each)
  - role_in_team: "captain" | "manager" | "member"

tournament_team_applications ── tournaments, teams, divisions (n:1 each)
  - status: "pending" | "approved" | "rejected"

players ────────────────────── teams (n:1)
  - number, position

courts ─────────────────────── tournaments (n:1)
  - display_order

matches ────────────────────── tournaments, divisions, groups, teams(x2), courts
  - team_a_id, team_b_id (FK to teams)
  - seed_a, seed_b (토너먼트 브래킷용)
  - status: "pending" | "in_progress" | "completed"
  - scheduled_at
  - winner_team_id

schedule_slots ─────────────── tournaments, courts, divisions, matches
  - slot_type: "break" | "maintenance" | "buffer" | "match" | "tournament_placeholder"
  - start_at, end_at

standings (group_standings) ── groups, teams
standings (division_standings) ─ divisions, teams
  - wins, losses, points_for, points_against, rank
```

### 4.3 권한 검증 이중화 구조

**Layer 1: Supabase RLS** (DB 레벨)
- profiles, tournaments, teams, matches, players 등 모든 핵심 테이블에 RLS 적용
- `auth.uid()` 기반 행 수준 접근 제어

**Layer 2: Server Action** (애플리케이션 레벨)
```typescript
// tournaments.ts - ensureOrganizer() 패턴 (이 파일에만 존재)
async function ensureOrganizer(): Promise<ActionResult> {
  const result = await getUserWithRole()
  if (result.status === "unauthenticated") return { ok: false, error: "..." }
  if (result.status === "error") return { ok: false, error: result.error ?? "..." }
  if (result.status === "empty") return { ok: false, error: "..." }
  if (result.role !== "organizer") return { ok: false, error: "..." }
  return { ok: true }
}

// results/actions.ts - 인라인 패턴 (다른 파일들)
const auth = await getUserWithRole()
if (auth.status === "unauthenticated") return { ok: false, error: "..." }
if (auth.status === "error" || auth.status === "empty") return { ok: false, error: "..." }
if (!isOperationRole(auth.role)) return { ok: false, error: "..." }
```

**Layer 3: tournamentGuards.ts** (단계별 검증)
- `assertTournamentStepAllowed()` - 대회 단계별 허용 여부를 통합 검증
- stepKey 기반으로 organizer 전용 / operation(organizer+manager) 분기

## 5. 주요 데이터 흐름 추적

### 5.1 대회 생성 → 종료 전체 워크플로우

```
1. 대회 생성 (organizer)
   app/admin/tournaments/new/Form.tsx
   → app/admin/tournaments/actions.ts (createTournamentAction)
   → lib/api/tournaments.ts (내부 구현)
   → supabase: tournaments 테이블 INSERT
   → redirect /admin/tournaments/[id]

2. Division 생성
   → supabase: divisions 테이블 INSERT

3. 상태 변경 (draft → open)
   app/admin/tournaments/[id]/actions.ts (changeTournamentStatusAction)
   → lib/api/tournaments.ts (changeTournamentStatus)
   → 현재 상태 조회 후 유효성 검증
   → supabase: tournaments UPDATE status

4. 팀 참가 신청 (player)
   app/tournament/[id]/apply/Form.tsx
   → app/tournament/[id]/apply/actions.ts
   → lib/api/applications.ts (applyToTournament)
   → supabase: tournament_team_applications INSERT

5. 팀 승인/거부 (organizer)
   app/admin/tournaments/[id]/applications/Form.tsx
   → app/admin/tournaments/[id]/applications/actions.ts
   → lib/api/applications.ts (setApplicationStatus)
   → supabase: tournament_team_applications UPDATE status

6. 그룹 및 경기 생성 (organizer)
   app/admin/tournaments/[id]/bracket/Form.tsx
   → app/admin/tournaments/[id]/bracket/actions.ts
   → lib/api/bracket.ts (createGroups, createGroupTeams, createMatches)
   → lib/api/applications.ts (listApprovedTeamsByDivision)
   → supabase: groups INSERT + matches INSERT

7. 코트 설정 + 일정 자동 생성 (organizer)
   app/admin/tournaments/[id]/schedule/Form.tsx
   → app/admin/tournaments/[id]/schedule/actions.ts
   → lib/api/schedule.ts (generateSchedule)
   → courts, divisions, matches 순서로 조회 후 scheduled_at/court_id 일괄 업데이트
   → 경기 수 × 개별 UPDATE (N+1 패턴)

8. 결과 입력 (organizer + manager)
   app/admin/tournaments/[id]/result/Form.tsx (신규)
   OR app/admin/tournaments/[id]/results/Form.tsx (구버전)
   → 각각의 actions.ts
   → lib/api/results.ts (saveLeagueResult / saveMatchResult)
   → supabase: matches UPDATE score_a, score_b, status, winner_team_id

9. 순위 계산 (organizer + manager)
   → lib/api/results.ts (calculateLeagueStandings, confirmLeagueStandings)
   → standings 테이블 업데이트
   → divisions.standings_dirty = false

10. 토너먼트 브래킷 시드 배치 (organizer)
    → lib/api/results.ts (seedTournamentTeamsFromConfirmedStandings)
    → standings 기반 matches.team_a_id / team_b_id 배치

11. 대회 종료 (organizer)
    → changeTournamentStatus("finished")
    → OR finishTournament()
```

### 🔴 문제 1: `result/` vs `results/` - 동일 기능 라우트 중복 (우선순위: 최고)

**현황:**
```
app/admin/tournaments/[id]/
├── result/          # 신규: 리그 결과 + 토너먼트 결과 통합 입력
│   ├── Form.tsx
│   ├── actions.ts   # saveLeagueResultsAction, saveTournamentResultAction 등
│   ├── components/ResultForm.tsx
│   └── page.tsx
└── results/         # 구버전: 경기별 단순 점수 입력
    ├── Form.tsx
    ├── actions.ts   # saveMatchScoreAction, completeMatchAction
    └── page.tsx
```

**`result/actions.ts` (신규):**
```typescript
saveLeagueResultsAction()     // lib/api/results.ts 호출
calculateLeagueStandingsAction()
confirmLeagueStandingsAction()
seedTournamentTeamsAction()
saveTournamentResultAction()
```

**`results/actions.ts` (구버전):**
```typescript
saveMatchScoreAction()    // lib/api/matches.ts 호출 (saveMatchResult)
completeMatchAction()     // lib/api/matches.ts 호출
```

**영향 범위:**
- `lib/api/results.ts`와 `lib/api/matches.ts` 두 파일의 함수가 각각 다른 경로에서 사용됨
- 라우팅 상 `/result`와 `/results` 둘 다 존재 → 사용자 혼동 가능
- `ProgressIndicator.tsx`의 링크가 어느 쪽을 가리키는지 확인 필요

**즉시 적용 가능한 개선:**
- `results/` 폴더가 구버전이라면 삭제 또는 deprecated 표시
- 두 기능의 차이를 명확히 하고 하나의 통합 페이지로 정리
- `lib/api/matches.ts`의 `saveMatchResult`를 `lib/api/results.ts`에 통합

### 🔴 문제 2: `schedule-slots.ts` vs `scheduleSlots.ts` - API 파일 이름 충돌 (우선순위: 최고)

**현황:**
```
lib/api/
├── schedule-slots.ts    # 대규모 파일 (30k+ tokens) - ScheduleSlot 조회/생성/편집 등 복잡한 로직
└── scheduleSlots.ts     # 단순 CRUD (listScheduleSlots, createScheduleSlot, updateScheduleSlot)
```

**`scheduleSlots.ts`의 타입:**
```typescript
export type ScheduleSlotRow = {
  slot_type: "break" | "maintenance" | "buffer" | "match" | "tournament_placeholder"
}
```

**`schedule-slots.ts`의 타입:**
```typescript
export type ScheduleSlot = {
  slot_type: string  // 타입이 느슨함
}
export type ScheduleSlotDivisionGroup = { ... }
export type ScheduleSlotCourtGroup = { ... }
// 더 많은 복잡한 그룹 타입들
```

### 🟠 문제 3: `ApiResult<T>` / `ActionResult` 타입 파일마다 재정의 (우선순위: 높음)

**현황:** 총 10개 이상의 파일에서 동일한 타입을 로컬로 선언:

```typescript
// tournaments.ts
type ApiResult<T> = { data: T | null; error: string | null }
type ActionResult = { ok: true } | { ok: false; error: string }

// schedule.ts
type ActionResult = { ok: true } | { ok: false; error: string }

// results.ts
type ApiResult<T> = { data: T | null; error: string | null }
type ActionResult = { ok: true; message?: string } | { ok: false; error: string }
// ↑ message?: string 이 있는 변형 - 일부 파일은 다르다!

// scheduleSlots.ts
type ActionResult = { ok: true } | { ok: false; error: string }

// tournamentGuards.ts
type GuardResult = { ok: true } | { ok: false; error: string }
// ↑ ActionResult와 구조 동일하지만 이름이 다름
```

**영향:**
- `results.ts`의 `ActionResult`에는 `message?: string`이 있고 다른 파일에는 없다.
- `GuardResult`와 `ActionResult`는 구조가 같지만 이름이 달라 혼용 불가.
- 타입 변경 시 모든 파일을 개별 수정해야 함.

**즉시 적용 가능한 개선 (`lib/types/api.ts` 신규 생성):**
```typescript
// lib/types/api.ts
export type ApiResult<T> = {
  data: T | null
  error: string | null
}

export type ActionResult<TData = void> =
  | (TData extends void ? { ok: true } : { ok: true; data: TData })
  | { ok: false; error: string }

export type ActionResultWithMessage =
  | { ok: true; message?: string }
  | { ok: false; error: string }
```

### 🟠 문제 4: 권한 검증 패턴 인라인 반복 (우선순위: 높음)

**현황:** `getUserWithRole()` 호출 후 동일한 상태 체크 로직이 여러 파일에 반복:

if (!isOperationRole(auth.role)) return { ok: false, error: "권한이 없습니다." }

// tournaments.ts (ensureOrganizer - 이 파일에만 헬퍼 존재)

- `ensureOrganizer()` 헬퍼가 `tournaments.ts` 내부에만 있고 export되지 않아 외부에서 재사용 불가
- 각 파일이 독립적으로 동일한 검증 로직을 작성
- 에러 메시지가 파일마다 미묘하게 다름 ("인증 오류가 발생했습니다." vs "Auth error." 등)

**즉시 적용 가능한 개선 (`src/lib/auth/guards.ts` 신규 생성):**
```typescript
// src/lib/auth/guards.ts
import { getUserWithRole, isOperationRole, Role } from "./roles"

type GuardOk = { ok: true; userId: string; role: Role }
type GuardFail = { ok: false; error: string }
type GuardResult = GuardOk | GuardFail

export async function requireOrganizer(): Promise<GuardResult> {
  const result = await getUserWithRole()
  if (result.status !== "ready") return { ok: false, error: AUTH_ERRORS[result.status] }
  if (result.role !== "organizer") return { ok: false, error: "권한이 없습니다." }
  return { ok: true, userId: result.user!.id, role: result.role }
}

export async function requireOperationRole(): Promise<GuardResult> {
  const result = await getUserWithRole()
  if (result.status !== "ready") return { ok: false, error: AUTH_ERRORS[result.status] }
  if (!isOperationRole(result.role)) return { ok: false, error: "권한이 없습니다." }
  return { ok: true, userId: result.user!.id, role: result.role! }
}

const AUTH_ERRORS: Record<string, string> = {
  unauthenticated: "로그인이 필요합니다.",
  error: "인증 오류가 발생했습니다.",
  empty: "프로필이 없습니다.",
}
```

## 8. 폴더 구조 및 책임 분리 분석

### 8.1 현재 `lib/` vs `src/lib/` 이원화 구조

```
lib/           # 도메인 로직 (api, constants, formatters)
src/lib/       # 인프라 레이어 (supabase, auth)
```

**현황:**
- `src/lib/` → 인프라 (Supabase 클라이언트, 인증 헬퍼)
- `lib/api/` → 도메인별 DB 접근 함수
- `lib/constants/` → 상수
- `lib/formatters/` → 순수 데이터 변환 함수

**평가:** 의도는 좋지만 `src/lib/`과 `lib/`이 분리되어 있어 import 경로가 `@/src/lib/...`과 `@/lib/...` 두 가지로 나뉨. 일관성 관점에서 모두 `lib/`으로 통일하거나 모두 `src/lib/`으로 이동하는 것이 더 깔끔함.

### 8.2 타입 정의 위치

현재 타입은 각 API 파일 내에 지역적으로 정의되어 있음:
- 외부에서 필요한 타입은 `export type`으로 공개
- `ApiResult<T>`, `ActionResult`는 각 파일에서 재선언

**제안:** `lib/types/` 폴더에 공유 타입 모음 (도메인 타입 제외, 인프라/패턴 타입만):
```
lib/types/
├── api.ts      # ApiResult, ActionResult
├── auth.ts     # Role, UserWithRoleResult (src/lib/auth/roles.ts에서 이동 or 재export)
└── index.ts    # barrel export
```

## 10. 현재 잘 구현된 부분 (유지해야 할 패턴)

1. **Server Component 기본 원칙 준수** - 모든 페이지가 서버 컴포넌트, 필요한 곳만 `"use client"`
2. **권한 이중 검증** - Supabase RLS + Server Action 레벨 검증으로 보안 강화
3. **soft delete 패턴** - `deleted_at` nullable 컬럼으로 복구 가능
4. **`standings_dirty` 플래그** - 매 경기마다 즉시 재계산 대신 필요 시 재계산 트리거
5. **Suspense 경계** - 홈페이지의 병렬 데이터 스트리밍
6. **React cache** - Supabase 클라이언트 요청 기간 캐싱
7. **`tournamentGuards.ts`** - 대회 단계별 권한을 한 곳에서 통합 관리
8. **`tournamentProgress.ts`** - 진행 상태를 단일 함수로 추상화 (성능 개선 필요하지만 구조는 좋음)
9. **Vertical Slice** - 기능별로 page/Form/actions가 함께 위치하여 응집도 높음
10. **마이그레이션 순차 관리** - 53개 마이그레이션으로 DB 변경 이력 추적 가능
