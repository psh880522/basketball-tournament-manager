## 4. DB 스키마 및 RLS 정책

### 4.1 app_role ENUM

**마이그레이션**: `0215_role_add_user.sql`

```sql
CREATE TYPE public.app_role AS ENUM ('organizer', 'manager', 'user', 'player');
```

| role | 의미 |
|------|------|
| `organizer` | 대회 주최자 (최고 권한) |
| `manager` | 대회 운영진 |
| `user` | 가입만 한 사용자 (온보딩 미완료) |
| `player` | 선수 등록 완료 (본인인증 완료) |

### 4.2 profiles 테이블

**마이그레이션**: `0002_profiles_roles.sql`, `0214_profiles_personal_info.sql`

```sql
CREATE TABLE public.profiles (
  id                   uuid PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  role                 public.app_role NOT NULL DEFAULT 'user',
  display_name         text,                    -- 표시명(닉네임)
  phone                text,                    -- 연락처 (현재 프로필 입력 단계에서 수집)
  birth_date           date,                    -- 생년월일 (현재 프로필 입력 단계에서 수집)
  identity_verified_at timestamptz DEFAULT NULL, -- 본인인증 완료 시점
  created_at           timestamptz NOT NULL DEFAULT now()
);
```

> **⚠️ 설계 검토 필요**: 현재 `phone`, `birth_date`는 프로필 입력 단계에서 수집되지만,
> 요구사항상 이 값들은 **본인인증 단계의 확정값**으로 관리해야 함.
> 프로필 입력 단계에서는 이 필드를 수집하지 않도록 흐름 변경이 필요.

**RLS 정책**:

```sql
-- SELECT: 본인 또는 organizer만 조회 가능
CREATE POLICY "profiles_select_own_or_organizer"
  ON public.profiles FOR SELECT
  USING (id = auth.uid() OR public.is_organizer());

-- UPDATE: 본인만 수정 가능 (column-level 보호는 API 레이어에서 처리)
CREATE POLICY "profiles_update_own"
  ON public.profiles FOR UPDATE
  USING (id = auth.uid()) WITH CHECK (id = auth.uid());
```

### 4.3 identity_verifications 테이블

**마이그레이션**: `0219_identity_verifications.sql`

```sql
CREATE TABLE public.identity_verifications (
  id             uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id        uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  verified_at    timestamptz NOT NULL DEFAULT now(),
  provider       text NOT NULL DEFAULT 'mock',   -- 'mock', 'pass' 등

);
```

**RLS 정책**:

```sql

CREATE POLICY "identity_verifications_select_own"
  ON public.identity_verifications FOR SELECT
  USING (user_id = auth.uid());

-- INSERT/UPDATE/DELETE: 금지 (promote_to_player() RPC를 통해서만 조작)
```

### 4.4 핵심 RPC 함수

**promote_to_player()** — `0220_promote_to_player_rpc.sql`

```sql
CREATE OR REPLACE FUNCTION public.promote_to_player(
  p_provider       text    DEFAULT 'mock',
  p_provider_tx_id text    DEFAULT NULL,
  p_raw_response   jsonb   DEFAULT NULL
)
RETURNS void LANGUAGE plpgsql SECURITY DEFINER AS $$
DECLARE
  v_user_id uuid := auth.uid();
BEGIN
  IF v_user_id IS NULL THEN RAISE EXCEPTION 'Unauthenticated'; END IF;

  -- 멱등 처리: 이미 player면 조용히 반환
  IF EXISTS (SELECT 1 FROM profiles WHERE id = v_user_id AND role = 'player') THEN
    RETURN;
  END IF;

  -- user role만 승격 가능
  IF NOT EXISTS (SELECT 1 FROM profiles WHERE id = v_user_id AND role = 'user') THEN
    RAISE EXCEPTION 'Only user role can be promoted to player';
  END IF;

  -- 1. 인증 이력 저장
  INSERT INTO public.identity_verifications (user_id, provider, provider_tx_id, raw_response)
  VALUES (v_user_id, p_provider, p_provider_tx_id, p_raw_response);

  -- 2. role 승격 + 인증 타임스탬프
  UPDATE public.profiles
  SET role = 'player', identity_verified_at = now()
  WHERE id = v_user_id;
END;
$$;
```

**update_user_role()** — `0218_update_user_role_add_user.sql`
- organizer 전용: 사용자 역할 수동 변경 (admin 영역)
- 허용값: `user`, `player`, `manager` (organizer 승격 불가)

### 4.5 DB 헬퍼 함수 (SECURITY DEFINER)

```sql
public.is_organizer()   → boolean  -- organizer 여부
public.is_manager()     → boolean  -- organizer or manager 여부
public.is_player()      → boolean  -- player 여부
```

> - `phone`, `birth_date`는 본인인증 단계의 확정값으로 이동해야 함

**흐름**:
```
/onboarding/profile → 입력 저장(updateMyProfile) → Toast → /onboarding/identity로 이동
```

### 6.3 본인인증 — ⚠️ Mock 단계 (실제 provider 미연동)

| 파일 | 역할 |

**현재 구현**:
```typescript

         ↓

         ↓

```

**실제 provider**: 미구현 (`requireIdentityVerification === true` 시 항상 오류 반환)

### 6.4 Role 전환 (user → player) — ✅ 구현 완료

- `promote_to_player()` RPC: 트랜잭션으로 인증 이력 저장 + role 승격 동시 처리
- 멱등성 보장: 이미 player면 오류 없이 반환
- 성공 후: `router.refresh()` + 1.5초 후 `/dashboard` 리다이렉트

### 6.5 선수 관련 추가 테이블 — ❌ 현재 없음

현재 `profiles` 테이블이 유일한 사용자 데이터 테이블.  
요구사항에 명시된 `성별`, `포지션`, `신장`, `경력 수준`, `활동 지역`, `등번호` 등을 저장할 별도 player 테이블/컬럼이 없음.

```typescript
"use client";
export default function MyForm({ initialValues }: Props) {
```

## 11. 구현 시 주의사항 및 제약사항

### 11.1 프로필 입력 항목 재설계 필요

**현재 상태**:
- `profiles` 테이블: `display_name`, `phone`, `birth_date`
- 온보딩 폼: `display_name`(필수), `phone`(필수), `birth_date`(선택)

**요구사항과의 차이**:
- `phone`, `birth_date`는 본인인증 단계 확정값 → 프로필 입력 단계에서 수집하면 안 됨
- 성별, 포지션, 신장, 경력 수준, 활동 지역 → 현재 DB/폼 모두 없음
- 서브 포지션, 체중, 등번호 → 현재 없음

**설계 결정 필요**: 이 필드들을 `profiles` 테이블 컬럼으로 추가할지, 별도 `player_profiles` 테이블로 분리할지 결정해야 함

### 11.2 본인인증 확정값 저장 위치

**요구사항**: 실명, 생년월일, 휴대폰번호는 본인인증 단계의 확정값으로 관리

**현재 상태**:
- `profiles.phone`: 프로필 입력 단계에서 수집 (충돌)
- `profiles.birth_date`: 프로필 입력 단계에서 수집 (충돌)
- `identity_verifications.raw_response`: provider 응답 JSONB로 저장 (비정형)

**권장 방향**:
```sql
-- profiles 테이블에 본인인증 확정 컬럼 추가 (별도 네이밍)
ALTER TABLE public.profiles
  ADD COLUMN verified_name      text DEFAULT NULL,      -- 실명 (본인인증 확정)
  ADD COLUMN verified_phone     text DEFAULT NULL,      -- 휴대폰 (본인인증 확정)
  ADD COLUMN verified_birth_date date DEFAULT NULL;     -- 생년월일 (본인인증 확정)
```

또는 `promote_to_player()` RPC를 확장하여 확정값 저장

### 11.3 미들웨어 없음

- 라우팅 보호는 **각 page.tsx 마다** 직접 `getUserWithRole()` 호출 필요
- `onboarding/identity/page.tsx`에 현재 프로필 입력 완료 여부 가드가 없음 → 추가 필요

### 11.4 약관 동의 타이밍

- 회원가입 시 서비스/개인정보 동의: 최초 1회
- 약관 변경 시 재동의 로직 필요 (버전 관리 필수)
- 마케팅 동의: 언제든 변경 가능 → 별도 관리

### 11.5 RLS Column-level 보호 미지원

Supabase RLS는 컬럼 단위 보호가 없음.  
API 레이어(`updateMyProfile`)에서 허용 컬럼만 명시하여 보호하는 현재 방식 유지 필요.  
본인인증 확정값 컬럼은 API에서 노출되지 않도록 별도 관리.

### 11.6 promote_to_player() RPC

- 현재: `user` role만 승격 가능
- 프로필 완료 검증 없음 → 프로필 미입력 상태에서도 본인인증 통과 시 player 전환됨
- 개선 방향: RPC에 프로필 완료 여부 사전 체크 추가 또는 Server Action에서 사전 검증

### 11.7 환경별 분기

```typescript
// src/lib/config/env.ts
const isProduction = process.env.NODE_ENV === "production";
export const requireIdentityVerification = isProduction;
```

- 개발 환경: mock adapter 사용 (provider 없이 즉시 player 전환)
- 프로덕션: 실제 provider 필수 (현재 오류 반환)

## 12. 결론 및 권장 구현 방향

### 12.1 구현 단계 제안

**Phase 1 — DB 설계 확정 (마이그레이션 먼저)**
1. `player_profiles` 테이블 또는 `profiles` 컬럼 확장 결정
2. `user_terms_consents` 테이블 생성
3. 본인인증 확정값 컬럼 위치 결정 (`profiles` 또는 별도 테이블)
4. `promote_to_player()` RPC 확장 (확정값 저장 포함)

**Phase 2 — 회원가입 약관 동의**
1. 약관 동의 API (`lib/api/terms.ts`)
2. `signup/Form.tsx` 체크박스 추가
3. `signup/actions.ts` 검증 + 저장

**Phase 3 — 프로필 입력 개편**
1. 선수 프로필 API (`lib/api/player-profile.ts`)
2. 프로필 입력 폼 전면 개편 (성별/포지션/신장/경력/지역)
3. 프로필 단계 약관 동의 추가

**Phase 4 — 본인인증 단계**
1. 실제 provider 선정 (PASS 권장)
2. Provider adapter 구현
3. `IdentityForm.tsx` 개편 (실명/생년월일/번호 입력)
4. 확정값 저장 로직 추가

### 12.2 핵심 아키텍처 유지

아래 구조는 검증된 패턴이므로 그대로 유지:
- `getUserWithRole()` 기반 Server Component 가드
- `ActionResult` 타입 기반 Server Action 반환
- `ApiResult<T>` 타입 기반 DB 레이어
- `promote_to_player()` RPC를 통한 role 전환 (SECURITY DEFINER)
- UI 컴포넌트 (`Input`, `Button`, `Card`, `Toast`) 재사용

### 12.3 주요 설계 원칙 (요구사항 기반)

- 프로필 입력 단계: `display_name`(표시명) 기준, 실명/번호/생년월일 수집 금지
- 본인인증 단계: 실명/생년월일/휴대폰 최종 확정, 해당 값만 공식 레코드에 저장
- role 전환 조건: 본인인증 완료 시에만 `user → player` 전환
- 프로필 입력만으로는 player 전환 불가
