### 현재 구현의 한계

- 다가오는 **경기 일정** 데이터 조회 없음 (`matches` 테이블은 존재하지만 선수 뷰 미구현)
- 요약 카드(소속 팀 수, 진행 중 대회 수, 이번 주 경기 수, 액션 수) 없음
- "지금 해야 할 일" 섹션 없음
- 팀별 대회 참가 현황 묶음 표시 없음
- 최근 결과 / 공지사항 없음
- `listMyApplications()`가 **캡틴(captain)인 팀의 신청만** 반환 → 일반 팀원은 자기 팀 신청 현황을 볼 수 없음
- `getUserTeamStatus()`가 단일 상태를 반환 → 다중 팀 멤버십 모델과 맞지 않음

### 5-2. 서버 액션 / API

**수정 필요 파일:**

| 파일 | 수정 내용 |
|------|---------|
| `lib/api/applications.ts` | `listMyApplications()` 확장: captain + player 역할 모두 포함 (현재는 captain만) |

**신규 필요 API 함수:**

| 함수명 | 파일 위치(제안) | 설명 |
|--------|----------------|------|
| `getMyUpcomingMatches()` | `lib/api/matches.ts` | 내 모든 팀의 `status='scheduled'` 경기 (7일 이내 또는 n건) |
| `getMyDashboardSummary()` | `lib/api/dashboard.ts` | 요약 카드 데이터 단일 호출 (팀 수, 진행 대회 수, 이번 주 경기 수, 액션 수) |
| `getMyPendingActions()` | `lib/api/dashboard.ts` 또는 inline | 액션 항목: 입금 대기, 로스터 미작성, 팀 가입 신청 승인 대기(캡틴) |
| `getMyTeamApplicationsActivity()` | `lib/api/team-applications.ts` | 캡틴인 팀의 pending 가입 신청 수 |

**`getMyUpcomingMatches()` 필요 조회:**
```sql
SELECT m.*, t.name as tournament_name, d.name as division_name,
       team_a.team_name, team_b.team_name, c.name as court_name
FROM matches m
JOIN teams team_a ON m.team_a_id = team_a.id
JOIN teams team_b ON m.team_b_id = team_b.id
JOIN tournaments t ON m.tournament_id = t.id
JOIN divisions d ON m.division_id = d.id
LEFT JOIN courts c ON m.court_id = c.id
WHERE (m.team_a_id IN (user's team ids) OR m.team_b_id IN (user's team ids))
  AND m.status = 'scheduled'
  AND m.scheduled_at IS NOT NULL
ORDER BY m.scheduled_at ASC
LIMIT 20
```

**`getMyPendingActions()` 액션 항목 정의:**
1. `payment_pending` 상태 신청 → "입금 완료 표시 필요" (캡틴만)
2. 로스터가 없거나 멤버 수 0명인 `confirmed` 신청 → "로스터 작성 필요" (캡틴만)
3. 캡틴인 팀에 pending 가입 신청 존재 → "팀원 신청 승인 대기"

## 7. 구현 시 주의사항

### 1. `listMyApplications()`의 현재 제한

현재 `lib/api/applications.ts:758` 기준:
```typescript
.eq("role_in_team", "captain")  // 캡틴 팀만 조회
```
일반 팀원도 자기 팀의 대회 참가 현황을 볼 수 있어야 하므로 이 필터를 제거하거나 별도 함수로 분리해야 함.

단, 일반 팀원은 신청/취소/로스터 수정 권한이 없으므로 UI에서 액션 버튼 노출 여부를 역할로 분기해야 함.

### 2. 다중 팀 멤버십 모델

사용자는 여러 팀에 동시에 소속될 수 있으며, 어느 팀에서는 captain, 어느 팀에서는 player일 수 있다.

- `getUserTeamStatus()` 현재 반환 타입 `"no_team" | "join_pending" | "team_member" | "captain"` — 단일 상태만 반환
- 대시보드에서는 이 함수에 의존하지 않고, `listMyTeams()` 결과를 직접 분기에 사용하는 것을 권장
- 빈 상태 판단: `teams.length === 0`

### 3. 다가오는 경기 - `scheduled_at` NULL 처리

`matches.scheduled_at`이 NULL인 경우(일정 미확정)는 "다가오는 경기" 섹션에서 제외하거나 별도 표시. UI에서 날짜 정렬 시 NULL safe 처리 필수.

### 4. 경기 단계(round) 레이블

현재 `matches` 테이블에 round나 단계를 나타내는 컬럼이 없음 (seed_a, seed_b만 존재). 단계 레이블이 필요하면:
- `groups` 테이블 JOIN으로 "조별리그" 여부 판단
- 토너먼트 단계는 별도 컬럼 추가 필요하거나 seed 기반 추론 → 초기 MVP에서는 "조별리그 / 토너먼트" 정도로 단순 분류

### 5. 공지사항 섹션

현재 `announcements` 또는 `notices` 테이블이 DB에 없음. 공지사항은 이번 구현 범위에서 제외하거나, 운영자가 tournament.description을 활용하는 임시 방편 검토.

### 6. 모바일 반응형 레이아웃

- 요약 카드: 모바일에서 2열 그리드, 데스크탑에서 5열
- 팀 카드: 모바일에서 전체 너비 스택, 데스크탑에서 카드 그리드
- 다가오는 경기: `DragScroll.tsx` 활용한 수평 스크롤 또는 세로 리스트

### 7. 성능 - 병렬 데이터 조회

대시보드에서 여러 API를 호출하므로 모두 `Promise.all()`로 병렬 처리:
```typescript

  getMyDashboardSummary(),

  listMyTeams(),
  getMyUpcomingMatches(),
  listMyApplications(),
  getMyRecentMatchResults(),  // 선택사항
```

## 9. 결론

### 현재 대시보드의 역할

`app/(app)/dashboard/page.tsx`는 인증 후 진입점으로 역할 분기(관리자/선수)와 온보딩 배너를 담당하는 최소 구현 상태. 선수용 개인 허브로서의 기능이 거의 없음.

### 구현 핵심 과제

| 과제 | 난이도 | 비고 |
|------|--------|------|
| `getMyUpcomingMatches()` 신규 구현 | 중 | matches 테이블 RLS 정책 확인 필요 |
| `getMyPendingActions()` 신규 구현 | 중 | 3종 액션 항목 집계 로직 |
| `listMyApplications()` 수정 | 하 | captain 필터 제거, 권한 분기는 UI에서 |
| 다중 팀 상태 분기 UI | 중 | 3가지 빈 상태 + 풀 상태 |
| 요약 카드 + 레이아웃 | 하 | 기존 Card/Badge 컴포넌트 재사용 |

### 기존 패턴 활용

- `Card`, `Badge`, `Button`, `EmptyState`, `DragScroll` 컴포넌트 그대로 재사용
- `force-dynamic` + `Promise.all()` 병렬 로딩 패턴 유지
- 신청 상태 라벨/색상 맵 기존 정의 재사용
- 역할 배지(captain/player) 기존 표현 방식 유지

### 제외 범위

- 공지사항 섹션 (테이블 없음)
- 실시간 경기 스코어 (WebSocket 미구현)
- 선수 개인 통계 (득점, 리바운드 등 - 별도 기능)
- 관리자 통계 (완전 분리)
