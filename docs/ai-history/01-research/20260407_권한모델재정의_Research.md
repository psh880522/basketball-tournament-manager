## 1. 전역 role 재정의 필요 여부 분석

### 1-1. 현재 구조의 문제점

현재 `profiles.role = 'player'`는 두 가지 완전히 다른 상태를 하나로 묶는다:

| 상태 | 설명 | 현재 role |
|------|------|-----------|
| 가입 직후 | 프로필 미입력, 이름/연락처 없음 | `player` |
| 선수 등록 완료 | 프로필 입력 + 본인인증 완료 | `player` |

이로 인해:
- 선수 등록 미완료 사용자가 팀 생성, 대회 참가 신청 RLS를 통과 가능
- 프론트엔드에서만 막고 있어 서버/DB 레벨 보안 미흡
- "내 팀", "대시보드" 메뉴가 선수 등록 전 사용자에게도 노출됨

### 1-2. 두 가지 해결 방향 비교

#### A안: `profiles.role`에 `user` 추가 (4종 enum)

```sql
-- 새 enum: 'organizer' | 'manager' | 'user' | 'player'
-- 신규 가입 기본값: 'user'
-- 선수 등록 완료 시: 'player'로 변경
```

- `user`: 가입 직후 상태 (조회 기능 위주)
- `player`: 프로필 입력 + 본인인증 완료 상태

#### B안: `profiles.role`은 최소화, 별도 상태 컬럼으로 분리

```sql
-- role enum 유지: 'organizer' | 'manager' | 'player'
-- 신규 가입 기본값: 'player' (유지)
-- 선수 등록 상태: profiles.player_status = 'pending' | 'active'
-- 또는: profiles.is_player_activated boolean DEFAULT false
```

- `player_status = 'pending'`: 가입 직후 (구 user)
- `player_status = 'active'`: 선수 등록 완료

### 1-3. 비교 분석 (상세)

| 비교 항목 | A안 (role 확장) | B안 (상태 컬럼 분리) |
|-----------|----------------|---------------------|
| **구현 난이도** | 중 — enum 타입 마이그레이션 + 모든 role 체크 코드 수정 필요 | 하 — 컬럼 추가만으로 가능, 기존 role 체크 코드 영향 없음 |
| **기존 코드 영향 범위** | 높음 — `Role` 타입, `isOperationRole`, `getUserWithRole`, 모든 RLS 함수, admin/users 역할 변경 UI까지 전파 | 낮음 — 신규 `isPlayerActivated()` 함수 추가, 기존 코드 변경 최소화 |
| **권한 모델 명확성** | 높음 — role만 보면 권한 상태 파악 가능 | 중 — role + status 두 필드를 함께 봐야 상태 파악 가능 |
| **RLS 정합성** | 중 — `is_player()` 같은 신규 DB 함수 필요, 기존 RLS 정책 다수 수정 | 높음 — 기존 RLS 함수(`is_organizer`, `is_manager`) 영향 없음, 신규 `is_activated_player()` 추가만 필요 |
| **admin/users 기능 영향** | 높음 — 역할 변경 UI에서 `user`→`player` 승격 표시 필요, `update_user_role` RPC 수정 필요 | 낮음 — admin에서 role 변경 흐름 유지, status 컬럼은 별도 관리 |
| **향후 확장성** | 높음 — 새로운 상태(e.g. `suspended`) 추가가 명확 | 중 — 상태 컬럼이 늘어나면 복잡도 증가 가능 |
| **on_auth_user_created 트리거** | 수정 필요 — DEFAULT 'player' → 'user'로 변경 | 변경 없음 — DEFAULT 'player' 유지 가능 |
| **기존 player 데이터 처리** | 마이그레이션 필요 — 기존 player를 어느 role로 처리할지 결정 필요 | 유지 가능 — `player_status = 'active'`로 일괄 backfill |

### 1-4. 권장안: **A안 (role 확장)**

**이유**:
1. **의미적 명확성**: `role='user'`는 "아직 선수가 아닌 사람"을 정확히 표현. B안의 `role='player' + status='pending'`는 "선수인데 미활성" 모순.
2. **RLS 단순성**: DB 함수 하나(`is_activated_player()` vs `is_player_role()`)로 동일한 표현력.
3. **접근 제어 명확성**: 향후 `user`에게는 읽기 전용 RLS, `player`에게는 팀/신청 RLS — role 하나로 정책 분기 가능.
4. **사용자 경험**: admin 화면에서 사용자 목록에 `user` / `player` 표시가 명확.

단, 마이그레이션 시 **기존 `player` 데이터 처리**가 핵심 과제. 아래 섹션에서 상세 분석.

## 2. 회원가입 후 기본 상태 변경 분석

### 2-1. 현재 흐름

```
auth.signUp() 
  → on_auth_user_created 트리거
  → INSERT INTO profiles (id) VALUES (new.id)  -- role DEFAULT 'player'
  → redirect("/dashboard")  -- 프로필 미완료면 /onboarding/profile
```

### 2-2. 변경 후 목표 흐름

```
auth.signUp()
  → on_auth_user_created 트리거
  → INSERT INTO profiles (id) VALUES (new.id)  -- role DEFAULT 'user'
  → redirect("/")  -- 랜딩 또는 "선수 등록 안내" 페이지
```

### 2-3. 수정이 필요한 DB / 마이그레이션

#### ① enum 확장 마이그레이션

```sql
-- 새 마이그레이션: 0215_role_add_user.sql

-- 1) 새 enum 생성 (기존 4종)
CREATE TYPE public.app_role_v3 AS ENUM ('organizer', 'manager', 'user', 'player');

-- 2) 의존성 제거 (storage 정책, DB 함수 등 app_role 타입 참조 제거)
DROP POLICY IF EXISTS poster_update_organizer ON storage.objects;
DROP POLICY IF EXISTS poster_delete_organizer ON storage.objects;
-- is_organizer(), is_manager() 함수도 재생성 필요

-- 3) profiles 컬럼 타입 교체
ALTER TABLE public.profiles
  ALTER COLUMN role DROP DEFAULT;

ALTER TABLE public.profiles
  ALTER COLUMN role TYPE public.app_role_v3
  USING (role::text::public.app_role_v3);

-- 4) 기존 player 중 프로필 미완료자를 'user'로 다운그레이드 (선택)
-- 또는 기존 사용자 전체를 'player'로 유지 (하위 호환성 우선)
UPDATE public.profiles SET role = 'user'
  WHERE role = 'player'
    AND (display_name IS NULL OR phone IS NULL);

-- 5) 신규 기본값 변경
ALTER TABLE public.profiles
  ALTER COLUMN role SET DEFAULT 'user';

-- 6) enum 이름 교체
DROP TYPE public.app_role;
ALTER TYPE public.app_role_v3 RENAME TO app_role;
```

#### ② on_auth_user_created 트리거 — 변경 없음

트리거는 `INSERT INTO profiles (id) VALUES (new.id)`만 실행.  
`DEFAULT 'user'`로 변경하면 트리거 코드 수정 불필요.

#### ③ DB 함수 수정

```sql
-- 기존 유지 (organizer/manager 체크는 변경 없음)
CREATE OR REPLACE FUNCTION is_organizer() RETURNS boolean ...
CREATE OR REPLACE FUNCTION is_manager() RETURNS boolean ...

-- 신규 추가
CREATE OR REPLACE FUNCTION is_player() RETURNS boolean
LANGUAGE sql SECURITY DEFINER SET search_path = public AS $$
  SELECT EXISTS (
    SELECT 1 FROM profiles WHERE id = auth.uid() AND role = 'player'
  )
$$;
```

### 2-4. 수정이 필요한 타입 파일

#### `src/lib/auth/roles.ts`

```typescript
// 변경 전
export type Role = "organizer" | "manager" | "player";

// 변경 후
export type Role = "organizer" | "manager" | "user" | "player";

// 신규 추가
export function isPlayerRole(role: Role | null): boolean {
  return role === "player";
}

export function isUserRole(role: Role | null): boolean {
  return role === "user";
}

// 기존 유지 (player도 operation role이 아님)
export function isOperationRole(role: Role | null): boolean {
  return role === "organizer" || role === "manager";
}
```

### 2-5. 기존 가입/로그인 흐름과의 충돌 여부

| 파일 | 충돌 여부 | 내용 |

| `app/(auth)/login/actions.ts` | **있음** — 분기 로직 재설계 필요 | 현재: player면 프로필 미완료 → onboarding. 변경 후: user면 랜딩, player면 dashboard |

## 3. 선수 등록(player 전환) 흐름 분석

### 3-1. 기존 방향 vs 새 방향

| 항목 | 기존 방향 (VS 계획) | 새 방향 |
|------|---------------------|---------|
| 진입 시점 | 로그인 직후 강제 분기 → `/onboarding/profile` | 랜딩/대시보드에서 CTA로 자발적 진입 |
| 강제 여부 | 프로필 미완료 시 다른 페이지 접근 차단 | 비강제. 단, player 전용 기능(팀, 대회신청) 시도 시 유도 |
| 완료 기준 | display_name + phone 입력 | 프로필 입력 + 본인인증 완료 |
| role 변경 시점 | 없음 (계속 player) | 선수 등록 완료 시 `user → player`로 승격 |

### 3-2. 선수 등록 플로우 설계

```
[랜딩 또는 대시보드]
  → "선수 등록하기" CTA 클릭
  
[Step 1: 프로필 입력] /onboarding/profile
  → display_name, phone, birth_date 입력
  → 저장 완료 (profiles 업데이트)
  
[Step 2: 본인인증] /onboarding/identity (또는 /identity)
  → 외부 인증 업체 연동
  → 인증 성공 콜백 → Server Action에서 검증

[승격 트리거]
  → identity_verifications.status = 'success'
  → profiles.is_identity_verified = true
  → profiles.role = 'player'  ← 이 시점에 user → player 승격
  → redirect("/dashboard")
```

### 3-3. role 변경 시점 분석

**선수 등록 완료 = 프로필 입력 AND 본인인증 완료** 두 조건 모두 충족 시점.

두 가지 구현 방안:

#### 방안 1: 본인인증 콜백에서 원자적 처리 (권장)

```typescript
// Server Action: confirmIdentityVerification()
// 1. 외부 업체 API로 결과 재검증
// 2. identity_verifications INSERT (status='success')
// 3. profiles 업데이트 (is_identity_verified=true, role='player')
// → 단일 트랜잭션 또는 RPC로 처리
```

- 장점: 단일 지점에서 승격 처리, 롤백 명확
- 단점: 프로필 입력과 본인인증이 별도 step이므로, 프로필 미입력 상태에서 인증만 완료하는 엣지 케이스 처리 필요

#### 방안 2: 체크 함수로 조건부 승격

```typescript
// 본인인증 완료 콜백에서
// is_profile_completed(userId) && identity_verified → role = 'player'
```

- 장점: 프로필 입력 완료 여부도 함께 확인
- 단점: 상태 의존성이 복잡

**권장: 방안 1** — 본인인증 완료 콜백에서 프로필 완료 여부도 함께 검증한 뒤 승격.  
미완료 시 프로필 입력 페이지로 redirect 후 재시도 유도.

### 3-4. 프로필 입력 페이지 재구성

기존 VS-2에서 설계된 `/onboarding/profile`을 **선수 등록 플로우의 Step 1**으로 재정의.

- 기존: "로그인하면 강제로 들어오는 페이지"
- 변경: "선수 등록 CTA를 클릭했을 때 진입하는 자발적 플로우의 첫 단계"
- 이미 `app/(app)/onboarding/` 폴더가 생성(untracked)되어 있으므로, 내부 로직만 조정

## 4. 대시보드 / 랜딩 / 사이드바 분기 구조 분석

### 4-1. 현재 구조 현황

#### 현재 사이드바 (`components/layout/Sidebar.tsx` 추정)

```
Organizer/Manager:
  대시보드, 대회관리, [organizer] 권한관리

Player:
  대시보드, 내팀
```

#### 현재 GlobalHeader (`components/nav/GlobalHeader.tsx` 실제 코드)

```typescript
const items: NavItem[] = [{ label: "대회", href: "/" }];
if (isLoggedIn) {
  items.push({ label: "대시보드", href: "/dashboard" });
}
```

- `isLoggedIn` 판단: `status === "ready" || status === "empty"` — role 구분 없음
- `user` role이 생기면 이 부분도 수정 필요

#### 현재 앱 레이아웃 (`app/(app)/layout.tsx`)

```typescript
// status="ready" → Sidebar + main content
// else → GlobalHeader + main content (공개 페이지)
```

- `user`도 authenticated이므로 `status="ready"` → Sidebar가 노출됨
- Sidebar에 어떤 메뉴를 보여줄지 role 기반 분기 필요

### 4-2. 두 가지 분기 방식 비교

#### 방식 X: user는 대시보드 없는 구조 (별도 라우팅)

```
user:
  / (랜딩, GlobalHeader)
  /tournaments (대회 목록)
  /register-player (선수 등록 CTA 페이지)

player:
  /dashboard (사이드바 포함)
  /tournaments
  /team
  /profile
```

- 장점: user와 player의 UI가 완전히 분리되어 혼란 없음
- 단점: `app/(app)/layout.tsx`에서 user는 Sidebar를 안 보여야 하는 로직 추가 필요

#### 방식 Y: 같은 대시보드를 상태에 따라 다르게 렌더링

```
user → /dashboard 접근 시:
  "선수 등록이 필요합니다" 배너 + 선수 등록 CTA + 대회 목록

player → /dashboard 접근 시:
  내 팀 현황 + 대회 참가 현황 + 대회 목록
```

- 장점: URL 구조 단순, 라우팅 분기 최소
- 단점: 같은 페이지에 두 가지 상태를 표현 → 컴포넌트 복잡도 증가

### 4-3. 권장 방식: **방식 X (분리형)** — 단, 점진적 구현 가능

**이유**:

## 5. captain 권한을 전역 role이 아닌 팀 컨텍스트 권한으로 유지하는 방향 분석

### 5-1. 현재 구조 (유지 권장)

```sql
-- team_members.role_in_team: 'captain' | 'player'
-- 0207_team_members_captain.sql에서 'manager' → 'captain'으로 명칭 통일 완료

-- DB 함수
is_team_member_for_team(team_uuid)  -- 팀 멤버 여부
is_team_manager_for_team(team_uuid) -- 팀 captain 여부 (role_in_team = 'captain')
```

현재 구조는 이미 "전역 role과 팀 컨텍스트 role 분리"를 올바르게 구현하고 있다.

### 5-2. 혼동 위험 지점 분석

| 혼동 가능 지점 | 현재 상태 | 개선 방향 |
|--------------|-----------|-----------|
| `is_team_manager_for_team()` 함수명 | "manager"라는 단어가 전역 role과 혼동 가능 | `is_team_captain_for_team()`으로 rename 권장 |
| `role_in_team = 'player'` 값 | 전역 `role = 'player'`와 동일 문자열 | `role_in_team = 'member'`로 변경 권장 |
| TypeScript 타입 | `Role` 타입에 player 있고, team_member에도 player 있음 | 타입 네이밍으로 명확히 분리 필요 |

### 5-3. 네이밍 통일 권장안

#### DB 레벨

```sql
-- 마이그레이션: 0216_team_members_rename_player_to_member.sql
UPDATE team_members SET role_in_team = 'member' WHERE role_in_team = 'player';

-- 함수 rename (SECURITY DEFINER 재생성)
DROP FUNCTION is_team_manager_for_team(uuid);
CREATE FUNCTION is_team_captain_for_team(team_uuid uuid) RETURNS boolean ...
  -- role_in_team = 'captain' 확인
```

#### TypeScript 레벨

```typescript
// 전역 권한 타입 (src/lib/auth/roles.ts)
export type GlobalRole = "organizer" | "manager" | "user" | "player";
// 기존 호환성: export type Role = GlobalRole; (alias)

// 팀 내 권한 타입 (lib/types/team.ts 또는 lib/api/teams.ts)
export type TeamRole = "captain" | "member";

// 팀 멤버 타입
export type TeamMember = {
  team_id: string;
  user_id: string;
  role_in_team: TeamRole;  // 'captain' | 'member'
  created_at: string;
};
```

#### 가드 함수 분리

```typescript
// src/lib/auth/guards.ts

// 전역 가드 (기존 유지)
requireOrganizer()       // organizer만
requireOperationRole()   // organizer + manager

// 신규 추가
requirePlayer()          // player만 (user 접근 차단)

// 팀 컨텍스트 가드 (lib/api/teams.ts 또는 별도 파일)
requireTeamCaptain(teamId)  // team_members.role_in_team = 'captain' 확인
```

### 5-4. `role_in_team = 'player'` → `'member'` 변경 영향 범위

```
supabase/migrations/  → 신규 마이그레이션 (UPDATE + 함수 재생성)
lib/api/teams.ts      → TeamMember 타입, listMyTeams(), listMyManagedTeams()
lib/api/applications.ts → captain 조회 쿼리 (role_in_team='captain'은 유지)
app/(app)/team/       → 팀 멤버 표시 UI (role_in_team 값 표시 부분)
app/(app)/dashboard/  → TeamSection에서 role_in_team 표시
```

## 7. 기존 구현과의 충돌 가능성 분석

### 7-1. 로그인 후 redirect 구조

**파일**: `app/(auth)/login/actions.ts`

**현재 코드**:
```typescript
if (isOperationRole(roleResult.role)) redirect("/admin");
const profileResult = await getMyProfile();
if (!isProfileCompleted(profileResult.data)) redirect("/onboarding/profile");
redirect("/dashboard");
```

**충돌**: 현재는 `player`(프로필 미완료) → `/onboarding/profile` 강제 분기.  
새 방향에서는 `user` → 랜딩, `player` → 대시보드, 프로필 완료 강제 없음.

**수정 필요**: VS-R3에서 전면 재설계.

---

### 7-2. `getUserWithRole()` 패턴

**파일**: `src/lib/auth/roles.ts`

**현재 반환 타입**:
```typescript
type UserWithRoleResult = {
  status: "unauthenticated" | "error" | "empty" | "ready";
  user: AuthUser | null;
  role: Role | null;  // "organizer" | "manager" | "player"
  error: string | null;
};
```

**충돌**: `Role` 타입에 `"user"` 추가 시, 모든 `role === "player"` 비교가 의도치 않게 `"user"` 상태에서도 false를 반환함 → 기존에 "player면 ok" 패턴이 "user도 ok"로 오해 가능성 없음 (user는 player가 아니므로 안전).

단, `isOperationRole()` 에서 명시적으로 `"user"`를 포함하지 않음을 확인 필요.  
현재 구현: `role === "organizer" || role === "manager"` — 안전.

**영향**: `Role` 타입 확장 후, 소비처에서 `role === "player"` 체크가 이제 "선수 등록 완료"를 의미함을 코드 리뷰로 확인 필요.


### 7-5. team 생성 / 대회 참가 신청 흐름

**현재 RLS**:
```sql
-- teams INSERT
POLICY: created_by = auth.uid()  -- 인증만 되면 팀 생성 가능

-- tournament_team_applications INSERT
POLICY: is_team_manager_for_team(team_id) AND applied_by = auth.uid()
-- captain이면 신청 가능 (role 체크 없음)
```

**충돌**:
- `user` role 사용자도 팀을 생성하고 captain이 될 수 있음 (RLS가 전역 role을 체크하지 않음)
- `user`가 팀을 만들고 대회 신청을 시도할 수 있는 구조 → 불가

**수정 방향**:

```sql
-- teams INSERT 정책 수정: is_player() 조건 추가
CREATE POLICY teams_insert_player ON teams
  FOR INSERT WITH CHECK (
    created_by = auth.uid() AND is_player()  -- user는 팀 생성 불가
  );

-- tournament_team_applications INSERT 정책에 is_player() 추가
CREATE POLICY applications_insert_captain ON tournament_team_applications
  FOR INSERT WITH CHECK (
    is_team_manager_for_team(team_id)
    AND applied_by = auth.uid()
    AND is_player()  -- user는 신청 불가
  );
```

## 8. 권장 구조 최종 제안 (A안 vs B안)

### 최종 결론: **A안 채택** (role enum 확장)

#### 이유 요약

1. **의미 정확성**: `user`와 `player`는 실제로 다른 접근 권한을 가지는 개념. role enum이 이를 명확히 표현함.

2. **RLS 구현 단순성**: B안(`player_status='pending'`)은 모든 RLS 정책에서 `role = 'player' AND player_status = 'active'`처럼 두 조건을 조합해야 함. A안은 `role = 'player'` 하나로 표현 가능.

3. **기존 코드 영향이 예상보다 제한적**:
   - `isOperationRole()`: 변경 없음
   - 모든 admin 가드: 변경 없음
   - RLS의 `is_organizer()`, `is_manager()`: 변경 없음
   - 실질적으로 변경 필요한 곳: `Role` 타입 + `login/actions.ts` + `Sidebar` + 신규 RLS 정책

4. **기존 player 데이터 처리**: 마이그레이션에서 프로필 미완료 player를 `user`로 downgrade하는 SQL 한 줄로 처리 가능 (`display_name IS NULL OR phone IS NULL` 조건).

5. **본인인증 미구현 상태에서의 단계적 적용 가능**: 본인인증 없이도 `user → player` 승격을 "프로필 입력 완료"만으로 처리하는 방식으로 시작하고, 이후 본인인증 연동 후 조건 강화 가능.

## 11. 주요 리스크 및 주의사항

### 리스크 1: 기존 player 데이터 backfill 정책

A안 도입 시 "기존 player를 어떻게 처리할지" 결정 필요.

**옵션 A**: 기존 player 전체를 `player` 유지 (선수 등록 완료로 간주)
- 단점: 프로필 미입력 기존 사용자도 player로 유지되어 의미 혼탁

**옵션 B**: 프로필 미완료 기존 player를 `user`로 downgrade
```sql
UPDATE profiles SET role = 'user'
WHERE role = 'player' AND (display_name IS NULL OR phone IS NULL);
```
- 단점: 기존 player가 갑자기 기능을 잃을 수 있음 (공지 필요)

**권장**: 현재 서비스가 초기 단계이고 실제 사용자 수가 적다면 **옵션 B** 선택.  
사용자 수가 많다면 별도 공지 후 마이그레이션.

### 리스크 2: `user` 상태에서 team 생성 허용 여부

현재 RLS는 인증된 사용자면 팀 생성 가능. `user`도 팀을 만들 수 있게 할지 결정 필요.

**권장**: 팀 생성도 `player` 전용으로 제한. 이유: 팀 생성 → 대회 신청 흐름이 `player` 전용이므로, `user`가 팀을 만들어도 대회 신청을 못 하면 혼란.

### 리스크 3: `status = "empty"` 처리

`getUserWithRole()`에서 `status: "empty"`는 auth.users는 있으나 profiles 행이 없는 엣지 케이스.  
새 흐름에서도 이 상태는 `/onboarding/profile`로 유도해야 한다.

### 리스크 4: 본인인증 없이 phase별 적용

본인인증 연동은 외부 업체가 필요하므로 장기 작업. Phase 1-3을 먼저 적용하면:
- `user → player` 승격 조건을 "프로필 입력 완료"로 임시 설정 가능
- 이후 본인인증 완료 조건 추가 시 승격 기준만 강화하면 됨

### 리스크 5: TypeScript 타입 전파

`Role` 타입에 `"user"` 추가 시 TypeScript exhaustive check(`switch`문 등)에서 컴파일 오류 발생 가능.  
특히 `switch(role)` 구문이나 `role satisfies` 패턴 사용 코드를 전수 확인 필요.
