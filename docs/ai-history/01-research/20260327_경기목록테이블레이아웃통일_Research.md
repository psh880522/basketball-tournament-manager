
### 4.3 데이터 구조

```typescript
// 슬롯 그룹핑 타입
type ScheduleSlotCourtGroup = {

type ScheduleSlotDivisionGroup = {
  division: { id: string; name: string } | null;
```

```tsx

                `}
                draggable={slot.slot_type === "match"}
                onDragStart={...}
                onDragOver={...}
                onDrop={...}
              >
```

### 4.7 드래그앤드롭 방식

ScheduleSlotsBoard는 네이티브 HTML5 드래그 이벤트를 사용:
```tsx
// HTML5 기반 (dnd-kit 미사용)
draggable={slot.slot_type === "match"}
onDragStart={(e) => onDragStart(sectionId, slot.id)}
onDragOver={(e) => { e.preventDefault(); setDragOver(slot.id); }}
onDrop={(e) => onDrop(sectionId, slot.id)}
```

서버 액션:
- `swapSlotMatchAssignmentsAction` — 두 슬롯의 경기 교환
- `assignMatchToEmptySlotAction` — 빈 슬롯에 경기 배정

## 8. 추가 통찰 및 주의사항

### 8.1 현재 구조의 강점 (변경 불필요)
- 두 UI 모두 코트 → 디비전 → 테이블의 동일한 계층 구조
- Card 컴포넌트로 일관된 외형
- 포맷터 함수로 라벨 생성 로직 분리
- Next.js App Router 서버 컴포넌트 패턴 준수

### 8.2 변경 시 주의사항
1. **서버 컴포넌트**: `matches/page.tsx`는 서버 컴포넌트이므로 클라이언트 훅 사용 불가
2. **Filters 컴포넌트**: `Filters.tsx`는 URL 쿼리 기반으로 동작 — 레이아웃 변경 시 구조에 영향 없음
3. **StatusBadge**: 스케줄 UI에는 없는 요소 — 경기 목록에서만 유지
4. **table-fixed 추가**: `table-fixed` 적용 시 colgroup 너비가 실제 렌더링에 영향 — 텍스트 잘림 검토 필요

### 8.3 구현 순서 권장
1. ScheduleSlotsBoard.tsx 코드 정독 (기준 UI 확인)
2. matches/page.tsx 전체 읽기
3. 클래스명 차이 기반으로 A~F 항목 순차 적용
4. 브라우저에서 두 페이지 나란히 비교
