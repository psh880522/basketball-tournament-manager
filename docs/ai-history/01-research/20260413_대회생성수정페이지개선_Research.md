# 대회 생성·수정 페이지 개선 리서치

- **날짜**: 2026-04-13
- **기능명**: TournamentCreateEditImprovement
- **요구사항 요약**: 대회 생성 폼을 3단계 스텝 위저드로 개선하고, 생성 단계부터 디비전별 참가비·정원·신청기간을 입력할 수 있도록 한다. 편집 페이지는 탭 기반 UI로 재구조화하여 디비전 카드에 신청 현황을 표시하고 공개설정 탭을 분리한다.

---

## 요구사항

### 기능 개요

대회 신청·결제 플로우 개선(20260409) 이후 디비전에 `entry_fee`, `capacity`, `application_open_at`, `application_close_at` 필드가 추가되었다.
그러나 **대회 생성 폼**에서는 이 필드들을 입력받지 않아, 생성 후 편집 페이지에서 다시 설정해야 하는 이중 작업이 발생한다.
또한 편집 페이지는 세로 스크롤 섹션 구조로 되어 있어 관리 항목이 많아질수록 UX가 떨어진다.

---

### 구현 목표

- 대회 생성 폼을 **3단계 스텝 위저드**로 개선한다
- 생성 단계에서 디비전별 **참가비, 정원, 신청기간**을 입력할 수 있다
- 생성 서버 액션이 신규 디비전 필드를 DB에 저장한다
- 편집 페이지를 **탭 기반 UI**로 재구조화한다 (기본정보 / 디비전 / 코트 / 포스터 / 공개설정)
- 편집 디비전 카드에 **신청 현황** (확정/대기 팀수)을 표시한다
- 공개설정 탭에서 대회 상태(draft → open → closed)를 관리한다

---

### 상태 / 플로우 정의

#### 대회 생성 위저드 흐름

```
[1단계: 기본 정보]
  - 대회명 (required)
  - 장소 (optional)
  - 날짜 범위 (required)
  - 시작 시간 (optional)
  - 설명 (optional)
  - 포스터 업로드 (optional)
       ↓ 다음
[2단계: 디비전 설정]
  - 디비전 카드 목록 (추가/삭제)
  - 각 디비전 입력 항목:
    · 부문명 (required)
    · 조별 팀수 (required, ≥2)
    · 토너먼트 형식 (4강/8강/16강/없음)
    · 참가비 (required, ≥0, 0이면 무료)
    · 정원 (optional, ≥1)
    · 신청 시작일 (optional)
    · 신청 종료일 (optional, > 신청 시작일)
       ↓ 다음
[3단계: 운영 설정]
  - 코트 목록 (추가/삭제)
  - 최대 팀수 (optional)
       ↓ 생성
→ 생성 완료 후 /admin/tournaments/[id]/edit 로 리다이렉트
```

#### 편집 탭 구조

```
탭1: 기본정보  ← 대회명, 장소, 날짜, 시간, 설명
탭2: 디비전    ← 디비전 카드 목록 (신청 현황 포함) + 추가/편집/삭제
탭3: 코트      ← 코트 목록 + 추가/편집/삭제
탭4: 포스터    ← 포스터 업로드/삭제
탭5: 공개설정  ← 대회 상태 변경 (draft→open→closed), 최대 팀수
```

---

### 주요 규칙 및 제약

- 디비전 `capacity`는 현재 활성 신청 팀 수(payment_pending/paid_pending_approval/confirmed/waitlisted)보다 낮게 변경 불가 (기존 `divisions.ts` validateCapacityReduction 로직 유지)
- 디비전 삭제는 신청 기록이 존재하면 불가 (기존 `deleteDivision` 로직 유지)
- 대회 상태가 open/closed일 때 참가비·정원 변경 시 경고 메시지 표시 ("이미 신청한 팀이 있습니다.")
- 신청기간 종료일은 시작일보다 늦어야 함
- 스텝 이동 시 현재 단계 유효성 검사 후 다음 단계 이동 허용
- 대회 생성 시 디비전이 0개여도 생성 가능 (디비전은 선택 사항)
- 최대 팀수(`max_teams`)는 디비전별 `capacity` 합산과는 별개 필드 (독립적 운용)

---

### 이번 구현 범위에서 제외되는 것

- 디비전별 할인 정책 (discount_amount) 설정 UI는 제외
- 대기열 수동 관리 UI는 제외
- 대회 상태 변경 시 참가 팀에게 알림 발송은 제외 (notifications 테이블 준비는 되어 있으나 발송 로직 미구현)
- 스텝 위저드 중간 저장(draft 자동저장) 기능은 제외
- 모바일 최적화는 기존 수준 유지 (별도 반응형 개선 작업 아님)

## 3. 현재 구현 상태

### 대회 생성 폼 (`new/Form.tsx`) — 문제점

**현재 DivisionDraft 타입:**
```typescript
type DivisionDraft = {
  id: string;
  name: string;
  groupSize: number;
  tournamentSize: string;
  // ← entry_fee, capacity, application_open_at, application_close_at 없음
};
```

**현재 createTournamentAction divisions INSERT:**
```typescript
const divisionRows = divisions.map((d, i) => ({
  tournament_id: tournamentId,
  name: d.name,
  group_size: d.groupSize,
  tournament_size: d.tournamentSize ? Number(d.tournamentSize) : null,
  sort_order: i,
  // ← entry_fee, capacity, application dates 전달 안 함 → DB default(0, null, null, null)
}));
```

**결과**: 생성된 모든 디비전의 `entry_fee = 0`, `capacity = null`, 신청기간 없음 → 편집 페이지에서 다시 입력 필요

### 대회 편집 폼 (`edit/Form.tsx`) — 현황

- `AddDivisionForm`: `entryFee`, `capacity`, `applicationOpenAt`, `applicationCloseAt` 이미 입력 필드 존재 ✅
- `DivisionItem` (인라인 편집): 동일 필드 존재 ✅
- `SettingsSection`: max_teams, DivisionsSection, CourtsSection을 하나의 섹션에 세로 배치 → UI가 복잡
- 디비전 카드 표시 정보: `그룹: X · 토너먼트: Y강 · 참가비: Z원 · 정원: W` (신청 현황 없음)
- 대회 상태 변경: 편집 페이지와 별도 관리 페이지에 분산되어 있음

### 편집 서버 액션 (`edit/actions.ts`) — 현황

- `createDivisionAction`: entry_fee, capacity, applicationOpenAt/CloseAt 처리 ✅
- `updateDivisionAction`: 동일 ✅
- `deleteDivisionAction`: divisions.ts의 검증 로직 호출 ✅
- **대회 상태 변경 액션**: 별도 파일(`app/(app)/admin/page.tsx` 또는 `admin/tournaments/[id]/actions.ts`)에 위치

## 4. 재사용 가능한 패턴

### 1. DivisionDraft → DivisionRow 매핑 패턴
편집 폼의 `AddDivisionForm`이 이미 완성된 필드 구조를 가짐. 생성 폼의 `DivisionDraft` 타입을 동일하게 확장하면 로직 재사용 가능.

### 2. react-day-picker DateRange
기본 정보 섹션의 날짜 범위 선택 컴포넌트 (`DayPicker`) 이미 구현됨. 신청기간 날짜 선택에 동일 패턴 적용 가능.

### 3. useTransition + 서버 액션 패턴
현재 모든 편집 섹션이 `useTransition` + 서버 액션 패턴으로 통일되어 있음. 탭 기반 UI로 변경해도 동일 패턴 유지.

### 4. 인라인 편집 (isEditing 토글)
DivisionItem의 `isEditing` state 토글 패턴. 탭 내 카드 편집에도 그대로 활용 가능.

### 5. `FieldHint` / `Button` / `Card` UI 컴포넌트
`components/ui/` 하위에 공통 컴포넌트 존재. 스텝 위저드 UI에도 재사용.

### 1. 생성 폼 DivisionDraft 초기값
```typescript
// entryFee 기본값을 0으로 설정해야 기존 "무료" 동작과 일치
const initialDivision = (): DivisionDraft => ({
  id: crypto.randomUUID(),
  name: "",
  groupSize: 4,
  tournamentSize: "",
  entryFee: 0,
  capacity: "",
  applicationOpenAt: "",
  applicationCloseAt: "",
});
```

### 2. capacity: null vs 빈 문자열 처리
- 폼 input은 string이지만 DB는 `integer | null`
- 저장 시: `capacity: d.capacity ? Number(d.capacity) : null`
- 유효성 검사: `capacity`가 입력된 경우 `Number(d.capacity) >= 1`

### 3. application_open_at / close_at 시간대 처리
- `datetime-local` input은 로컬 시간 반환
- 기존 `schedule_start_at` 처리 패턴 참고: KST offset(`+09:00`) 붙여서 UTC 변환 후 저장
- 편집 폼에서도 이미 동일 패턴 사용됨 (`parseKSTDatetime` 유틸 여부 확인)

### 4. 스텝 위저드 — 뒤로 가기 처리
- 스텝 1로 돌아갈 때 기입한 디비전/코트 데이터 유지
- 전체 폼 상태를 최상위 컴포넌트에서 관리 (스텝별로 state 분리하지 않음)

### 5. 편집 폼 탭 활성 상태
- URL 쿼리 파라미터(`?tab=divisions`)로 탭 상태 유지하면 새로고침 후에도 유지 가능
- 단순 `useState`로 관리해도 기능상 문제 없음 (팀 결정 사항)

### 6. 대회 상태 변경 시 조건 체크
- open 전환 가능 조건: 디비전이 1개 이상 존재
- closed 전환 가능 조건: 현재 플로우에서는 별도 제약 없음 (운영자 판단)
- 기존 `changeTournamentStatusAction` 동작 및 정책 확인 필요

### 7. 신청 현황 데이터 조회 방법
- 편집 페이지 서버 컴포넌트(`page.tsx`)에서 디비전별 신청 현황을 함께 로드하거나
- 클라이언트에서 별도 fetch하는 방식 중 선택
- 편집 페이지는 server component이므로 데이터 로드를 page.tsx에서 처리하는 것이 패턴상 일관적

### 8. Form.tsx 파일 크기
- 현재 `edit/Form.tsx`는 DivisionsSection, CourtsSection, SettingsSection, PosterSection, TournamentEditForm 등 모두 포함하여 상당히 큰 파일
- 탭 기반으로 분리 시 파일을 나누는 것이 유지보수에 유리

## 9. 결론

이번 개선의 핵심은 **생성 폼과 편집 폼 간의 기능 갭 해소**다.
20260409 대회신청·결제 플로우 개선으로 디비전이 참가비/정원/신청기간을 가지게 됐지만,
생성 폼에서는 여전히 이 필드들을 입력받지 않아 운영자가 대회 생성 후 다시 편집 페이지에서 디비전을 하나씩 수정해야 하는 불편함이 있다.

**Phase 1** (생성 폼 필드 추가 + actions.ts 수정)은 DB 변경 없이 프론트엔드와 서버 액션 수정만으로 완료 가능하며, 즉시 실질적인 UX 개선 효과가 있다.

**Phase 2~4**는 UI/UX 완성도를 높이는 작업으로, 순차적으로 진행하거나 Phase 1 이후 우선순위를 재검토해 진행할 수 있다.
