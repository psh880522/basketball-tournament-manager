## 3. 현재 인증 관련 파일 및 구조

### 3-1. 회원가입 (`app/(auth)/signup/`)

**`actions.ts`**:
```typescript
export async function signUpWithPassword(input: { email, password })
  → SignUpActionResult { ok: true; requiresEmailConfirmation: boolean } | { ok: false; error: string }
```
- `lib/api/auth.ts`의 `signUpWithPassword()` 호출 → Supabase `auth.signUp()`
- `data.session`이 null이면 이메일 인증 필요 → `requiresEmailConfirmation: true` 반환
- 자동 인증 환경이면 즉시 `redirect("/dashboard")`
- 에러 메시지 한국어 번역 맵 포함
- **가입 후 프로필 입력 유도 없음** (곧바로 `/dashboard` 이동)

**DB 연동**:
- `auth.signUp()` 성공 시 `on_auth_user_created` 트리거 실행
- `public.profiles`에 `(id)` 자동 INSERT (role 기본값 `'player'`)
- profiles 테이블: `id`, `role`, `created_at` — **이름, 연락처 등 없음**

### 3-2. 로그인 (`app/(auth)/login/`)

**`page.tsx`**:
- 이미 로그인된 사용자 감지: `getUserWithRole()` 호출
- `status === "ready"` 이면 role에 따라 `/admin` 또는 `/dashboard`로 redirect

**`actions.ts`**:
```typescript
export async function signInWithPassword(input: { email, password })
  → ActionResult
```
- `lib/api/auth.ts`의 `signInWithPassword()` 호출 → Supabase `auth.signInWithPassword()`
- 성공 후 `getCurrentUserRole()` 조회
- `isOperationRole(role)` → `/admin`, 아니면 `/dashboard`
- **프로필 완료 여부 / 본인인증 완료 여부 분기 없음**
- **`returnTo` 쿼리 파라미터 복귀 처리 없음**

### 3-3. 핵심 인증 유틸 (`src/lib/auth/roles.ts`)

```typescript

type UserWithRoleResult = {

  // status "empty": auth.users는 있으나 profiles 행 없음 (이론적 엣지 케이스)

export function isOperationRole(role: Role | null): boolean

```

### 3-4. 가드 함수 (`src/lib/auth/guards.ts`)

```typescript
requireOrganizer()       → ActionResult  // organizer만 허용
requireOperationRole()   → ActionResult  // organizer + manager 허용
```

### 3-5. 라우팅 보호 패턴 (현재 방식)


## 4. DB 구조 및 관련 테이블

### 4-1. `public.profiles` (핵심 테이블)

```sql
CREATE TABLE public.profiles (
  id         uuid PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  role       public.app_role NOT NULL DEFAULT 'player',
  created_at timestamptz NOT NULL DEFAULT now()
);
```

- `app_role` enum: `'organizer'`, `'manager'`, `'player'` (0205 마이그레이션으로 3종 확정)
- **이름, 연락처, 생년월일, 전화번호 등 일체 없음**
- 신규 가입 시 `on_auth_user_created` 트리거로 자동 생성 (`role='player'` 기본값)

### 4-2. `public.user_profiles` (VIEW)

```sql
CREATE VIEW public.user_profiles WITH (security_invoker=true) AS
  SELECT p.id, p.role, p.created_at, u.email
  FROM public.profiles p
  JOIN auth.users u ON u.id = p.id;
```

- `security_invoker=true` → 호출자 RLS 적용 (organizer: 전체, 일반: 본인만)
- `lib/api/users.ts`의 `listUsersWithRole()` 에서만 사용 (admin/users 페이지)

### 4-3. `public.teams`

```sql
CREATE TABLE teams (
  id              uuid PRIMARY KEY,
  tournament_id   uuid NULLABLE,          -- 독립 팀 허용 (대회 참가 전)
  team_name       text NOT NULL,
  captain_user_id uuid NULLABLE,          -- 레거시, 미사용
  contact         text,
  status          text,                   -- 'pending', 'approved', 'rejected'
  is_dummy        boolean,
  created_by      uuid NOT NULL REFERENCES auth.users(id),
  created_at, updated_at timestamptz

```

### 4-4. `public.team_members`

```sql
CREATE TABLE team_members (
  team_id      uuid REFERENCES teams(id) ON DELETE CASCADE,
  user_id      uuid REFERENCES auth.users(id) ON DELETE CASCADE,
  role_in_team text,                      -- 'captain' 또는 'player'

```

- 팀 captain = `role_in_team='captain'` (0207 마이그레이션: 'manager' → 'captain' 변경)

## 9. 핵심 구현 질문 분석

### Q1. 상태 분기 위치: middleware vs 서버 컴포넌트

**현재 방식**: middleware.ts 없음 → 서버 컴포넌트에서 redirect()
**권장 방향**:
- 단순 "로그인 여부" 보호는 서버 컴포넌트 방식 유지 (일관성)
- 프로필 미완료 / 인증 미완료 유도는 서버 컴포넌트에서 처리
- middleware를 새로 추가할 경우 Supabase SSR 세션 새로고침 로직 함께 구현 필요 (공식 권장)

**미들웨어 추가 시 고려사항**:
```typescript
// middleware.ts 추가 시 반드시 세션 갱신 포함
export async function middleware(request: NextRequest) {
  const { supabase, response } = createServerClient(...)
  await supabase.auth.getUser() // 세션 갱신
  return response
}
```

### Q2. `is_profile_completed` 판단 기준

**옵션 A**: DB 함수로 계산
```sql
CREATE FUNCTION is_profile_completed(user_id uuid) RETURNS boolean AS $$
  SELECT display_name IS NOT NULL AND phone IS NOT NULL
  FROM profiles WHERE id = user_id
$$ LANGUAGE sql SECURITY DEFINER;
```

**옵션 B**: TypeScript 함수로 계산
```typescript
function isProfileCompleted(profile: Profile): boolean {
  return !!profile.display_name && !!profile.phone;
}
```

**권장**: TypeScript 함수 방식이 프로젝트 일관성에 맞음 (기존 `isOperationRole()` 패턴)

### Q3. 본인인증 서버 검증 구조

**핵심 원칙**: 프론트엔드가 전달하는 성공 결과를 신뢰하지 않고, 외부 인증 업체의 결과를 서버에서 직접 검증 후 DB 업데이트

```
[클라이언트] → 인증 UI 시작
     ↓
[외부 업체] ← 서버 콜백 URL 설정 (Server Action or API Route)
     ↓
[Server Action] → provider_tx_id로 외부 업체 API 재검증
     ↓
[DB] identity_verifications.status = 'success'
     profiles.is_identity_verified = true
     profiles.identity_verified_at = now()
```

**주의**: `profiles.is_identity_verified` 업데이트는 반드시 server-side에서만 가능하도록 RLS 설정 필요 (사용자 본인도 직접 UPDATE 불가)

### Q4. 로그인 후 상태 기반 리다이렉트 흐름

**현재** (`app/(auth)/login/actions.ts:48`):
```typescript
const nextPath = isOperationRole(roleResult.role) ? "/admin" : "/dashboard";
redirect(nextPath);
```

**변경 후 (권장 방향)**:
```typescript
// 1. 역할 분기 (현재와 동일)
if (isOperationRole(role)) redirect("/admin");

// 2. 프로필 완료 여부 확인
const profile = await getProfileById(userId);
if (!isProfileCompleted(profile)) redirect("/onboarding/profile");

// 3. 본인인증 완료 여부 확인
if (!profile.is_identity_verified) redirect("/identity");

// 4. 최종 대시보드
redirect("/dashboard");
```

**`returnTo` 처리**:
- 현재 미구현. 향후 구현 시 `/login?returnTo=/tournament/xxx/apply` 패턴 권장
- Server Action에서 `headers().get("referer")` 또는 폼 hidden input으로 전달

### Q5. 대회 참가 차단 위치

**현재**: `app/(app)/tournament/[id]/apply/actions.ts`에서 captain 여부, 대회 status만 검증

**추가 필요**:

## 11. 구현 시 주의사항 및 충돌 가능성

### 주의사항 1: profiles 테이블 ALTER 시 RLS 재검토
- `profiles` 컬럼 추가 시 기존 `profiles_update_own` RLS 정책이 모든 컬럼 UPDATE를 허용함
- `is_identity_verified`, `identity_verified_at`는 사용자가 직접 수정 불가하도록 추가 정책 필요
- 권장: `SECURITY DEFINER` 함수를 통해서만 인증 상태 업데이트

### 주의사항 2: on_auth_user_created 트리거 충돌 방지
- 현재 트리거: `INSERT INTO profiles (id) VALUES (new.id) ON CONFLICT DO NOTHING`
- profiles 컬럼 추가 시 `DEFAULT NULL` 설정 필수 (트리거가 id만 삽입하므로)

### 주의사항 3: status = "empty" 처리
- `getUserWithRole()`의 `status: "empty"`: auth.users는 있으나 profiles 행 없는 상태
- 현재 dashboard에서 에러 메시지 표시 후 아무 처리 없음
- 프로필 입력 플로우 추가 시 `empty` 상태도 `/onboarding/profile`로 유도해야 함

### 주의사항 4: 미들웨어 부재
- 현재 방식(서버 컴포넌트 redirect)은 각 페이지마다 중복 검증 필요
- 인증 로직이 분산되어 있어 프로필/인증 체크 추가 시 여러 파일을 동시에 수정해야 함
- 중앙화가 필요하다면 `middleware.ts` 도입 고려 (단, Supabase SSR 세션 갱신 로직 필수)

### 주의사항 5: team_members captain 역할 명칭
- `role_in_team='captain'` ≠ `profiles.role='manager'`
- 혼동 주의: 팀 내 역할(captain/player)과 앱 전역 역할(organizer/manager/player)은 별개

### 주의사항 6: 본인인증 상태를 profiles vs 별도 테이블로 저장
- `profiles.is_identity_verified boolean` (단순): 현재 인증 여부만 필요하면 충분
- `identity_verifications` 별도 테이블: 인증 히스토리, 재인증, 감사 로그 필요 시
- 향후 인증 업체 변경 또는 재인증 정책 확장 가능성이 있으므로 **별도 테이블 + profiles에 캐시 컬럼** 권장

## 13. 결론 및 권장 구현 방향

### 최소 변경으로 구현 가능한 방향

1. **DB**: `public.profiles`에 `display_name`, `phone`, `is_identity_verified` 컬럼 추가 (단계적 확장 가능). `identity_verifications` 별도 테이블은 실제 인증 업체 연동 전 단계에서 준비.

2. **라우팅 보호**: middleware 도입 없이 기존 서버 컴포넌트 패턴 유지. `getUserWithRole()` 확장보다 `getProfileWithStatus()` 같은 새 헬퍼 함수 추가 권장.

3. **로그인 분기**: `app/(auth)/login/actions.ts`의 `signInWithPassword()` 함수 내 분기 로직 추가가 가장 적은 변경으로 구현 가능. `isProfileCompleted()`, `isIdentityVerified()` 헬퍼 함수 TypeScript로 구현.

4. **대회 참가 차단**: `app/(app)/tournament/[id]/apply/actions.ts`에 `is_identity_verified` 검증 추가 + `page.tsx`에서 UI 차단/안내. RLS는 선택적으로 이중 보호.

5. **프로필 입력 폼**: 기존 `app/(auth)/signup/Form.tsx` 패턴 재사용 (useState + useTransition + Server Action 구조).

### 구조 분리 권장 사항

```
lib/api/profiles.ts     # getProfile(), updateProfile() — 신규
lib/api/identity.ts     # getIdentityStatus(), startVerification(), confirmVerification() — 신규
src/lib/auth/roles.ts   # isProfileCompleted(), isIdentityVerified() 함수 추가 또는 별도 파일
```

현재 프로젝트는 `lib/api/*.ts` + `src/lib/auth/*.ts` 구조가 명확하므로 이 패턴을 그대로 따르는 것이 최선.
