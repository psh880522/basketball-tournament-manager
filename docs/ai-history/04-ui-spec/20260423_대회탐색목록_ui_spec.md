# 대회 탐색/목록 UI 설계

---

## 1. 리서치 반영 요약

### 기존 UI 패턴에서 재사용할 것

| 항목 | 파일 | 재사용 방식 |
|---|---|---|
| `Card` 컴포넌트 | `components/ui/Card.tsx` | 대회 카드 컨테이너 (variant: default) |
| `Badge` 컴포넌트 | `components/ui/Badge.tsx` | 상태 뱃지, 신청 상태 뱃지 |
| `Button` 컴포넌트 | `components/ui/Button.tsx` | CTA 버튼 (primary/secondary/ghost) |
| `EmptyState` 컴포넌트 | `components/ui/EmptyState.tsx` | 빈 목록 상태 |
| `DragScroll` 컴포넌트 | `components/ui/DragScroll.tsx` | 모바일 탭 가로 스크롤 |
| Suspense + 스켈레톤 패턴 | `app/(public)/page.tsx` | 탭 전환/필터 변경 시 로딩 |
| URL searchParams 필터 패턴 | `app/(app)/admin/tournaments/[id]/matches/Filters.tsx` | 탭/필터 상태 URL 반영 |
| `listAllMyTeamApplications()` | `lib/api/applications.ts` | 카드별 내 신청 상태 매핑 |
| 페이지 레이아웃 | `app/(app)/dashboard/page.tsx` | `min-h-screen bg-page px-4 py-8`, max-w-5xl |

### 새로 설계할 것

- `TournamentCard` — 기존 단순 카드를 대체하는 확장 카드 (부문 칩, 참가비, 정원, 신청상태)
- `FilterTabBar` — 상태별 탭 (전체/모집중/진행중/종료/내 신청)
- `SearchBar` — 키워드 검색 인풋 (debounce)
- `FilterPanel` — 데스크톱 사이드 필터 또는 모바일 바텀시트
- `TournamentListSkeleton` — 목록 로딩 스켈레톤

### 기존 패턴에서 의도적으로 변경할 것

| 변경 사항 | 이유 |
|---|---|
| 고정 3섹션 → 탭 기반 단일 목록 | 상태별 탐색 UX 개선, 스크롤 없이 원하는 상태만 빠르게 조회 |
| 리스트형 카드 → 정보 밀도 높은 카드 | 참가비/정원 등 신청 판단에 필요한 정보를 목록에서 바로 제공 |
| `closed` 레이블 "모집마감" → "진행중" | 탐색 화면 맥락에서는 "대회가 진행 중"이라는 정보가 핵심 (상세 페이지는 유지) |
| `force-dynamic` 유지 + Suspense 추가 | 탭 전환 시 로딩 피드백 제공 |

## 2. 기존 UI 패턴 분석

### 2-1. 사용 중인 UI 컴포넌트

| 컴포넌트 | 주요 props / variant |
|---|---|
| `Card` | `variant: default(bg-white shadow-sm) / highlight(bg-[#FFF5EC]) / muted(bg-[#f0f0f0])`, `rounded-xl p-5` |
| `Badge` | `variant: default(slate) / success(emerald) / warning(amber) / danger(rose) / info(sky) / live(#FF6B00/10)` |
| `Button` | `variant: primary(그래디언트 주황) / secondary(border 주황) / ghost(slate)`, `rounded-[4px]` |
| `EmptyState` | 빈 상태 메시지 컴포넌트 |
| `DragScroll` | 가로 스크롤 래퍼, `-mx-4 px-4 [scrollbar-width:none]` 패턴 |
| `Input` | `label / hint / error / rightElement` 지원 |

외부 UI 라이브러리 없음 — 모든 컴포넌트가 Tailwind CSS 기반으로 직접 구현되어 있음.

### 2-2. 레이아웃 패턴

```
<main className="min-h-screen bg-page px-4 py-8">
  <div className="mx-auto max-w-5xl flex flex-col gap-6">
    <header>...</header>
    <section>...</section>
  </div>
</main>
```

- 최대 너비: 모바일 위주 페이지는 `max-w-2xl`, 대시보드/상세는 `max-w-5xl`
- 대회 목록은 `max-w-2xl` 유지 (현재 패턴) → 모바일 최적화
- 배경색: `bg-page` (= `bg-gray-50 / #f9fafb`) — CSS 변수로 정의

### 2-3. 디자인 토큰

| 항목 | 값 |
|---|---|
| 브랜드 주황 | `#FF6B00` |
| 브랜드 네이비 | `#1A237E` (= `bg-secondary`) |
| 페이지 배경 | `bg-page` = `#f9fafb` |
| 카드 배경 | `bg-white` |
| 카드 하이라이트 | `bg-[#FFF5EC]` |
| 폰트 (헤딩) | `font-space-grotesk` |
| 폰트 (본문) | `Inter` |
| 섹션 헤딩 | `text-2xl font-semibold text-slate-800` |
| 서브 텍스트 | `text-sm text-slate-500` |
| 라벨 (small) | `text-[11px] font-semibold uppercase tracking-wide text-slate-400` |
| 카드 내부 텍스트 | `text-sm text-slate-700` |
| 화살표 아이콘 | `h-4 w-4 text-slate-400` (lucide 또는 inline SVG) |
| 라운드 | 카드 `rounded-xl`, 버튼 `rounded-[4px]`, 뱃지 `rounded-full` |
| 그림자 | 카드 `shadow-sm` |

**대회 상태 → Badge variant 매핑 (목록 페이지 기준)**

| DB status | 레이블 | Badge variant |
|---|---|---|
| `open` | 모집중 | `success` (emerald) |
| `closed` | 진행중 | `info` (sky) — 기존 상세 페이지의 danger(rose)와 다름, 탐색 맥락에 맞게 조정 |
| `finished` | 종료 | `default` (slate) |
| `draft` | 준비중 | `default` (slate) |

**신청 상태 → Badge variant 매핑** (기존 패턴 그대로 재사용)

| ApplicationStatus | 레이블 | Badge variant |
|---|---|---|
| `payment_pending` | 입금 대기 | `warning` |
| `paid_pending_approval` | 승인 대기 | `info` |
| `confirmed` | 참가 확정 | `success` |
| `waitlisted` | 대기 중 | `warning` |
| `expired` | 만료됨 | `default` |
| `cancelled` | 취소됨 | `danger` |

### 4-1. 대회 탐색/목록 페이지 (`/tournaments`)

#### 목적

카드에서 기본 정보와 내 팀의 신청 여부를 한눈에 확인한다.

#### ⑤ TournamentCard — 확장 카드 설계

**카드 레이아웃:**

```
┌─────────────────────────────────────────────────┐
│ [모집중]                          [참가 확정 ✓]  │  ← 상태뱃지 / 내 신청상태뱃지
│                                                 │
│  서울 오픈 3X3 농구 대회                          │  ← 대회명 (font-medium text-base)
│  2026.05.10 ~ 05.11 · 잠실 실내 체육관           │  ← 날짜·장소 (text-sm text-slate-500)
│                                                 │
│  ─────────────────────────────────────────────  │
│                                                 │
│  [A부] [B부] [오픈부]                            │  ← 부문 칩 (최대 3개, 초과시 +N)
│  참가비  30,000 ~ 50,000원                       │  ← 참가비 범위
│  정원    24 / 30팀                               │  ← 전체 집계 (capacity 있을 때만)
│                                                 │
│                  [상세보기]  [참가 신청 →]         │  ← CTA
└─────────────────────────────────────────────────┘
```

**카드 구현 스펙:**

```tsx
<Link href={`/tournament/${tournament.id}`}>

    {/* Row 1: 상태 뱃지 + 내 신청 상태 */}

            </span>
          ))}
          {tournament.divisions.length > 3 && (
            <span className="text-xs text-slate-400">+{tournament.divisions.length - 3}</span>
          )}
        </div>

        {/* 참가비 + 정원 */}
        <div className="flex flex-wrap gap-x-4 gap-y-1">

          {totalCapacity !== null && (

    {/* Row 4: CTA 버튼 */}

```

**entryFeeRange 계산 로직:**

```typescript
const fees = divisions.map(d => d.entry_fee)
const minFee = Math.min(...fees)
const maxFee = Math.max(...fees)
const entryFeeRange = minFee === 0 && maxFee === 0
  ? '무료'
  : minFee === maxFee
    ? `${minFee.toLocaleString()}원`
    : `${minFee.toLocaleString()} ~ ${maxFee.toLocaleString()}원`
```

**totalCapacity / confirmedCount 계산 로직:**

```typescript
// 모든 부문 중 하나라도 capacity === null이면 "무제한" 포함이므로 전체 정원 표시 생략
const allHaveCapacity = divisions.every(d => d.capacity !== null)
const totalCapacity = allHaveCapacity
  ? divisions.reduce((sum, d) => sum + (d.capacity ?? 0), 0)
  : null
const confirmedCount = divisions.reduce((sum, d) => sum + d.confirmed_count, 0)
const isNearFull = totalCapacity !== null && confirmedCount / totalCapacity >= 0.8
```

**CTA 버튼 조건 분기:**

| 조건 | CTA |
|---|---|
| `status !== 'open'` 또는 비인증 | [상세보기] 만 표시 |
| `status === 'open'` + 내 신청 없음 | [상세보기] [참가 신청 →] |
| `status === 'open'` + `payment_pending` | [상세보기] [입금 대기 →] |
| `status === 'open'` + `confirmed` / `waitlisted` | [상세보기] [내 신청 보기] |
| `status === 'open'` + `cancelled` / `expired` | [상세보기] [다시 신청] |

#### 빈 상태 (EmptyState)

| 탭/상황 | 메시지 |
|---|---|
| `all` 탭 + 검색 결과 없음 | "검색 결과가 없습니다." |
| `open` 탭 | "현재 모집 중인 대회가 없습니다." |
| `closed` 탭 | "현재 진행 중인 대회가 없습니다." |
| `finished` 탭 | "종료된 대회가 없습니다." |
| `mine` 탭 | "신청한 대회가 없습니다. 모집 중인 대회를 찾아보세요." (대회 탐색 버튼 링크 포함) |

## 8. 구현 시 주의사항

### 8-1. 카드 클릭 vs 버튼 클릭 이벤트 충돌

카드 전체 영역이 `<Link href="/tournament/[id]">`로 감싸질 경우, 내부의 `[참가 신청]` 버튼 클릭 시 상세 페이지가 아닌 `/apply`로 이동해야 한다.

```tsx
// 해결 방법: 버튼을 별도 Link로 처리 + 이벤트 전파 차단
<Link href={`/tournament/${id}/apply`} onClick={(e) => e.stopPropagation()}>
  <Button variant="primary" size="sm">참가 신청</Button>
</Link>
```

### 8-2. N+1 쿼리 방지

`getTournamentList()`에서 divisions 정보를 포함해야 하는데, 이를 개별 쿼리로 처리하면 N+1 발생.

```typescript
// Supabase 관계형 select로 한 번에 조회
const { data } = await supabase
  .from('tournaments')
  .select(`
    id, name, location, start_date, end_date, status, poster_url,
    divisions(id, name, entry_fee, capacity)
  `)
  .in('status', statusFilter)
  ...

// confirmed_count는 별도 집계 쿼리로 처리
// 또는 초기 MVP에서는 confirmed_count 표시 생략 (capacity만 표시)
```

**MVP 권장**: `confirmed_count` 표시를 초기 버전에서는 생략. 정원(`capacity`)만 표시해도 신청 판단에 충분.

### 8-3. "내 신청" 탭 데이터 처리

`mine` 탭은 `listAllMyTeamApplications()`에서 이미 로드된 데이터를 필터링해서 사용.  
별도 API 호출 없이 `myApplicationMap`에서 tournamentId 목록을 추출 후 `getTournamentList()`에 id 필터로 전달.

```typescript
if (tab === 'mine') {
  const myTournamentIds = [...myApplicationMap.keys()]
  const tournaments = await getTournamentListByIds(myTournamentIds)
}
```

또는 `listAllMyTeamApplications()` 결과에 tournament 정보가 포함되어 있으면 별도 조회 없이 직접 사용 가능. (현재 `MyApplicationRow`에 `tournament_name`, `tournament_status` 포함 여부 확인 필요)

### 8-4. 바텀시트 체크사항

- 스크롤 잠금: 바텀시트 오픈 시 body 스크롤 방지 (`document.body.style.overflow = 'hidden'`)
- 접근성: `role="dialog"`, `aria-modal="true"`, 포커스 트랩 처리
- iOS 안전 영역: `pb-safe` 또는 `padding-bottom: env(safe-area-inset-bottom)` 처리

### 8-5. 반응형 처리 우선순위

1. 모바일 (< 768px) — 바텀시트 필터, 탭 가로 스크롤, 1열 카드
2. 데스크톱 (≥ 768px) — 인라인 필터, 고정 탭바, 최대 너비 제한

### 8-6. 접근성

- 활성 탭에 `aria-selected="true"`, 비활성에 `aria-selected="false"`
- `FilterBottomSheet`에 `role="dialog"`, `aria-labelledby="filter-title"`
- 검색 인풋에 `aria-label="대회 검색"` 또는 시각적으로 숨긴 `<label>`
- 빈 상태 메시지에 `role="status"`

### 8-7. `application_open_at` / `application_close_at` 제거 대응

마이그레이션 `0240_drop_division_application_dates.sql`로 해당 컬럼이 drop됨.  
기존 `DivisionRow` 타입이나 `getDivisionsByTournament()` 반환 타입에 이 컬럼이 남아 있다면 제거 필요.  
신청 마감 임박 표시는 구현하지 않음.

## 9. 결론 및 플랜 단계 전달 메모

### 설계 요약

이번 UI는 `/tournaments` 페이지를 **탭 기반 탐색 + 확장 카드 + 필터** 구조로 전면 개편한다.  
기존 서버 컴포넌트 + Suspense 패턴을 유지하며, 필터 상태는 URL searchParams로 관리한다.  
모바일은 바텀시트 필터로, 데스크톱은 인라인 필터로 처리한다.

### 플랜 단계로 전달할 핵심 결정 사항

1. **MVP에서 `confirmed_count` 생략**: 정원 카운트 표시는 `capacity`만 표시 (`N팀 정원`)
2. **탭 전환 방식**: URL searchParams (`router.push`) — 서버 재렌더링 방식
3. **검색 방식**: 클라이언트 debounce + `router.replace` → 서버 `.ilike()` 쿼리
4. **바텀시트**: Tailwind CSS `fixed + translate-y` 직접 구현 (외부 라이브러리 없음)
5. **카드 클릭 처리**: 전체 카드는 상세 페이지 링크, 내부 버튼은 별도 Link + stopPropagation
6. **`closed` 레이블**: 목록에서는 "진행중" 사용 (상세 페이지 "모집마감"과 의도적으로 다름)
7. **내 신청 탭 데이터**: `listAllMyTeamApplications()` 결과에서 필터링 (추가 쿼리 없음)
