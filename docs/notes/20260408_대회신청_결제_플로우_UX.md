# 대회 신청 ~ 결제(MVP) UX 정리
> 기준: 계좌이체 + 운영자 수동 입금 확인 + 디비전별 참가비 + 정원/대기열 지원

## 목적
이 문서는 **유저(팀장/신청자)** 와 **운영자** 관점에서  
대회 신청부터 참가 확정까지의 UX 흐름을 정리한 참고 문서다.

기술 구현보다는 아래에 초점을 둔다.

- 사용자가 어떤 화면/문구/상태를 경험해야 하는가
- 운영자가 어떤 화면/액션/확인 흐름을 가져야 하는가
- 정원, 대기열, 입금대기, 입금확인 같은 상태가 UX상 어떻게 보여야 하는가

---

# 1. UX 설계 기본 원칙

## 1-1. 사용자는 항상 “지금 내 상태가 무엇인지” 바로 이해할 수 있어야 한다
사용자 입장에서 가장 중요한 것은 다음 3가지다.

- 내가 신청은 된 건지
- 자리는 확보된 건지
- 입금만 하면 되는 건지
- 아니면 대기열인지
- 이미 참가 확정인지

즉, 상태명 자체보다도 **행동 가능한 안내**가 중요하다.

예:
- 결제 대기
- 입금 확인 중
- 참가 확정
- 대기열 3번
- 입금 기한 만료

---

## 1-2. 운영자는 “무엇을 지금 처리해야 하는지” 우선순위가 보여야 한다
운영자 입장에서 중요한 것은 모든 신청을 그냥 나열하는 것이 아니라  
**당장 처리할 건**이 먼저 보여야 한다는 점이다.

운영자가 가장 먼저 보고 싶은 항목:
- 입금 확인 대기 건
- 곧 입금 기한이 만료되는 건
- 대기열이 있는 디비전
- 정원이 거의 다 찬 디비전
- 증원 시 바로 승격 가능한 대기팀

---

## 1-3. 상태는 내부적으로 복잡해도 화면에서는 이해하기 쉽게 보여야 한다
예를 들어 내부 상태가 `paid_pending_approval` 이어도  
사용자에게는 다음처럼 보이는 것이 더 좋다.

- “입금 확인 중”
- “운영자가 입금을 확인하고 있어요”
- “확인 후 참가가 확정됩니다”

즉, **DB 상태명 ≠ 사용자 노출 문구** 로 생각하는 것이 좋다.

---

# 2. 유저 UX 정리

# 2-1. 대회 상세 / 신청 시작 화면

## 목적
사용자가 신청 전에 아래를 빠르게 이해해야 한다.

- 어떤 디비전에 신청 가능한지
- 참가비가 얼마인지
- 정원이 얼마나 남았는지
- 지금 신청 가능한지 / 대기열인지
- 신청 후 어떤 절차가 이어지는지

---

## 화면에 보여야 하는 정보
각 디비전 카드/리스트에 최소 표시:

- 디비전명
- 참가비
- 모집 정원
- 현재 상태
- 신청 가능 기간
- 참가 조건 요약

예시 상태 문구:
- 신청 가능
- 마감 임박
- 대기 신청 가능
- 신청 마감
- 모집 종료

---

## UX 포인트
### 1) 참가비는 명확하게
- `참가비 150,000원`
- 무료면 `참가비 무료`

### 2) 상태는 행동 중심으로
- `지금 신청 가능`
- `정원 마감, 대기 신청 가능`
- `신청 마감`

### 3) 신청 이후 절차를 미리 안내
예:
- “신청 후 계좌이체를 완료하면 운영자 확인 후 참가가 확정됩니다.”
- “자리가 없는 경우 대기열로 접수됩니다.”

이 문구가 있으면 사용자 혼란이 줄어든다.

---

# 2-2. 신청서 작성 UX

## 목적
사용자가 큰 고민 없이 신청을 마칠 수 있어야 한다.

입력 과정에서 가장 중요한 것은:
- 어떤 팀으로 신청하는지
- 어떤 디비전에 신청하는지
- 참가비가 얼마인지
- 신청 후 무엇을 해야 하는지

---

## 화면 구성 추천
### 상단
- 대회명
- 선택한 디비전명
- 참가비

### 본문
- 신청 팀 선택
- 대표 연락처 확인
- 유의사항/환불정책 동의
- 필요 시 추가 메모

### 하단 고정 영역
- 최종 참가비
- 신청 버튼

---

## UX 포인트
### 1) 팀 정보는 되도록 자동 채움
가능하면 팀장 기준으로:
- 팀명
- 대표자명
- 연락처

자동 채움 후 수정만 가능하게 하는 것이 좋다.

### 2) 최종 금액은 마지막에 다시 보여주기
- “최종 참가비: 150,000원”
- “신청 후 입금 안내가 표시됩니다.”

### 3) 신청 버튼 문구도 중요
단순히 `신청하기` 보다 상황에 따라:

- `신청하고 입금 안내 보기`
- `신청하기`
- `대기 신청하기`

처럼 다르게 주는 것이 좋다.

---

# 2-3. 신청 직후 UX

신청 직후 UX는 가장 중요하다.  
이때 사용자가 “내가 지금 뭘 해야 하는지” 바로 이해해야 한다.

---

## 케이스 A. 자리가 있는 경우
상태: `payment_pending`

### 사용자에게 보여줄 핵심 메시지
- 신청이 접수되었어요
- 자리가 임시로 확보되었어요
- 입금 기한 내 계좌이체를 해주세요
- 운영자 확인 후 참가가 확정돼요

### 화면에 보여야 하는 정보
- 신청 완료 배너
- 현재 상태: 입금 대기
- 입금 기한
- 참가비
- 계좌번호 / 은행 / 예금주
- 입금자명 규칙
- “입금 완료했어요” 버튼
- 문의 방법

### 예시 문구
- “신청이 접수되었고, 참가 자리가 확보되었습니다.”
- “2026-04-10 23:59까지 입금해 주세요.”
- “운영자 확인 후 참가가 최종 확정됩니다.”

---

## 케이스 B. 자리가 없는 경우
상태: `waitlisted`

### 사용자에게 보여줄 핵심 메시지
- 현재 정원이 마감되었어요
- 대기열로 신청되었어요
- 자리 발생 또는 증원 시 순서대로 안내드려요
- 지금은 입금할 필요가 없어요

### 화면에 보여야 하는 정보
- 현재 상태: 대기열
- 대기 순번
- 대기열 안내 문구
- 승격 시 알림 예정 문구

### 예시 문구
- “현재 모집 정원이 마감되어 대기열로 접수되었어요.”
- “현재 대기 순번은 3번입니다.”
- “자리 발생 시 순서대로 입금 안내가 제공됩니다.”

---

# 2-4. 입금 완료 표시 UX

## 목적
사용자가 입금 후 “내가 뭘 더 해야 하는지” 명확히 알게 해야 한다.

---

## 추천 기능
- `입금 완료했어요` 버튼
- 입금자명 입력
- 전달 메모 입력(선택)

---

## 버튼 클릭 후 UX
상태: `paid_pending_approval`

### 사용자에게 보여줄 핵심 메시지
- 입금 완료 요청이 접수되었어요
- 운영자가 확인 중이에요
- 확인 완료 후 참가가 확정돼요

### 예시 문구
- “입금 완료 요청이 접수되었습니다.”
- “운영자가 입금 내역을 확인하고 있어요.”
- “확인 완료 시 참가 확정으로 변경됩니다.”

---

## UX 포인트
### 1) 입금 완료 버튼을 눌렀다고 확정처럼 보이면 안 됨
사용자가 종종 `입금완료 = 참가확정` 으로 오해할 수 있다.

그래서 반드시:
- `입금 확인 중`
- `운영자 승인 대기`
같은 문구를 명확히 보여줘야 한다.

### 2) 입금 후 예상 확인 흐름 안내
예:
- “보통 운영자가 순차적으로 확인합니다.”
- “확인 전까지 상태는 ‘입금 확인 중’으로 표시됩니다.”

---

# 2-5. 참가 확정 UX

## 상태
`confirmed`

## 사용자에게 보여줄 핵심 메시지
- 참가가 확정되었어요
- 이제 일정/공지/팀 관리로 이동할 수 있어요

---

## 화면에 보여야 하는 정보
- 확정 배지
- 디비전명
- 참가비
- 확정 일시
- 이후 액션 버튼

예:
- 대회 상세 보기
- 내 팀 보기
- 참가 신청 내역 보기

---

## 예시 문구
- “참가가 최종 확정되었습니다.”
- “대회 공지와 경기 일정을 확인해 주세요.”

---

# 2-6. 입금 기한 만료 UX

## 상태
`expired`

## 사용자에게 보여줄 핵심 메시지
- 입금 기한이 지나 신청이 만료되었어요
- 자리가 반환되었어요
- 상황에 따라 다시 신청하거나 대기열로 안내될 수 있어요

---

## 예시 문구
- “입금 기한이 지나 신청이 만료되었습니다.”
- “다시 신청이 가능한 경우 재신청해 주세요.”

---

## UX 포인트
### 1) 왜 만료됐는지 이유를 분명히
그냥 “신청 취소됨” 보다는
- “입금 기한 초과로 자동 만료”
가 훨씬 이해하기 쉽다.

### 2) 다음 행동 제안
- 다시 신청하기
- 대회 보기
- 문의하기

---

# 2-7. 유저 마이페이지 / 신청 내역 UX

## 목적
사용자가 나중에 다시 와도 상태를 바로 확인할 수 있어야 한다.

---

## 신청 내역 목록에 보여줄 것
- 대회명
- 디비전명
- 신청일
- 현재 상태
- 참가비
- 다음 액션 버튼

---

## 상태별 액션 예시
### `payment_pending`
- 입금 안내 보기
- 입금 완료 표시

### `paid_pending_approval`
- 상태 확인
- 문의하기

### `confirmed`
- 대회 상세 보기

### `waitlisted`
- 대기열 상태 보기

### `expired`
- 재신청 가능 시 재신청

---

# 3. 운영자 UX 정리

# 3-1. 운영자 대시보드 / 신청 관리 화면

## 목적
운영자는 신청 건이 많아져도 지금 처리해야 할 일을 빠르게 파악해야 한다.

---

## 우선 보여야 하는 요약 정보
디비전별로:
- 정원
- 확정 팀 수
- 입금 대기 수
- 입금 확인 중 수
- 대기열 수
- 남은 자리 수

---

## 상단 요약 카드 예시
- 오늘 신규 신청
- 입금 확인 대기
- 기한 임박 신청
- 대기열 보유 디비전 수
- 증원 검토 필요 디비전

이렇게 요약이 있으면 운영자가 전체 상황을 빠르게 볼 수 있다.

---

# 3-2. 신청 목록 UX

## 리스트에 기본으로 보여야 하는 컬럼
- 신청일시
- 대회명
- 디비전명
- 팀명
- 대표자
- 연락처
- 참가비
- 상태
- 입금자명
- 입금 기한

---

## 정렬/필터가 중요
운영자가 많이 쓰게 될 필터:

- 상태별 필터
- 디비전별 필터
- 입금 기한 임박순
- 신청일 순
- 대기열 순번 순

---

## 운영자 입장에서 가장 중요한 필터
### 1) 입금 확인 필요
- `paid_pending_approval`

### 2) 곧 만료될 건
- `payment_pending` + 만료 임박

### 3) 대기열 있는 디비전
- `waitlisted`

---

# 3-3. 신청 상세 UX

## 운영자가 상세에서 보고 싶은 것
- 팀 정보
- 디비전 정보
- 신청 시점 금액
- 현재 상태
- 입금자명
- 입금 완료 표시 여부
- 메모
- 상태 변경 이력

---

## 운영자 액션 버튼 예시
### `payment_pending`
- 입금 확인 처리
- 수동 만료 처리
- 취소 처리

### `paid_pending_approval`
- 입금 확인 후 확정
- 확인 보류
- 취소 처리

### `waitlisted`
- 수동 승격
- 대기열 순번 조정(정책상 허용 시)
- 취소 처리

### `confirmed`
- 취소
- 메모 추가

---

# 3-4. 입금 확인 UX

## 운영자의 실제 목적
운영자는 통장 내역과 신청 내역을 빠르게 대조해야 한다.

따라서 상세/리스트에서 아래 정보가 잘 보여야 한다.

- 팀명
- 참가비
- 입금자명
- 신청자명
- 입금 완료 요청 시간
- 운영 메모

---

## 확인 버튼 UX
운영자가 `입금 확인` 버튼을 누르면:

- 상태가 `confirmed`로 변경
- 확정 시간 기록
- 필요 시 사용자 알림 발송

---

## 확인 보류 UX
입금 정보가 애매할 수 있으므로:
- `보류`
- `운영 메모`
기능이 있으면 좋다.

예:
- “입금자명 불일치”
- “금액 확인 필요”

---

# 3-5. 대기열 UX

## 운영자에게 필요한 것
대기열은 단순히 “있다” 보다 아래가 보여야 의미가 있다.

- 디비전별 대기열 수
- 현재 대기 순번
- 증원 시 몇 명 승격 가능한지
- 취소분 발생 시 누구를 올릴지

---

## 추천 UX
### 디비전 상세에 표시
- 정원: 20
- 확정: 18
- 입금대기: 2
- 대기열: 7

### 버튼
- 증원하기
- 대기열 상위 n명 승격
- 특정 팀 수동 승격

---

# 3-6. 정원 증원 UX

## 목적
운영자가 “20팀 → 30팀” 같은 결정을 했을 때  
후속 처리를 쉽게 해야 한다.

---

## 증원 화면에서 보여야 하는 것
- 기존 정원
- 현재 확정 수
- 현재 입금대기 수
- 대기열 수
- 증원 후 추가 가능 수

---

## 추천 동작
운영자가 정원을 30으로 수정하면:

- “추가로 10팀 승격 가능합니다”
- “대기열 상위 10팀을 입금 대기로 전환할까요?”

같은 확인 UX가 있으면 좋다.

---

## 좋은 UX 포인트
정원 변경은 숫자만 바꾸는 것이 아니라  
**대기열 후속 처리까지 한 번에 연결**되어야 운영이 편하다.

---

# 3-7. 만료 처리 UX

## 운영자 입장에서 필요한 것
- 어떤 신청이 곧 만료되는지
- 이미 만료된 신청이 무엇인지
- 만료 후 대기열 승격이 필요한지

---

## 추천 UX
### 기한 임박 강조
- 1시간 이내: 강한 강조
- 당일 만료: 강조 배지

### 만료 후
- 자동 `expired`
- “다음 대기 팀 승격” 액션 노출

---

# 4. 상태별 UX 문구 가이드

# 4-1. 유저용 상태 문구

## `payment_pending`
- 입금 대기
- 자리가 확보되었어요. 입금 기한 내 계좌이체를 완료해 주세요.

## `paid_pending_approval`
- 입금 확인 중
- 운영자가 입금 내역을 확인하고 있어요.

## `confirmed`
- 참가 확정
- 참가가 최종 확정되었습니다.

## `waitlisted`
- 대기열 접수
- 현재 정원이 마감되어 대기열로 접수되었어요.

## `expired`
- 입금 기한 만료
- 입금 기한이 지나 신청이 만료되었습니다.

## `cancelled`
- 신청 취소
- 신청이 취소되었습니다.

# 4-2. 운영자용 상태 문구

## `payment_pending`

## `paid_pending_approval`

- 입금 요청 접수 / 운영 확인 필요

## `confirmed`

- 참가 확정 완료

# 5. UX 관점의 핵심 차별 포인트

## 5-1. 사용자에게는 “다음 행동”이 보여야 한다
좋은 UX는 상태만 보여주는 것이 아니라  
다음에 뭘 하면 되는지 같이 보여준다.

예:
- 입금 대기 → 입금 안내 보기
- 입금 확인 중 → 기다리기 / 문의하기
- 참가 확정 → 일정 보기
- 대기열 → 순번 확인
- 만료 → 재신청

---

## 5-2. 운영자에게는 “처리 우선순위”가 보여야 한다
운영자는 전체 목록보다  
**지금 처리해야 하는 항목 큐**가 중요하다.

예:
- 입금 확인 필요 8건
- 오늘 만료 예정 5건
- 대기열 있는 디비전 2개

---

## 5-3. 상태 전환 이유를 사용자도 이해할 수 있어야 한다
예:
- 왜 확정이 안 됐는지
- 왜 대기열인지
- 왜 만료됐는지

이유가 보여야 CS가 줄어든다.

---

# 6. MVP UX 결론

## 유저 UX 핵심
- 신청 가능/대기열 여부를 신청 전부터 명확히 보여준다
- 신청 직후에는 “자리 확보 여부”와 “입금 기한”을 분명히 안내한다
- 입금 후에는 “확정”이 아니라 “입금 확인 중”임을 분명히 보여준다
- 대기열일 경우 현재 순번과 향후 처리 방식을 안내한다
- 마이페이지에서 언제든 현재 상태와 다음 행동을 확인할 수 있게 한다

---

## 운영자 UX 핵심
- 디비전별 정원/확정/입금대기/대기열 현황을 한눈에 볼 수 있어야 한다
- 입금 확인 대기 건과 기한 임박 건을 우선 처리할 수 있어야 한다
- 정원 증원 시 대기열 승격까지 자연스럽게 이어져야 한다
- 신청 상세에서 입금 확인, 보류, 취소, 승격 등의 액션이 쉬워야 한다
- 상태별 필터와 우선순위 큐가 잘 제공되어야 한다

---

## 최종 한 줄 정리
MVP UX의 핵심은  
**유저에게는 “지금 내 상태와 다음 행동”을 명확히 보여주고,  
운영자에게는 “지금 처리해야 할 신청”을 빠르게 다룰 수 있게 만드는 것**이다.
