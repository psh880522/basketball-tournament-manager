# 본인인증 / 가입 / 프로필 / 대회참가 흐름 정리

작성일: 2026-04-02  
상태: 아이디어 정리 / 추후 설계 및 티켓화 예정

---

## 1. 목적

서비스의 초기 가입 장벽은 낮추되, 실제 대회 참가와 같은 민감한 액션은 본인인증을 완료한 사용자만 가능하도록 흐름을 분리한다.

목표는 다음과 같다.

- 회원가입 전환율을 높인다.
- 대회 참가자의 신뢰도를 확보한다.
- 중복 참가, 허위 정보, 운영상 분쟁 가능성을 줄인다.
- 향후 본인인증 정책을 확장 가능한 구조로 설계한다.

---

## 2. 제안하는 전체 흐름

### 사용자 관점 흐름
1. 회원가입(기본 입력)
2. 프로필 입력
3. 본인인증
4. 대회 참가 신청

### 정책 관점 흐름
- 회원가입은 누구나 가능
- 프로필 입력은 서비스 사용 편의를 위해 유도
- 본인인증 완료 사용자만 대회 참가 가능
- 필요 시 팀 생성, 팀장 기능, 참가 확정 등도 본인인증 사용자만 허용 가능

---

## 3. 왜 이 흐름이 괜찮은가

### 3.1 가입 장벽을 낮출 수 있음
가입 단계에서 바로 본인인증까지 요구하면 이탈 가능성이 커진다.  
기본 계정만 먼저 만들 수 있게 하면 사용자는 서비스 탐색을 먼저 할 수 있고, 실제 대회 참가 시점에만 인증을 요구할 수 있다.

### 3.2 본인인증 필요성을 설명하기 쉬움
사용자는 “가입하려면 왜 본인인증을 해야 하지?”보다  
“대회 참가를 위해 본인확인이 필요합니다”를 더 자연스럽게 받아들일 가능성이 높다.

### 3.3 운영 정책 확장에 유리함
초기에는 대회 참가만 인증 필수로 두고, 나중에는 다음처럼 확장할 수 있다.

- 특정 대회만 인증 필수
- 팀 생성/팀장 기능 인증 필수
- 미성년자 대회 시 보호자 정보 추가
- 상금/상품 수령 시 추가 인증
- 운영 승인형 참가 흐름 추가

---

## 4. 권장 구조

겉으로는 아래처럼 보여줄 수 있다.

- 회원가입 완료
- 참가자 정보 입력
- 본인인증 완료 후 대회 참가 가능

하지만 내부 시스템 상태는 분리하는 것이 좋다.

### 권장 내부 상태
1. `registered`
   - 기본 회원가입 완료
   - 로그인 가능
   - 서비스 탐색 가능

2. `profile_completed`
   - 대회 참가에 필요한 기본 프로필 입력 완료

3. `verified`
   - 본인인증 완료
   - 대회 참가 가능

즉, UI 상으로는 프로필 단계가 하나처럼 보여도 실제 데이터 상태는 최소한 아래 두 가지를 분리해 관리하는 것이 좋다.

- 프로필 입력 완료 상태
- 본인인증 완료 상태

## 5. 추천하는 최종 흐름

### 추천안
1. 회원가입
   - 이메일/비밀번호 또는 소셜 로그인으로 기본 계정 생성

2. 프로필 작성
   - 이름
   - 닉네임
   - 생년월일
   - 성별
   - 소속
   - 포지션
   - 기타 참가에 필요한 정보

3. 본인인증
   - 휴대폰/PASS/기타 인증 수단 연동
   - 인증 완료 시 참가 가능 상태 활성화

4. 대회 참가 신청
   - 본인인증 완료 사용자만 신청 가능

- 향후 결제/환불/보상 관련 기능

## 7. 본인인증 도입 시 실제 처리 흐름

본인인증은 보통 회원가입 로직에 직접 섞기보다, 외부 인증 서비스와 연동한 별도 흐름으로 처리한다.

### 일반적인 처리 순서
1. 사용자가 본인인증 시작
2. 서버가 인증 요청 세션 생성
3. 사용자를 외부 인증사 화면으로 이동
4. 외부 인증사가 인증 수행
5. 인증 결과를 서버 callback 또는 프론트 복귀 형태로 반환
6. 서버가 결과를 검증
7. 검증 완료 후 사용자 상태를 `verified`로 변경
8. 이후 대회 참가 가능

### 핵심 원칙
- 프론트에서 “인증 성공”이라고 전달하는 값만 믿지 않는다.
- 서버가 반드시 인증 결과의 진위 여부를 검증해야 한다.
- 같은 인증 결과가 재사용되지 않도록 해야 한다.
- 개인정보는 최소 범위로만 저장해야 한다.

---

## 8. 데이터/모델링 방향

본인인증이 들어오더라도 회원 데이터와 인증 이력 데이터는 분리하는 것이 좋다.

### 권장 분리 예시
- `users`
- `profiles`
- `identity_verifications`

### 예시 역할
#### `users`
- 계정 자체
- 로그인 수단
- 계정 생성 시각

#### `profiles`
- 사용자 기본 정보
- 대회 참가용 부가 정보
- 프로필 완성 여부

#### `identity_verifications`
- 인증 provider
- transaction id
- verification status
- verified at
- 고유 식별자(CI/DI 등 정책에 따라)
- 최소한의 인증 메타데이터

---

## 9. 상태값 예시

### 프로필/인증 상태 예시 필드
- `signup_completed_at`
- `profile_completed_at`
- `identity_verification_status`
  - `pending`
  - `verified`
  - `failed`
- `identity_verified_at`

또는 서비스 레이어에서 아래처럼 판단할 수 있다.

- `is_profile_completed`
- `is_identity_verified`
- `can_join_tournament`

단, 참가 가능 여부는 단순 플래그 하나로 끝내기보다 정책 기반으로 계산되게 두는 것이 확장에 유리하다.

## 10. 특히 주의할 점

### 10.1 프로필 완료와 본인인증 완료를 분리할 것
둘을 하나의 상태로 합치면 아래 경우가 애매해진다.

- 프로필은 입력했지만 본인인증 실패
- 본인인증은 완료했지만 참가 필수 정보 누락
- 일부 대회만 추가 정보가 필요한 경우

### 10.2 프론트 성공값만 신뢰하지 말 것
서버가 인증사 응답의 위변조 여부, 거래 ID 일치 여부, 만료 여부, 재사용 여부를 반드시 확인해야 한다.

### 10.3 개인정보 최소 보관 원칙 적용
이름, 생년월일, 휴대폰 번호, 인증 원문 응답 등은 반드시 서비스 정책과 법적 필요 범위를 검토한 뒤 최소한으로 저장한다.

### 10.4 운영 정책 확장 가능하게 설계할 것
초기에는 대회 참가만 제한하더라도, 향후 팀 생성/팀장/결제/환불/상금 수령 등으로 확장될 수 있도록 흐름을 느슨하게 설계한다.

## 11. 현재 시점의 추천 결론

현재 서비스 방향에서는 아래 구조가 가장 합리적이다.

**회원가입(기본 계정 생성) → 프로필 작성 → 본인인증 완료 → 대회 참가 가능**

이 구조의 장점은 다음과 같다.

- 가입 장벽이 낮다.
- 사용자가 본인인증 필요성을 이해하기 쉽다.
- 운영 정책을 단계적으로 강화할 수 있다.
- 향후 기능 확장에 유리하다.

## 14. 메모

현재 단계에서는 “회원가입과 본인인증을 분리하되, 대회 참가에는 본인인증을 강제한다”는 원칙만 확정해도 충분하다.  
실제 구현 시에는 인증 provider 선정, 개인정보 처리 정책, DB 구조, 권한 체크 지점을 별도 설계 문서에서 구체화하는 것이 바람직하다.
