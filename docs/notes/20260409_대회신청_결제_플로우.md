# 대회 신청 ~ 결제(MVP) 플로우 정리

## 목적
부문별(디비전별) 참가비가 다른 농구 대회에서,  
**대회 신청부터 결제(계좌이체) 및 참가 확정까지의 흐름**을 MVP 기준으로 정리한다.

이 문서는 향후 다음을 설계/구현할 때 참고한다.

- 디비전 참가비 관리
- 신청 상태 설계
- 정원 초과/대기열 처리
- 계좌이체 기반 입금 확인
- 추후 PG 결제 확장

---

# 1. 기본 방향

## 1-1. 참가비는 디비전 기준으로 관리
각 디비전은 기본 참가비를 가진다.

예:
- 남자부 A조: 150,000원
- 여자부: 120,000원
- 청소년부: 80,000원

따라서 디비전에 기본 참가비 필드가 필요하다.

예시:
- `divisions.entry_fee`

---

## 1-2. 디비전 기본 참가비와 실제 신청 금액은 분리한다
운영 중에는 다음과 같은 상황이 생길 수 있다.

- 참가비 변경
- 특정 팀 할인
- 무료 처리
- 수기 금액 조정

그래서 **디비전의 기본 참가비**와 **신청 당시 확정된 금액**은 분리한다.

예:
- `divisions.entry_fee`: 현재 디비전 기본 참가비
- `applications.base_entry_fee`: 신청 당시 기준 참가비
- `applications.final_amount`: 실제 청구 금액

즉, 신청 생성 시 참가비를 스냅샷으로 저장한다.

---

# 2. MVP 결제 방식

## 2-1. MVP는 PG가 아니라 계좌이체 기준으로 운영
초기 MVP에서는 PG(카드/간편결제) 연동을 바로 하지 않고,  
**계좌이체 + 운영자 입금 확인** 방식으로 운영한다.

이유:
- 구현 복잡도 낮음
- 실제 운영 검증 가능
- 결제 모듈 없이도 대회 운영 시작 가능

---

## 2-2. MVP에서는 자동 승인이 아니라 수동 승인이 기본
계좌이체는 시스템이 자동으로 입금을 정확히 알기 어렵다.

시스템이 바로 알 수 없는 것:
- 실제 입금 여부
- 입금자명 일치 여부
- 금액 정확성
- 중복 입금 여부

따라서 MVP에서는 아래 흐름이 기본이다.

1. 대회 신청
2. 입금 대기
3. 운영자 입금 확인
4. 참가 확정

즉, **입금 완료 = 자동 확정 아님**  
**운영자 확인 후 확정**이다.

---

# 3. 신청/결제 기본 플로우

## 3-1. 전체 흐름
1. 팀장이 대회 상세 페이지 진입
2. 신청 가능한 디비전 확인
3. 디비전 선택 후 신청
4. 시스템이 자리 가능 여부 판단
5. 자리가 있으면 결제 대기 상태로 진입
6. 입금 안내 노출
7. 운영자가 입금 확인
8. 참가 확정

---

## 3-2. 사용자가 보는 기본 흐름
### 자리 있는 경우
- 대회 신청
- 결제 대기(입금 대기)
- 운영자 입금 확인
- 참가 확정

### 자리 없는 경우
- 대회 신청
- 대기열 등록
- 정원 증가/취소 자리 발생 시 승격
- 결제 대기(입금 대기)
- 운영자 입금 확인
- 참가 확정

---

# 4. 상태 설계

## 4-1. 상태를 구분해야 하는 이유
“대기”라는 표현 안에 서로 다른 의미가 섞이면 운영/개발이 헷갈린다.

반드시 구분해야 하는 두 가지:

### A. 결제 가능한 대기
- 자리는 확보됨
- 입금하면 확정 가능

### B. 자리 기다리는 대기
- 아직 자리 없음
- 정원 증가/취소분 발생 시 기회 부여

따라서 상태를 아래처럼 나누는 것이 적절하다.

---

## 4-2. 추천 상태값(MVP)
- `payment_pending`
- `paid_pending_approval`
- `confirmed`
- `waitlisted`
- `expired`
- `cancelled`

---

## 4-3. 상태 정의

### `payment_pending`
- 자리를 확보한 상태
- 입금 대기 상태
- 입금 기한 내 입금해야 함

### `paid_pending_approval`
- 사용자가 입금 완료 의사를 표시했거나
- 운영자가 아직 확인 중인 상태
- 아직 최종 참가 확정은 아님

### `confirmed`
- 운영자가 입금 확인 완료
- 참가 확정 상태

### `waitlisted`
- 현재 자리가 없어 대기열에 있는 상태
- 아직 결제 가능 상태가 아님

### `expired`
- 입금 기한이 지나 자동 만료된 상태
- 확보했던 자리는 반환됨

### `cancelled`
- 사용자가 취소하거나
- 운영자가 취소 처리한 상태

---

# 5. 왜 `payment_pending` 과 `waitlisted` 를 분리해야 하는가

## 5-1. `payment_pending`
이 상태는 **이미 자리를 확보한 상태**다.

즉:
- 정원 계산에 포함된다
- 입금만 완료되면 참가 확정 가능하다

---

## 5-2. `waitlisted`
이 상태는 **아직 자리를 확보하지 못한 상태**다.

즉:
- 정원 계산에 바로 포함되지 않는다
- 결제 가능 상태가 아니다
- 정원 증가 또는 취소분 발생 시 순번대로 승격된다

---

## 5-3. 분리하지 않으면 생기는 문제
정원이 이미 꽉 찼는데도 모든 팀을 `결제 대기`로 두면,
누가 실제 자리를 확보한 팀인지 구분이 안 된다.

그 결과:
- 초과 확정 가능성 발생
- 운영자 혼선
- 사용자 안내 혼선
- 정원 계산 오류

따라서 **자리 확보 대기**와 **단순 대기열**은 반드시 분리한다.

---

# 6. 정원(capacity) 처리 원칙

## 6-1. 정원은 운영 중 변경 가능해야 한다
예:
- 처음엔 20팀 모집
- 신청이 많아 30팀으로 확대

따라서 디비전 정원은 고정값이 아니라 운영자가 수정 가능한 값이어야 한다.

예:
- `divisions.capacity`

---

## 6-2. 정원 증가 시 처리 원칙
정원을 늘렸을 때는 단순히 숫자만 바꾸는 것이 아니라  
**대기열을 어떻게 재평가할지**가 중요하다.

### 권장 방식
- `waitlisted` 상태 팀 중 순번대로 승격
- 승격된 팀을 `payment_pending`으로 변경
- 입금 기한 부여
- 기한 내 미입금 시 다음 대기 팀에게 기회 이동

즉, 정원 증가 시 바로 새 신청자에게 열기보다  
**기존 대기열 우선 처리**가 공정하다.

---

## 6-3. 정원 감소 시 처리 원칙
정원 감소는 증가보다 더 조심해야 한다.

권장 정책:
- 현재 `confirmed` + `payment_pending` 수보다 낮게는 줄일 수 없게 한다
- 이미 확보된 자리를 자동 박탈하지 않는다

즉, 활성 신청 수보다 낮게 정원을 내리는 것은 막는 것이 안전하다.

---

# 7. 신청 시 정원 처리 방식(MVP)

## 7-1. 자리가 있으면
신청 시점에 자리가 있으면:
- 신청 생성
- 상태를 `payment_pending`으로 설정
- 입금 기한 부여

즉, **MVP에서도 자리가 있으면 먼저 확보하고 입금 대기로 보낸다.**

---

## 7-2. 자리가 없으면
신청 시점에 자리가 없으면:
- 상태를 `waitlisted`로 설정
- 대기열 순번 부여

즉, 이 팀은 바로 입금 단계로 보내지 않는다.

---

## 7-3. 정원 계산에 포함되는 상태
정원 계산 시 포함해야 하는 상태:

- `confirmed`
- `payment_pending`
- 필요시 `paid_pending_approval`

운영 정책상 `paid_pending_approval`도 자리를 점유하는 상태로 보는 것이 자연스럽다.

---

# 8. 입금 대기 처리

## 8-1. 입금 안내 정보
`payment_pending` 상태가 되면 사용자에게 아래 정보를 보여준다.

- 참가비
- 입금 계좌번호
- 은행명
- 예금주
- 입금자명 규칙
- 입금 기한

예:
- 국민은행 123456-78-901234
- 예금주: 홍길동
- 입금자명: 팀명_대표자명
- 입금 기한: 2026-04-10 23:59

---

## 8-2. 입금 기한이 필요한 이유
자리를 무기한 점유하면 다른 팀이 신청할 수 없게 된다.

따라서 `payment_pending` 상태에는 반드시 입금 기한이 필요하다.

예:
- 신청 후 12시간
- 신청 후 24시간
- 당일 자정까지

---

## 8-3. 입금 기한 만료 처리
기한 내 입금 확인이 되지 않으면:
- 상태를 `expired`로 변경
- 확보한 자리를 반환
- 필요 시 대기열 다음 팀을 승격

---

# 9. 사용자 입금 완료 표시 기능

## 9-1. 있으면 좋은 기능
MVP에서도 아래 기능이 있으면 운영이 편해진다.

- “입금 완료했어요” 버튼
- 입금자명 입력
- 입금 시간 메모
- 송금자명 불일치 메모

---

## 9-2. 상태 반영
사용자가 입금 완료 표시를 하면:
- 상태를 `paid_pending_approval` 로 변경 가능

다만 이것은 **자동 확정이 아니라 운영자 확인 대기**를 의미한다.

---

# 10. 운영자 입금 확인 플로우

## 10-1. 운영자 확인 시
운영자는 신청/입금 목록에서 다음을 보고 확인한다.

- 팀명
- 디비전
- 참가비
- 입금자명
- 신청 시간
- 상태
- 입금 확인 여부

---

## 10-2. 운영자 액션
### 입금 확인 성공
- 상태를 `confirmed`로 변경
- 참가 확정

### 입금 불일치/미확인
- 그대로 유지하거나
- 메모 추가
- 필요 시 사용자 문의

### 입금 기한 초과
- `expired` 처리
- 자리 반환

---

# 11. “정원 다 찬 뒤 증원” 시나리오 처리

## 11-1. 예시 상황
- 원래 정원 20팀
- 이미 자리가 다 참
- 추가 신청 팀들은 `waitlisted`
- 운영자가 “30팀까지 받자”고 결정

---

## 11-2. 처리 방식
1. 디비전 정원을 30으로 수정
2. 추가 확보된 10자리 계산
3. `waitlisted` 상위 팀부터 순서대로 `payment_pending` 승격
4. 승격된 팀에게 입금 기한 부여
5. 입금 확인 후 `confirmed`
6. 기한 내 미입금 시 다음 대기 팀 승격

---

## 11-3. 핵심 원칙
정원을 늘렸다고 해서 새 신청자를 바로 먼저 받기보다,  
**기존 대기열을 먼저 처리하는 것이 공정하다.**

---

# 12. 자동 승인과 수동 승인 차이

## 12-1. 자동 승인
결제 완료 시 시스템이 즉시 참가를 확정하는 방식

장점:
- 사용자 경험이 빠름
- 운영자 업무 감소
- 실시간 선착순 처리 용이

단점:
- 잘못된 신청도 즉시 확정될 수 있음
- 문제 발생 시 취소/환불 필요

---

## 12-2. 수동 승인
결제/입금 후 운영자가 확인 후 참가 확정하는 방식

장점:
- 운영 통제 쉬움
- 예외 처리 용이
- 입금 누락/오입금 대응 가능

단점:
- 사용자 입장에서 확정까지 시간이 걸림
- 운영자 업무가 늘어남

---

## 12-3. 현재 MVP 결론
현재 MVP는 계좌이체이므로  
**수동 승인 방식이 기본**이다.

즉:
- `payment_pending` → `paid_pending_approval` → `confirmed`

이 흐름이 현실적이다.

---

# 13. 추후 PG 결제 확장 시 방향

## 13-1. PG 도입 후
향후 PG를 붙이면 다음이 가능해진다.

- 카드 결제
- 간편결제
- 결제 성공 콜백 처리
- 자동 승인

---

## 13-2. PG 도입 후 가능한 승인 구조
### 자동 승인형
- `payment_pending` → `confirmed`

### 수동 승인형
- `payment_pending` → `paid_pending_approval` → `confirmed`

즉, 장기적으로는 대회/디비전별로 자동 승인/수동 승인 옵션을 둘 수 있다.

---

# 14. MVP 기준 추천 정책 요약

## 14-1. 핵심 정책
- 디비전별 기본 참가비를 가진다
- 신청 생성 시 참가비를 스냅샷으로 저장한다
- MVP 결제 방식은 계좌이체다
- 계좌이체는 운영자 수동 확인 후 확정한다
- 자리가 있으면 `payment_pending`
- 자리가 없으면 `waitlisted`
- 정원 증가 시 `waitlisted` 를 순서대로 `payment_pending` 으로 승격한다
- 입금 기한이 지나면 `expired` 처리한다

---

## 14-2. 추천 상태 흐름
### 자리가 있는 경우
`payment_pending`  
→ `paid_pending_approval`  
→ `confirmed`

또는 미입금 시:

`payment_pending`  
→ `expired`

---

### 자리가 없는 경우
`waitlisted`  
→ 정원 증가/취소분 발생  
→ `payment_pending`  
→ `paid_pending_approval`  
→ `confirmed`

---

# 16. 최종 결론

현재 MVP 기준으로 가장 적절한 구조는 아래와 같다.

1. 디비전별 참가비 관리
2. 신청 시 실제 청구 금액 스냅샷 저장
3. 신청 시 자리가 있으면 `payment_pending`
4. 자리가 없으면 `waitlisted`
5. 계좌이체 방식으로 입금 안내
6. 사용자가 입금 완료 표시 가능
7. 운영자가 입금 확인 후 `confirmed`
8. 입금 기한 초과 시 `expired`
9. 정원 증가 시 `waitlisted` 상위 팀부터 `payment_pending` 승격

즉, 현재 MVP는  
**“신청 → 자리 확보/대기열 → 계좌이체 입금 대기 → 운영자 확인 → 참가 확정”**  
구조로 설계하는 것이 가장 현실적이다.
